## Settings Screen Audit

### Executive Summary

The Settings screen is architecturally sound in its top-level design: all 4 tabs are `React.lazy()` loaded and only render when active. No data fetches occur on initial `/settings` entry before a tab mounts. However, the audit uncovered **3 duplicate DAL patterns**, **1 iOS stacking context violation**, **2 near-identical adapter components**, and **1 identity subscription that could be lifted**. No over-fetching was found at the tab level — each tab fetches only when active and only what it needs.

### Current Runtime Behavior

1. User navigates to `/settings` (default `?tab=privacy`).
2. `SettingsScreen` renders the header + tab bar. No hooks, no DAL calls, no identity reads.
3. The active tab's chunk is lazy-loaded via `React.lazy()`.
4. Only the active tab renders and mounts its hooks. Inactive tabs are fully unmounted (`tab === item.key` guard on line 130).
5. Switching tabs unmounts the previous tab and mounts the new one (no `keepAlive`).
6. URL `?tab=` param is synced via `useSearchParams` with `replace: true`.

### Initial Load Behavior

On `/settings` entry (before any tab content renders):
- **React hooks in SettingsScreen:** `useNavigate`, `useSearchParams`, `useState`, `useMemo`, `useRef`, `useCallback`, `useEffect` (URL sync only)
- **DAL calls:** Zero
- **Identity reads:** Zero
- **Network requests:** Zero (only the lazy chunk fetch for the default tab)

This is correct. The shell is inert.

### Tab-by-Tab Dependency Map

#### Privacy Tab (`PrivacyTab.view.jsx`)

- **Data fetches on mount:**
  1. `MyBlocksProvider` -> `ctrlListMyBlocks` -> `dalListMyBlocks` (moderation.blocks) + `hydrateActorsFromRows` (blocked actors)

  4. `UserLookup` — no fetch on mount (user-initiated search only)
  5. `BlockedUsersSimple` — consumes `MyBlocksProvider` context (no additional fetch)
- **Total mount queries:** 3 parallel (blocks list, privacy setting, follow requests)

#### Profile Tab (`ProfileTab.jsx` -> `useProfileController`)
- **Identity:** `useIdentity()` — reads `actorId`, `kind`
- **Auth:** `useAuth()` — reads `user.id`
- **Data fetches on mount:**
  1. Resolves `subjectId`: user mode -> `user.id`, vport mode -> `ctrlResolveVportIdByActorId` -> `dalReadVportIdByActorId` (vc.actors)
  2. Loads profile: `loadProfileCore` -> `fetchProfile` (public.profiles or vc.vports)
- **Total mount queries:** 1-2 sequential (resolve subject, then load profile)

#### Account Tab (`AccountTab.view.jsx` -> `useAccountController`)
- **Identity:** `useIdentity()` — reads `actorId`, `kind`
- **Auth:** `useAuth()` — reads `user.id`, `logout`
- **Data fetches on mount:**
  1. If vport: `ctrlResolveVportIdByActorId` -> `dalReadVportIdByActorId` (vc.actors)
- **Total mount queries:** 0-1 (only if vport identity active)
- **No data display on mount** — purely action-oriented (logout, delete)

#### Vports Tab (`VportsTab.view.jsx` -> `useVportsController`)
- **Identity:** `useIdentity()` — reads `identity`, `switchActor`
- **Auth:** `useAuth()` — reads `user.id`
- **Data fetches on mount:**
  1. `useVportsList` -> `listMyVports` (vc.actor_owners + vc.actors + vc.vports join)
  2. `useProfileActor` -> `ctrlGetProfileActorId` -> `readActorOwnersByUserDAL` (vc.actor_owners + vc.actors)
- **Total mount queries:** 2 parallel

### Duplicate Logic Findings

**DUPLICATE 1: `dalReadVportIdByActorId` — 2 identical copies**
- `apps/VCSM/src/features/settings/profile/dal/actors.read.dal.js` (used by Profile tab)
- `apps/VCSM/src/features/settings/account/dal/account.read.dal.js` (used by Account tab)
- Both do `vc.actors.select('vport_id').eq('id', actorId).maybeSingle()`
- Both import different Supabase clients (`vc` vs `supabase` — functionally equivalent via `vcClient`)
- **Impact:** Code duplication, two files doing the same query. Should consolidate into one shared DAL.

**DUPLICATE 2: `ctrlResolveVportIdByActorId` — 2 identical controllers**
- `apps/VCSM/src/features/settings/profile/controller/resolveVportIdByActorId.controller.js`
- `apps/VCSM/src/features/settings/account/controller/account.controller.js` (exported as `ctrlResolveVportIdByActorId`)
- Both are thin wrappers that call their respective duplicate DAL.
- **Impact:** Should be a single shared controller.

**DUPLICATE 3: `resolveVportIdFromActor` in Blocks.controller**
- `apps/VCSM/src/features/settings/privacy/controller/Blocks.controller.js` lines 23-33
- Does the same `vc.actors` lookup via `dalReadActorKindAndVportId` (slightly different — also reads `kind`).
- Not an exact duplicate but overlapping purpose.

**DUPLICATE 4: `UserProfileTab.jsx` and `VportProfileTab.jsx` are near-identical**
- Both adapters at `apps/VCSM/src/features/settings/profile/adapter/` have identical logic for draft state, file picking, save, and "Saved" toast.
- Only difference: loading text ("Loading profile..." vs "Loading VPORT...").
- **Impact:** ~120 lines of duplicated adapter code.

**DUPLICATE 5: `useVportsList` and `useProfileActor` both query `vc.actor_owners`**
- `useVportsList` -> `listMyVports` queries `actor_owners` joined to `actors` joined to `vports`
- `useProfileActor` -> `readActorOwnersByUserDAL` queries `actor_owners` joined to `actors`
- Both run on Vports tab mount. The profile actor ID could be derived from the same dataset that `listMyVports` returns.

### Identity Subscription Findings

4 independent `useIdentity()` calls across the 4 tabs:
1. **Privacy tab** (`PrivacyTab.view.jsx` line 11) — reads `actorId`, `kind`
2. **Profile tab** (`useProfileController.js` line 19) — reads `actorId`, `kind`
3. **Account tab** (`useAccountController.js` line 13) — reads `actorId`, `kind`
4. **Vports tab** (`useVportsController.js` line 13) — reads `identity`, `switchActor`

Since only one tab is active at a time, only one `useIdentity()` subscription is live at any moment. **This is correct and not wasteful.** No unnecessary concurrent subscriptions exist.

The SettingsScreen shell itself does NOT call `useIdentity()` — also correct.

### Lazy-Loading Findings

**Correct behavior:**
- All 4 tabs use `React.lazy()` with dynamic `import()`.
- Active tab renders inside `<Suspense>` with a loading fallback.
- Inactive tabs are fully unmounted (not just hidden) due to the `tab === item.key` conditional on line 130.
- Tab chunks are only fetched once (browser caches the module).

**Potential concern:**
- Switching tabs unmounts and remounts. If the user switches Privacy -> Profile -> Privacy, the Privacy tab will re-fetch all its data (blocks, privacy setting, follow requests). There is no caching layer for settings tab data.
- This is acceptable for a Settings screen (data should be fresh), but could feel slow on repeated switches.

### URL State Findings

- Tab state synced to `?tab=` via `useSearchParams` with `replace: true` (correct — no history pollution).
- `initialTab` is computed from URL on mount via `useMemo` with `[search]` dep — this is fine for initial render.
- The `useEffect` on lines 36-43 syncs `tab` state back to URL. It runs on every `tab` or `search` change.
- **Potential issue:** The `useMemo` for `initialTab` depends on `[search]`, but `initialTab` is only used as the initial value for `useState(initialTab)`. Since `useState` only uses the initial value once, the `[search]` dependency in `useMemo` is wasted computation (harmless but unnecessary).

### Risks

1. **iOS stacking context violation** — `AccountTab.view.jsx` renders a `ConfirmModal` with `position: fixed` + `backdrop-blur-sm` (line 107). The modal renders inside the settings page which has `backdrop-blur-xl` on the header (line 86 of SettingsScreen). Per CLAUDE.md iOS stacking rules, this could cause z-index issues on Safari/iOS. The modal should be a portal or fragment sibling.

2. **VportsTab.view.jsx** also renders a fixed-position creator modal (line 98) inside the settings container. Same iOS stacking concern.

3. **No error boundary** — If a lazy tab chunk fails to load (network error), the `<Suspense>` fallback will show indefinitely. There is no `<ErrorBoundary>` wrapping the lazy tabs.

4. **`console.error` calls** in `usePendingFollowRequestActions.js` (lines 28, 47) and `useVportsList.js` (line 16) and `useAccountController.js` (line 33) and `useProfileActor.js` (line 20) — these should be dev-only per project rules (no console.log/error in production).

5. **`useVportSwitch` navigates away from settings** after switching actors (to `/notifications` or `/vport/notifications`). This is intentional but could be surprising UX — user is in settings, switches actor, and lands on notifications.

### Architecture Violations

1. **Profile.controller.core.js line 56-59** — Direct `supabase` query inside a controller (`supabase.schema('vc').from('actors').select('id')...`). Controllers should call DALs, not query directly. This violates `DAL -> Model -> Controller` layer order.

2. **`useVportsList.js`** imports from `@/features/vport/dal/vport.read.vportRecords.dal` — this is a cross-feature DAL import. The vports settings tab reaches into the vport feature's DAL directly instead of going through an adapter or shared DAL.

3. **`UserProfileTab.jsx` and `VportProfileTab.jsx`** both create `URL.createObjectURL()` but never call `URL.revokeObjectURL()` on cleanup — minor memory leak on repeated avatar/banner picks within the same tab mount.

### Recommended Fix Sequence

1. **[LOW] Consolidate duplicate `dalReadVportIdByActorId`** — Create a single shared DAL in `apps/VCSM/src/features/settings/shared/dal/` or reuse the profile one from both account and profile controllers. Delete the account copy.

2. **[LOW] Consolidate duplicate `ctrlResolveVportIdByActorId`** — Single controller, imported by both profile and account hooks.

3. **[LOW] Merge `UserProfileTab` and `VportProfileTab`** into a single adapter with a loading text prop. ~120 lines of identical code.

4. **[MEDIUM] Fix iOS stacking context** — Move `ConfirmModal` in AccountTab and the creator modal in VportsTab to use React portals (`createPortal` to `document.body`).

5. **[MEDIUM] Add `ErrorBoundary`** wrapper around lazy tab `<Suspense>` blocks in `SettingsScreen.jsx`.

6. **[LOW] Extract direct Supabase query** from `Profile.controller.core.js` into a DAL function.

7. **[LOW] Combine `useVportsList` + `useProfileActor`** queries in the Vports tab — derive the profile actor ID from the actor_owners data already fetched by `listMyVports`.

8. **[LOW] Revoke object URLs** on unmount in UserProfileTab and VportProfileTab.

9. **[LOW] Remove `console.error` calls** or gate behind `import.meta.env.DEV`.

10. **[LOW] Clean up `initialTab` useMemo** — remove `[search]` dependency since it only matters on first render.
