## Execution Summary

### 1. Executive Summary

Client-centric, direct-to-Cloudflare-R2 architecture. All uploads go through `uploadToCloudflare()` + `buildR2Key()` (duplicated across VCSM and Wentrex). A minimal stateless Cloudflare Worker at `upload.vibezcitizens.com` receives multipart FormData, calls `R2_BUCKET.put()`, returns a public URL. Full URLs are stored in PostgreSQL via Supabase. All files are publicly accessible via `https://cdn.vibezcitizens.com/`. No AWS S3, no Supabase Storage, no signed URLs, no server-side validation, no orphan cleanup, no rate limiting.

---

### 2. Upload System Architecture

**Direct-to-R2 via Worker:**
- React component → file picker → optional client-side compression → FormData POST to `https://upload.vibezcitizens.com`
- Worker: `apps/VCSM/cloudflare-worker-upload/worker.js` + `apps/wentrex/cloudflare-worker-upload/worker.js` (identical logic, CORS differs)
- Worker extracts form fields, calls `env.R2_BUCKET.put()`, returns JSON `{url}`
- Auth: Bearer token in `Authorization` header; if CORS preflight fails → retry without headers (unauthenticated)
- Single R2 bucket: `post-media` (wrangler binding)
- Public CDN: `https://cdn.vibezcitizens.com/{objectKey}`
- Traffic (Next.js static) has NO upload capability

**Key Modules:**
- `apps/VCSM/src/services/cloudflare/uploadToCloudflare.js` — Worker request, CORS retry logic
- `apps/VCSM/src/services/cloudflare/buildR2Key.js` — key generation
- `apps/wentrex/src/features/services/cloudflare/uploadToCloudflare.js` — **duplicate**
- `apps/wentrex/src/features/services/cloudflare/buildR2Key.js` — **duplicate**

### 4. Storage / Bucket Strategy

- **Single bucket**: `post-media` (Cloudflare R2)
- **CDN**: `https://cdn.vibezcitizens.com/` (custom domain on R2 bucket)
- **Worker URL**: `https://upload.vibezcitizens.com` — hardcoded in both `uploadToCloudflare.js` files
- **Public access**: ALL files are public — no auth required to read any R2 object
- **No private/restricted bucket**
- **No tenant isolation** — VCSM and Wentrex both write to the same `post-media` bucket
- **No env vars** for bucket name or region — all config in `wrangler.toml`
- **Risk**: Private data (e.g., assignment submissions) is publicly accessible via CDN if key is known

### 5. Object Key / Folder Strategy

**Deterministic format**: `{prefix}/{ownerId}/{extraPath?}/{YYYY}/{MM}/{DD}/{timestamp}-{randomHex}.{ext}`

| Prefix | Owner | Use Case |
|--------|-------|----------|
| `vibes` | actorId | Social posts |
| `stories` | actorId | 24-drops |
| `vdrops` | actorId | Wanders |
| `avatar-photos` | userId | User avatar |
| `avatar-banners` | userId | User banner |
| `vport-avatar-photos` | vportId | Vport avatar |
| `vport-avatar-banners` | vportId | Vport banner |
| `flyers` | vportId | Flyer assets |
| `vox` | conversationId | Chat images |

**Key construction**:
- Timestamp: `Math.floor(Date.now() / 1000)` (seconds)
- Random: `crypto.getRandomValues()` → 3 bytes → 6 hex chars
- Extension: from MIME map; fallback from filename; final fallback `bin`

**Issues**:
- `buildR2Key()` is duplicated in VCSM and Wentrex (identical code)
- Chat (`vox/`) builds key inline, bypassing `buildR2Key()` — different random generation
- No centralized prefix registry — prefixes defined as string literals at call sites
- Avatar flows use `userId`/`vportId` (DB row IDs), not `actorId` — inconsistency with actor-based identity system

### 6. Media URL Generation Strategy

| Flow | URL Stored In DB | Format |
|------|-----------------|--------|
| Posts | `vc.post_media.url` | Full URL |
| User avatar | `profiles.photo_url` | Full URL |
| User banner | `profiles.banner_url` | Full URL |
| Vport avatar | `vport.profiles.avatar_url` | Full URL |
| Vport banner | `vport.profiles.banner_url` | Full URL |
| Menu item | `vport.actor_menu_items.image_url` | Full URL |
| Chat attachment | `chat.message_attachments.public_url` | Full URL |
| Chat attachment | `chat.message_attachments.storage_path` | R2 key |
| Submission files | `learning.submission_files.storage_path` | R2 key only |

**Composition formula**:
```js
// apps/VCSM/src/services/cloudflare/uploadToCloudflare.js
export function publicUrlForKey(key) {
  const clean = String(key || '').replace(/^\/+/, '');
  return `${R2_PUBLIC}/${clean}`;
}
// R2_PUBLIC = 'https://cdn.vibezcitizens.com'
```

**Issues**:
- Most flows store full URLs (tight coupling to CDN domain — if domain changes, all stored URLs break)
- Submission files store only key (good pattern) but it's inconsistent with all other flows
- Chat stores both URL and key (dual storage, keys never used on read)
- No single policy across the codebase

### 7. Validation / Safety Review

| Context | Size Limit | MIME Check | Dimension |
|---------|-----------|-----------|-----------|
| General upload (`classifyFile.js`) | 50 MB | Strict whitelist | None |
| Profile images | 5 MB | `startsWith('image/')` | 600px (via compression) |
| Chat images | Dynamic (compress triggers at 750KB iOS / 2MB desktop) | `startsWith('image/')` | 1600px (via compression) |
| Vibe/post | 50 MB (classifyFile) | MIME whitelist | 1080px (via compression) |

**Missing**:
- No server-side validation — Worker accepts any file
- No rate limiting on upload frequency or concurrency
- No malware scan
- No duplicate detection (same file → two R2 objects)
- No orphan prevention (failed DB write after upload = orphaned R2 object)
- No cleanup on entity delete (deleted post = orphaned R2 object)
- Inconsistent validation — posts use `classifyFile()` strict check; chat uses loose inline check
- No multi-part upload support for large videos

### 8. DB Persistence / Read Model Review

**Posts**: Legacy dual-write (`posts.media_url` + `post_media.url`). All reads use `post_media`. Legacy columns kept for "backwards compat." Should be removed post-migration.

**Profiles**: Full URL columns on `profiles` and `vport.profiles`. Simple, no transforms.

**Chat attachments**: Dual columns (`public_url` + `storage_path`). Reads use `public_url` only. `storage_path` stored but unused — over-engineering.

**Submission files**: R2 key stored only. Caller must compose URL. Correct pattern but only used here — inconsistent with rest of codebase.

**Shared pattern problem**: Full URL storage = CDN domain change requires DB migration. Key-only storage = caller must know CDN domain. No unified approach.

### 10. Recommendations Before Redesign

**Before any storage restructure:**

1. **De-duplicate** `uploadToCloudflare.js` + `buildR2Key.js` → move to `platform/` shared package; remove from both apps
2. **Consolidate chat key builder** — bring `vox/` prefix into `buildR2Key()` as named call; eliminate inline key generation
3. **Decide URL vs key storage policy** — pick one: store full URLs everywhere (simple) OR store keys + compose on read (future-proof for CDN changes). Current mix is the worst of both.
4. **Remove `posts.media_url` + `posts.media_type` legacy columns** — run migration, confirm all reads use `post_media`, drop columns
5. **Separate prefixes by owner type** — avatar flows use `userId`/`vportId` (DB IDs) but everywhere else uses `actorId`. Standardize on `actorId` for actor-owned content.
6. **Implement private bucket for restricted content** — submission files, private documents should not be on the public CDN; require signed URLs
