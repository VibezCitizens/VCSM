## Execution Summary

### What was completed

**SF-01 — Block guard: ctrlSubscribe (follow.controller.js)**
- Imported `ctrlGetBlockStatus` from `@/features/block` barrel
- Added bidirectional block check immediately after self-follow guard
- If `isBlocked === true`: throws `'Cannot follow a blocked actor'` — no follow row written, no request row written, no notification sent
- Removed dead `ctrlUnsubscribe` export (lines 101–123 of old file) — `dalDeactivateFollow` import also cleaned since it was only used by the dead function

**SF-01 — Block guard: ctrlSendFollowRequest (followRequests.controller.js)**
- Imported `ctrlGetBlockStatus` from `@/features/block` barrel
- Added same bidirectional block check after self-follow guard
- If `isBlocked === true`: throws `'Cannot send follow request to a blocked actor'` — no request row created, no notification sent

**SF-05 — Accept cache invalidation (followRequests.controller.js)**
- Imported `invalidateFeedFollowCache` from `@/features/feed/adapters/feedCache.adapter`
- After `dalInsertFollow` + `dalUpdateRequestStatus('accepted')` succeed: calls `invalidateFeedFollowCache(requesterActorId)` AND `invalidateFeedFollowCache(targetActorId)`
- Both actors' feed follow caches are busted immediately on accept

**SF-03 — assertingActorId on ctrlCancelFollowRequest (followRequests.controller.js)**
- Added `assertingActorId` parameter
- Guard: `assertingActorId !== requesterActorId` → throws before any DB touch
- Matches the existing pattern on accept and decline

**SF-02 — Block guard: useProfileGate (useProfileGate.js)**
- Imported `useBlockStatus` from `@/features/block` barrel
- Called as `useBlockStatus(viewerActorId, targetActorId)` (positional args — confirmed from source)
- `blockLoading` included in gate's `loading` derivation (no content flash while block resolves)
- `canView` rewritten: `isSelf || (!isBlocked && (!isPrivate || isFollowing))`
  - Self-view always passes
  - Any block in either direction → `canView = false` regardless of privacy/follow state
- `isBlocked` added to return object for UI to render `<BlockedState />` vs `<PrivateProfileNotice />`

**AF-01 — assertingActorId caller update (useFollowActorToggle.js)**
- Passed `assertingActorId: followerActorId` to `ctrlCancelFollowRequest`
- `followerActorId` is the session actor in this context — assumption is correct
