# BUGSBUNNY — Profile Routing Redirect Failure

**Date:** 2026-04-18
**Scope:** VCSM
**System:** Profile routing + canonical slug pipeline

## Symptom
User cannot access any profile (vport or user). Navigating to a profile
redirects back to /feed or loops without rendering content.

## BUG SUMMARY
Observed behavior: Clicking profile links → redirected to /feed or infinite skeleton
Expected behavior: Profile renders at UUID-free canonical URL
Where it occurs: ActorProfileScreen — all profile types
Visible error/logs: None (silent redirect)
Initial hypotheses: is_deleted column missing; slug resolution mismatch; RLS on vc.actors or vport.profiles

## Debugger Registry
BUILDDED BUGBUNNY ON FILE apps/VCSM/src/features/profiles/screens/ActorProfileScreen.jsx
BUILDDED BUGBUNNY ON FILE apps/VCSM/src/features/profiles/dal/readActorSeoData.dal.js
BUILDDED BUGBUNNY ON FILE apps/VCSM/src/features/profiles/controller/buildActorCanonicalSlug.controller.js

## Root Cause — Confirmed

BREAK POINT: Round-trip slug mismatch.
- Forward: `vport.profiles.slug` is NULL → canonical derived from `normalizeSlugPart(name)` e.g. `"tyba-restaurant"`
- Reverse: `resolveActorBySlugOrUsernameDAL("tyba-restaurant")` queries `vport.profiles WHERE slug = 'tyba-restaurant'` → NO MATCH → notFound=true → redirect to /feed

## Fix Applied — Pass 1 (Round-trip state handoff)

FILES MODIFIED:
- apps/VCSM/src/features/profiles/hooks/useActorSlugRedirect.js
  - Added `resolvedActorId` param, passes it as `state: { actorId }` on navigate()
- apps/VCSM/src/features/profiles/screens/ActorProfileScreen.jsx
  - Added `useLocation()` import
  - Reads `location.state?.actorId` as `actorIdFromState`
  - `slugToResolve` skips DB resolution when `actorIdFromState` is present
  - `resolvedActorId = uuidFromParam ?? actorIdFromState ?? actorIdFromSlug ?? null`
  - Passes `resolvedActorId` to `useActorSlugRedirect`
  - Updated probe to show `actorIdFromState`

REASONING: UUID URLs already have the actorId. When redirecting to the canonical slug URL,
passing that actorId in router state eliminates the need for the reverse-resolution DB query.
This is correct regardless of whether `vport.profiles.slug` is populated in the DB.

RISK: Low. State is only consumed once at the destination. Direct slug navigation (bookmarks,
shared links) still uses the DB resolution path.

FOLLOW-UP: When vport.profiles.slug is populated for all actors, `resolveActorBySlugOrUsernameDAL`
will work for direct slug navigation too. Until then, redirect-initiated navigation works via state.

Logan update required: Yes — profile routing doc needs to reflect state handoff pattern
Engine audit update likely: No

## Fix Applied — Pass 2 (Strict routing + self redirect)

PROBE FINDINGS:
- /profile/self: resolvedActorId=null, canonicalSlug=null → shows ⚠ REDIRECTING TO /feed
  Cause: "self" early return rendered content at /profile/self instead of redirecting,
  AND actorIdForSelf was never fed into the resolvedActorId pipeline
- /profile/architect1: slugToResolve=null (bypassed via actorIdFromState), strict validation skipped

FILES MODIFIED:
- apps/VCSM/src/features/profiles/screens/ActorProfileScreen.jsx
  - Removed useLocation / actorIdFromState bypass entirely
  - Added actorIdForSelf = isSelf ? identity?.actorId : null
  - slugToResolve: ONLY skips for "self" and UUID (no state bypass)
  - resolvedActorId priority: actorIdForSelf ?? uuidFromParam ?? actorIdFromSlug
  - Removed "self" early return — redirect handles self → canonical
  - Updated probe: shows isSelf + actorIdForSelf, removed actorIdFromState
- apps/VCSM/src/features/profiles/hooks/useActorSlugRedirect.js
  - Updated comment: state is informational only, not a bypass

BEHAVIOR AFTER FIX:
- /profile/self → resolvedActorId=identity.actorId → canonical computed → redirect → /profile/{canonical}
- /profile/architect1 → slugToResolve="architect1" → DB resolution → actorId found → render
- /profile/{uuid} → uuidFromParam extracted → canonical computed → redirect → /profile/{canonical}
- All routes strictly validated through their declared source only

OPEN DATA ISSUE: vport.profiles.slug is NULL for some actors. Reverse resolution
(resolveActorBySlugOrUsernameDAL) will fail for name-derived canonicals on direct navigation.
This is a data integrity issue — slugs must be written to DB for all vport actors.
BUGSBUNNY does not modify DB directly. DB fix required separately.
