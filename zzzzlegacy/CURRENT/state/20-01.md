## Execution Summary

### What was completed

**Migration** βÄî `apps/VCSM/supabase/migrations/20260420010000_vport_soft_delete_rpc.sql`

Added two columns to `vport.profiles`:
- `deleted_at TIMESTAMPTZ DEFAULT NULL`
- `deleted_by_actor_id UUID DEFAULT NULL`

Created `vport.soft_delete_vport(p_vport_id uuid)`:
- Verifies `auth.uid()` owns the vport via `owner_user_id`
- Checks `is_deleted = false` (idempotency guard)
- Sets `is_deleted=true, is_active=false, deleted_at=now(), deleted_by_actor_id=actor_id`
- Returns `jsonb { ok, vport_id, actor_id }`
- SECURITY DEFINER, search_path locked, REVOKE from public

Created `vport.restore_vport(p_vport_id uuid)`:
- Verifies ownership + `is_deleted = true`
- Clears `is_deleted=false, is_active=true, deleted_at=NULL, deleted_by_actor_id=NULL`
- Returns `jsonb { ok, vport_id, actor_id }`

**Design decision:** `vc.actor_owners.is_void` is NOT touched βÄî the `one_void_per_user` UNIQUE constraint would block users with multiple actors. Soft delete is enforced entirely via `vport.profiles.is_deleted` + the existing RLS policy `USING (is_active = true AND is_deleted = false)` which already filters public reads.

**DAL βÄî `account.write.dal.js`**
- `dalDeleteMyVport`: updated from `supabase.rpc('delete_my_vport')` (wrong client, missing function) βÜ£ `vportSchema.rpc('soft_delete_vport')` with error message normalization
- `dalRestoreVport`: new function wrapping `vport.restore_vport`
- `dalDeleteOwnedVportById`: kept but marked deprecated

**DAL βÄî `vport.core.dal.js`**
- `softDeleteVport(vportId)`: calls `vport.soft_delete_vport` RPC, normalizes errors
- `restoreVport(vportId)`: calls `vport.restore_vport` RPC, normalizes errors
- Both exported on default object
