# Planning — April / 09 / Sequence 05

TASK — ENFORCE FULL APP-WIDE IDENTITY SWITCH CONSISTENCY BETWEEN CITIZEN AND VPORT

OBJECTIVE
Make actor switching fully consistent across the entire app.

When switching between:
- citizen/user
- vport/business

the active identity must fully switch everywhere in the application, not just in the identity panel or global snapshot.

PRIMARY GOAL
After a switch completes, every screen, provider, hook, query, component, cache, and action path must use the new active actor and must never continue rendering or acting as the previous actor.

THIS MUST APPLY TO ALL SCREENS
That includes, but is not limited to:
- feed
- profile
- vport profile
- booking/calendar
- reviews
- messages/chat if applicable
- tabs
- navbar/header identity displays
- permissions/CTA logic
- create/post flows
- notifications
- dashboards
- any viewer-dependent data source

CORE RULE
After switch completes, the application must behave as if the newly selected actor is the only active actor.

There must be no stale previous-actor state left in:
- UI
- query caches
- stores
- providers
- derived viewer models
- route-scoped state
- memoized objects
- request payload builders

SUCCESS RULE
After switch settles, all actor-dependent layers must match the same active identity:

- engine resolved actor
- committed identity state
- global identity snapshot
- feed viewer
- screen-level viewer objects
- query keys
- fetched data
- permissions
- visible CTAs
- create/update actions

If the new actor is citizen:
- every screen must behave as citizen

If the new actor is vport:
- every screen must behave as vport

NEVER ALLOW
- global identity = citizen while some screens still use vport
- global identity = vport while some screens still use citizen
- stale feed viewer
- stale booking permissions
- stale review permissions
- stale profile data
- stale query cache for previous actor
- UI labels from old actor after switch

## Execution Summary

### Slice 1 — Audit Results

Audited all 59 `useIdentity()` consumers across the VCSM app. Searched for:
- `useRef` holding actorId — **none found**
- `useState` initialized from identity — **none found**
- `useMemo` with missing identity deps — **none found** (all have correct deps)
- `useEffect` with empty deps `[]` using identity — **none found** (all empty-dep effects are UI handlers or cleanup, not identity-dependent)
- Query caches without actorId keys — **not applicable** (no centralized query cache like React Query)

**Verdict: The codebase is architecturally clean.** All 59 consumers derive identity from React context via `useIdentity()` on every render. When `setIdentity(nextIdentity)` is called during a switch, React context propagation causes all consumers to re-render with the new identity. No stale state patterns exist.

The `const actorId = identity?.actorId ?? null` pattern (used in 20+ files) is correct — it re-derives from the context value on every render cycle, not from cached local state.

### Slice 2 — Dev Consistency Check

Since no stale patterns were found, no fixes were needed. Instead, added a dev-only regression detection system:

**Created:** `debuggers/identity/useActorConsistencyCheck.js`
- Dev-only hook that compares a feature's local actorId against global identity
- Logs `[ACTOR_MISMATCH]` console warning if they diverge
- Fires on every identity/actorId change

**Wired into 4 high-risk screens:**
- `CentralFeedScreen.jsx` — `useActorConsistencyCheck('feed', actorId, identity?.kind)`
- `useVportBookingView.js` — `useActorConsistencyCheck('booking', viewerActorId, identity?.kind)`
- `VportReviewsView.jsx` — `useActorConsistencyCheck('reviews', sessionActorId, identity?.kind)`
- `ActorProfileScreen.jsx` — `useActorConsistencyCheck('profile', identity?.actorId, identity?.kind)`
