## Execution Summary

### What was completed

**V-001 (P0 — HIGH) — `raw: row` + `profile_id` leak — RESOLVED**

Fix 1 — `readVportPublicDetails.rpc.dal.js`
Removed `profile_id` from the column select list. `actor_id` is the canonical identity reference
already selected. `profile_id` is a banned identity surface (no public use). `lat`, `lng`,
`social_links`, `location_text` remain in the select — they are consumed internally by
`buildDirectionsUrl` / `getSocialLink` in the model and are never returned in output.

Fix 2 — `vportPublicDetails.model.js`
Removed `raw: row` from the details return object. `address: row.address ?? null` was already
in the explicit field mapping (line 202). Added explanatory comment noting raw removal is
intentional (VENOM V-001). All publicly required fields are now explicitly mapped — future
view columns cannot silently leak via `raw.*`.

Fix 3 — `VportPublicMenuView.jsx:52`
Removed `?? details?.raw?.address` fallback from the `address` property in the profile useMemo.
`details.address` is now explicitly mapped in the model — the raw fallback was dead and
constituted an identity leak surface while `raw: row` existed.

**V-005 (P1 — MEDIUM) — `onBack` UUID in `/vport/${actorId}` — RESOLVED**

`VportActorMenuFlyerView.jsx` — Replaced the `navigate('/vport/${actorId}')` fallback branch
in `onBack` with the canonical slug pattern: `navigate('/profile/${canonicalSlug}')` when slug
is resolved, `/actor/${actorId}/menu` (internal redirect) when not yet available.
`canonicalSlug` is already resolved in this view (used for `menuUrl`) — the fix adds it to the
`onBack` useCallback dependency array.

**V-004 (P1 — MEDIUM) — UUID encoded in printed QR before slug resolves — RESOLVED**

`VportActorMenuFlyerView.jsx` — Added `UUID_RE` and `isQrSafe` constants after `canonicalSlug`
hook call (same UUID guard pattern as `VportPublicMenuQrView` and `VportPublicReviewsQrView`).
Gates:
  - Flyer body: shows "Preparing flyer…" skeleton when `!isQrSafe`
  - Print button: disabled with "Preparing…" label when `!isQrSafe`
Ensures no QR code or print surface is available until canonical slug resolves to a confirmed
non-UUID value. Eliminates the race condition where a quick print would encode actorId in
permanent physical materials.

**V-003 (P1 — MEDIUM) — `useQrLinks` identity surface violation — RESOLVED**

Fix 1 — NEW `features/booking/dal/getVportProfileIdByActorId.dal.js`
Resolves actorId → `vport.profiles.id` via `vportClient.from("profiles").select("id").eq("actor_id", actorId)`.
Mirrors the established `getVportSlugByActorId.dal.js` pattern. Returns null on error — callers
handle null gracefully.

Fix 2 — `features/booking/hooks/useQrLinks.js`
Refactored from `{ organizationId, profileId, enabled }` to `{ actorId, enabled }`.
Uses `useRef` to cache the resolved `profileId` across renders without triggering re-renders.
Resets the ref when `actorId` changes (separate useEffect) to force fresh resolution when the
actor switches. The `organizationId` and `profileId` params are fully removed — no caller can
provide these banned identity surfaces.

Fix 3 — `features/profiles/kinds/vport/screens/booking/components/BookingQrLinksPanel.jsx`
Updated `useQrLinks` call from `{ organizationId: null, profileId: null, enabled: false }` to
`{ actorId, enabled: !!actorId }`. Removed the TODO block (adapter is now built). Updated
component JSDoc to document the resolved identity contract. QR links panel is now functional
for authenticated VPORT owners.

**V-002 (DB — MEDIUM) — RLS inspection — FINDING REPORTED, no code change**

Migration `20260503052543` policy:
```sql
CREATE POLICY qr_links_select_authenticated ON vport.qr_links
  FOR SELECT TO authenticated USING (is_active = true);
```
Intentionally broad — any authenticated user can read any active QR link (slug routing design).
The list functions (`listQrLinksByOrganization`, `listQrLinksByProfile`) add a WHERE filter but
have no app-layer ownership gate. Any authenticated caller who knows an organizationId or
profileId can enumerate all QR links for that entity (labels, scan counts, destination paths).
Follow-up: Wolverine + Carnage pass to add `requestActorId` + `assertActorCanManage*` to
`listQrLinks*` controllers in the booking engine.
