## Architecture

### Canonical Flow
```
Any feature needing actor data
  → engines/hydration/src/hydrate.js (single entry point)
    → extractActorIds (from any row shape)
    → dedup against store (skip fresh actors)
    → call getActorSummariesByIds DAL (single RPC source)
    → normalize to canonical shape (one function)
    → upsert into Zustand store (safe merge, no null-overwrites)
    → return results
```

### Store Shape (per actor)
```js
{
  id: string,
  kind: 'user' | 'vport',
  displayName: string | null,
  username: string | null,
  photoUrl: string | null,
  bannerUrl: string | null,
  bio: string | null,
  vportName: string | null,
  vportSlug: string | null,
  vportAvatarUrl: string | null,
  vportBannerUrl: string | null,
  // Freshness metadata
  _hydratedAt: number (Date.now()),
  _stale: boolean (computed from TTL),
  // Legacy compat (generated from canonical)
  display_name, photo_url, banner_url, actor_id
}
```

### Freshness Strategy
- `_hydratedAt` timestamp on each actor record
- `STALE_AFTER_MS = 5 * 60 * 1000` (5 minutes)
- `isStale(actor)` helper: `Date.now() - actor._hydratedAt > STALE_AFTER_MS`
- Hydration skips actors that are fresh (unless `force: true`)
- No automatic background refresh — staleness is checked on next hydration request
- No UI flicker — stale data is shown while refresh happens in background

### Safe Upsert Rules
- Never overwrite a non-null value with null
- Merge new fields into existing record
- Always update `_hydratedAt` on successful hydration
- Support `force: true` to overwrite everything

### Chat Integration
- In `chat/setup.js`, wrap `getActorSummariesByIds` to also call `upsertActors` on the global store
- This means every time the chat engine resolves actors, the global store also gets updated
