## VCSM PERFORMANCE STABILIZATION AUDIT

### Section 1 — Feed ownership audit

**Status: ALREADY CLEAN — no changes required**

`useFeed.js` (line 313-319) has a `didInitialFetchRef` guard that prevents double-fetch on the initial load:

```js
useEffect(() => {
  if (!viewerActorId) return;
  if (didInitialFetchRef.current) return;
  didInitialFetchRef.current = true;
  fetchPosts(true);
}, [viewerActorId, fetchPosts]);
```

`fetchPosts` is memoized with `[viewerActorId, realmId]` — stable. The reset effect (line 115-129) clears `didInitialFetchRef.current = false` on actor/realm change, so identity switch correctly re-triggers.

`PostFeedScreen` at `/posts` is a different route from `CentralFeed` at `/feed`. React Router renders them exclusively — they are never simultaneously mounted. `useFeed.adapter.js` is a direct re-export of `useFeed.js`.

Feed is already single-owner, request-versioned, and well-guarded. **Goal A is already achieved.**

Feed logging already distinguishes `fresh ? 'Fresh fetch' : 'Paginate'`. No additional logging changes needed.

### Section 2 — Bootstrap/session duplication audit

**Current state:** No shared Zustand store for session-level unread counts.

BottomNavBar mounts two independent hooks on every route change:
- `useNotiCount({ actorId, pollMs: 45_000 })` — polls notification engine every 45s, realtime sub on inbox items
- `useUnreadBadge({ actorId, refreshMs: 15_000 })` — polls `chat.inbox_entries` every 15s, realtime sub on chat

**Mitigation already in place:**
- Both hooks have module-level in-memory caches (15s TTL for notifications, 10s TTL for chat)
- Both have inflight dedup — parallel calls coalesce into one network request
- Both have realtime Supabase channel subscriptions that fire on changes

**What's missing:**
- No shared store — if any other component wanted unread counts, it would create a third subscription
- On every actor switch, both hooks restart subscriptions and fire initial queries
- No explicit invalidation surface — write paths (marking read, new message) dispatch `noti:refresh` events, which both hooks listen to independently

**Platform tables** (`user_app_actor_links`, `user_app_state`, `user_app_preferences`, `user_consents`):
- Resolved through the identity engine (`resolveAuthenticatedContext`) — NOT directly queried client-side
- Not duplicated — the profiler data from the task description predates engine integration
- No action needed for these
