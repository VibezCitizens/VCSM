## Execution Summary

### Key Finding: @notifications alias routes to runtime/index.js, NOT the engine

The `@notifications` vite alias resolves to:
`apps/VCSM/src/features/notifications/runtime/index.js`

This is a VCSM-local reimplementation, NOT the engine at `engines/notifications/`. The engine exists but is unused by VCSM's notification feature. All publish, read, and count operations go through `runtime/index.js` + `notificationRuntime.dal.js`.

### ACTUAL RESOLUTION (2026-05-03)
Root cause confirmed: missing EXECUTE grants on 3 recreated RPCs — `upsert_rendered`, `insert_inbox_item`, `update_recipient_status`. `create_event` and `insert_recipients` already had grants. Pipeline was blocked at the render/deliver stage; events + recipients rows existed but rendered/inbox_items rows never written, status stuck at 'pending'. Fix: GRANT EXECUTE on those 3 functions + NOTIFY pgrst. Post-fix: 32 notifications delivered, pipeline operational.

The audit's RC1 hypothesis (RPCs missing entirely) was directionally correct — the actual failure was EXECUTE permissions missing rather than functions absent.

Logan doc updated: `vcsm/notifications/vcsm.notifications.pipeline.md` Section 10 new incident, Section 11 status updated.

### Slice 4 — Root Cause Candidates (ranked)

RC1 — CRITICAL (PUBLISH FAILS): RPC functions not created in `notification` schema
- 5 RPCs required: `create_event`, `insert_recipients`, `upsert_rendered`, `insert_inbox_item`, `update_recipient_status`
- User provided TABLE DDL only — no function DDL
- If RPCs don't exist: `publishVcsmNotification()` silently returns `false`; 0 rows ever written; inbox always empty
- Verified via: `SELECT proname FROM pg_proc WHERE pronamespace = 'notification'::regnamespace`

RC2 — CRITICAL (READS RETURN EMPTY): No RLS policies or schema grants
- Direct table SELECT operations in runtime DAL use authenticated role
- Without `GRANT USAGE ON SCHEMA notification TO authenticated` + SELECT grants + policies → reads fail or return []
- Verified via: `SELECT tablename, rowsecurity FROM pg_tables WHERE schemaname = 'notification'`
- Verified via: `SELECT tablename, policyname FROM pg_policies WHERE schemaname = 'notification'`

RC3 — HIGH (DATA GAP): `notification.event_types` empty — no event seeds
- Not a blocker for current runtime (runtime's `publishEvent` doesn't query event_types)
- Needed for future preference evaluation and channel routing
- Fix: seed 14 event_types + 3 event_categories
