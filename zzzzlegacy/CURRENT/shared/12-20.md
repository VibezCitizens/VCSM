## Execution Summary

### What Was Done

Full end-to-end trace of all 7 VCSM bottom navigation icons. Each icon was traced from its click handler in `BottomNavBar.jsx` through route resolution, page component loading, controller orchestration, DAL queries, engine usage, state store interactions, and side effects (polling, realtime, caching).

### Output

Logan documentation written to: `logan/navigation/vcsm.navigation.audit.md`

### Key Findings

1. **BottomNavBar is always mounted** (CSS-hidden on chat sub-screens, auth, learning routes) to preserve realtime subscriptions and polling. Two badge hooks (`useNotiCount` at 45s, `useUnreadBadge` at 15s) run continuously.

2. **Upload is the only imperative navigation** -- uses a `<button>` with `navigate('/upload')` instead of `NavLink`.

3. **Feed is the heaviest route** -- triggers 6+ parallel DAL calls per page via `fetchFeedPagePipeline`, plus background actor hydration through the hydration engine.

4. **Chat inbox delegates to the `@chat` engine** -- VCSM's `useInbox` hook is a thin wrapper around the shared chat engine's `useInbox`.

5. **Notifications use the `@notifications` engine** -- all count, read, and mark-seen operations go through the shared notification engine. Block filtering and follow-request validation happen at the VCSM controller layer.

6. **Profile route branches on actor kind** -- resolves `user` vs `vport` via `useActorKind` hook, then dispatches to `PROFILE_KIND_REGISTRY[kind]`.

7. **Three realtime Supabase channels** are active at all times: `noti-badge-{actorId}` (2 listeners), `chat-badge-{actorId}` (3 listeners), plus chat engine internal channels.

8. **Explore has disabled discovery blocks** -- `SHOW_EXPLORE_DISCOVERY_BLOCKS = false` means the idle explore view only shows `OnboardingCardsView`.

9. **Settings is the lightest route** -- no polling, no realtime, lazy-loads tab content.
