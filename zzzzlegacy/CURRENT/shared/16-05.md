# Planning — April / 16 / Sequence 05

## Task

Fix the Vport portfolio insert RLS failure.

**Bug:** `new row violates row-level security policy for table "portfolio_items"`

**Root cause confirmed:** `createItem.controller.js:49` passes `createdByActorId: actorId` where `actorId` is the vport actor id (from URL params). `vc.current_actor_id()` in the RLS policy returns the user actor id from `auth.uid()`. These never match, so `created_by_actor_id = vc.current_actor_id()` always fails.

**Fix:** Set `createdByActorId: null` in the engine controller. INSERT policy explicitly allows null.

## Execution Summary

**What was completed:**

1. `engines/portfolio/src/controller/createItem.controller.js`
   - Removed `createdByActorId: actorId` from the insert payload — changed to `createdByActorId: null`
   - Added pre-insert debug emit via `getDebugReporter()` covering: owner check result, profile lookup result, and full insert payload
   - Added structured `insertPayload` variable for clarity and to make payload composition auditable

2. `apps/VCSM/src/features/portfolio/setup.js`
   - Added `portfolioTraceStore` — a lightweight pub/sub event store (dev-only) that accumulates engine trace events and notifies subscribers
   - Wired `debugReporter` into `configurePortfolioEngine` behind `import.meta.env.DEV` guard — routes all engine trace events to the store
   - Production builds: `debugReporter: null` — no overhead

3. `apps/VCSM/src/features/dashboard/vport/screens/VportDashboardPortfolioScreen.jsx`
   - Added `PortfolioBugsBunnyPanel` component (DEV only, never renders in production)
   - Panel capabilities:
     - "Run Probe" button: live queries identity, `vport.profiles`, `vport.profile_actor_access`, `vc.actor_owners`, auth session
     - Runs assertions: actorId present, kind === vport, profileId resolved, profile.actor_id matches, profile not deleted
     - Shows expected insert payload at runtime
     - Live tail of engine trace events (fed from `portfolioTraceStore`)
     - Color-coded: green = pass, red = fail, yellow = warning
   - Mounted in screen render behind `import.meta.env.DEV` guard

**Why done this way:**
- `createdByActorId: null` removes the ambiguity about which actor id `vc.current_actor_id()` resolves to at DB time
- The INSERT policy allows null explicitly — no policy change needed
- The BugsBunny panel makes it possible to see exactly what `profile_id` is being resolved and whether `profile_actor_access` contains a valid entry for the active actor
- If `actor_can_manage_profile` is still failing, the probe output will reveal whether the profile lookup or access rows are the problem

**Blockers / Follow-up:**
- If insert still fails after nulling `createdByActorId`, run the probe to check `profile_actor_access` rows — the `actor_can_manage_profile` DB function likely checks those rows and they may be missing or have wrong status
