# Planning — April / 09 / Sequence 23

> Awaiting tasks.
Goal

Create a legal document versioning + re-consent engine that does this:
	1.	publishes new legal versions
	2.	detects outdated user consent
	3.	blocks app access when needed
	4.	shows re-consent UI
	5.	records new acceptance
	6.	releases user into app
	7.	preserves full audit history

⸻

Engine architecture

You want 5 parts:

1. Legal document resolver

Responsible for loading the current active legal docs for the app.

Inputs
	•	appKey or appId

Outputs
	•	active terms_of_service
	•	active privacy_policy
	•	later: location_consent, marketing_email

Responsibility

This is the single source of truth for:
	•	current active doc version
	•	current doc id
	•	current URL

Example return shape

{
  terms_of_service: {
    id: '...',
    version: '1.1',
    content_url: '/legal/terms-of-service?v=1.1',
    title: 'VCSM Terms of Service'
  },
  privacy_policy: {
    id: '...',
    version: '1.0',
    content_url: '/legal/privacy-policy?v=1.0',
    title: 'VCSM Privacy Policy'
  }
}
⸻

2. User consent resolver

Responsible for loading the user’s latest accepted consent per required document type.

Inputs
	•	userId
	•	appId

Outputs

Latest accepted row for each type.

Example return shape{
  terms_of_service: {
    legal_document_id: '...',
    consent_version: '1.0',
    accepted_at: '...',
    accepted_via: 'signup'
  },
  privacy_policy: {
    legal_document_id: '...',
    consent_version: '1.0',
    accepted_at: '...',
    accepted_via: 'signup'
  }
}
⸻

3. Consent diff engine

This is the core brain.

It compares:
	•	active legal docs
	•	latest user accepted docs

And decides whether user is compliant or blocked.

Inputs
	•	activeLegalDocuments
	•	userConsents

Output{
  isCompliant: false,
  missingTypes: [],
  outdatedTypes: ['terms_of_service'],
  requiredActions: [
    {
      consent_type: 'terms_of_service',
      required_version: '1.1',
      current_version: '1.0',
      legal_document_id: '...',
      content_url: '/legal/terms-of-service?v=1.1'
    }
  ]
}Rules

A document requires action if:
	•	no accepted consent exists for that type
	•	accepted version != active version
	•	accepted row is revoked
	•	accepted row belongs to wrong app/doc context

⸻

4. Legal gate controller

This decides whether app access is allowed.

Inputs
	•	authenticated user
	•	app context
	•	consent diff result

Behavior
	•	if compliant → continue into app
	•	if not compliant → redirect to legal consent gate

This should run:
	•	on login
	•	on app bootstrap
	•	on token restore
	•	optionally after settings/legal refresh

Output

Either:
	•	ALLOW_ACCESS
	•	REQUIRE_RECONSENT

⸻

5. Re-consent writer

This writes the new consent records after the user accepts.

Inputs

For each required action:
	•	user_id
	•	app_id
	•	legal_document_id
	•	consent_type
	•	consent_version
	•	accepted_via = 'reconsent'
	•	locale
	•	ip_address
	•	user_agent

Behavior

Insert a new row, never update old rows.

Output
	•	success
	•	failure with reason

⸻

User flow from 1 to 7

Step 1. New legal version gets published

The engine does not publish content itself.
It only reacts after a new active legal doc exists.

At this point your app now has:
	•	old user consent
	•	new active doc version

That creates the mismatch.

⸻

Step 2. User opens app

Bootstrap runs:
	•	resolve app
	•	fetch active legal docs
	•	fetch user latest consents
	•	compare

⸻

Step 3. Diff engine finds mismatch

Example:
	•	active terms = 1.1
	•	user terms = 1.0

Result:/legal/reconsent
isCompliant = false
outdatedTypes = ['terms_of_service']


⸻

Step 4. Legal gate blocks app

Instead of loading home/feed/dashboard, route user to:
/legal/reconsent
This screen should receive:
	•	what document types need action
	•	current version
	•	URL to open
	•	titles

⸻

Step 5. User reviews and accepts

UI should show:
	•	message that legal docs were updated
	•	each required document
	•	clickable links
	•	accept button or checkbox flow

For example:
	•	updated Terms of Service
	•	updated Privacy Policy

If both are outdated, collect both in one screen.

⸻

Step 6. Re-consent writer inserts rows

When user clicks accept:
	•	gather locale
	•	gather user agent
	•	gather IP if available
	•	write new consent rows

Important:
	•	write only required types
	•	do not duplicate already current types
	•	do not edit old history

⸻

Step 7. Re-run check and release user

After insert succeeds:
	•	re-run consent status
	•	if now compliant → allow app
	•	route to previous destination or default home

Engine rules

Rule 1

Only compare against active legal docs.

Rule 2

Only latest accepted consent counts.

Rule 3

revoked_at means not valid anymore.

Rule 4

History is append-only.

Rule 5

No app access until required docs are accepted.

Rule 6

Do not rely on “continued use means consent” for material changes.

Rule 7

Never overwrite old document content.

## Execution Summary

Built the full legal consent versioning + re-consent engine per the planning spec.

- `buildConsentComplianceStatus({ activeDocs, userConsents })`
- Returns `{ isCompliant, missingTypes, outdatedTypes, requiredActions }`

- Fetches active docs → fetches user consents → runs compliance engine → returns decision

### Flow (Steps 1-7 from planning spec)
1. New legal doc version published → active version changes in DB
2. User opens app → ProtectedRoute renders → useLegalConsent runs
3. `resolveLegalGateForSession` → compliance engine finds mismatch
4. Returns `REQUIRE_RECONSENT` → ConsentGateScreen shown
5. User reviews and checks each updated document
6. `acceptRequiredConsents` writes new consent rows with `accepted_via: 'reconsent'`
7. Hook re-renders → `requiresConsent = false` → app loads
