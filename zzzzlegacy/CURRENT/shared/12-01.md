# Planning — April / 12 / Sequence 01

## Task

Build a production-grade neutral notification engine around the new `notification` schema.

### Summary

Design and implement the first version of a domain-neutral notification engine that powers creation, routing, rendering, delivery tracking, inbox state, and preferences. Must serve `vc`, `learning`, and future platform domains cleanly.

### Schema Scope (Source of Truth)

- notification.event_categories
- notification.event_types
- notification.events
- notification.recipients
- notification.rendered
- notification.delivery_attempts
- notification.preferences
- notification.inbox_items
- notification.templates

### Core Engine Responsibilities

1. **Event creation** — Normalized notification events from product actions
2. **Recipient resolution** — Determine recipients from event context (actor/user/app_account/broadcast)
3. **Preference evaluation** — Decide channels per recipient (mute, frequency, quiet hours)
4. **Rendering layer** — Map event types to templates, per-channel rendering, locale support
5. **Delivery orchestration** — in-app inbox, email/sms/push/webhook delivery attempts
6. **Inbox state model** — seen, read, opened, dismissed, badge_counted, archived, snoozed
7. **Domain neutrality** — No VC-only assumptions, support vc/learning/vport events
8. **Engine boundaries** — Clear separation between publishing, resolution, rendering, delivery

### Architectural Constraints

- Domain-neutral, reusable, event-driven
- Multiple recipient kinds: actor, user, app_account, broadcast_group
- Multiple channels: in_app, email, sms, push, webhook
- Modular and replaceable delivery providers
- Preference-aware before delivery
- Traceable and auditable
- Compatible with existing `vc.notifications` (bridge/migration strategy)

## Codebase Audit Results

### Current Notification System

All notifications use `vc.notifications` table with triggers as the primary creation path:
- `trg_actor_follows_notify` → follow
- `trg_follow_request_notify` → follow_request
- `trg_follow_accept_notify` → follow_request_accepted
- `trg_post_comments_notify` → comment / comment_reply
- `trg_comment_like_notify` → comment_like
- `trg_post_reactions_notify` → like / dislike / post_rose
- `trg_post_rose_gifts_notify` → post_rose
- `trg_post_mentions_notify` → post_mention

Client-side creation via `dalInsertNotification()` (RPC with direct INSERT fallback):
- Booking controllers (createBooking, confirmBooking, cancelBooking)
- Review controller (ctrlSubmitReview)

Read path: NotificationsScreen → useNotifications → Notifications.controller → DALs → vc.notifications
Badge: useNotiCount (bell) + useUnreadBadge (chat) — separate systems
Realtime: Postgres changes subscriptions on vc.notifications + chat.inbox_entries

### Key Finding: notification.* schema does NOT exist yet
No `notification.*` tables found in the database. All references are aspirational from the planning task. Engine code can be built but will need DB migration before runtime.

### No notification engine exists
`engines/notifications/` does not exist. Engine extraction plan documented in logan but not yet implemented.

## Execution Summary — Slices 1-3

### What Was Completed

Built the complete notification engine core at `engines/notifications/` with:

**Slice 1 — Scaffold + Types + Config (complete)**
- Engine root: `CLAUDE.md`, `index.js`
- DI config: `src/config.js` — supabaseClient, resolveRecipients, resolveActorCard, debugReporter
- Domain events: `src/events.js` — 12 event types (published, resolved, rendered, delivered, inbox state changes)
- Types: `src/types/index.js` — 22 JSDoc typedefs covering all schema entities + composed InboxNotification
- Models: 7 model files (Event, Recipient, InboxItem, Rendered, Template, Preference, DeliveryAttempt)
- Vite alias: `@notifications` added to `vite.config.js`

**Slice 2 — DAL Layer (complete)**
- `dal/events.write.dal.js` — insert event
- `dal/events.read.dal.js` — get event by ID
- `dal/eventTypes.read.dal.js` — get/list event types
- `dal/recipients.write.dal.js` — insert recipients, update status
- `dal/recipients.read.dal.js` — list by event, get by ID
- `dal/inbox.write.dal.js` — insert inbox item, mark seen/read, dismiss, archive
- `dal/inbox.read.dal.js` — count unseen, list inbox recipients, get inbox items by IDs
- `dal/rendered.write.dal.js` — upsert rendered content, read by recipient IDs
- `dal/templates.read.dal.js` — find template with priority cascade (exact → fallback locale → fallback channel → base)
- `dal/preferences.read.dal.js` — list preferences by actor, get specific preference
- `dal/deliveryAttempts.write.dal.js` — insert attempt, update attempt status

All DALs use explicit column selection, `notification` schema, and trace support.

**Slice 3 — Core Services + Controllers (complete)**
- `services/templateRenderer.service.js` — template lookup with cascade + `{{variable}}` interpolation
- `services/preferenceEvaluator.service.js` — preference cascade (specific → general → default), mute/frequency/quiet hours
- `services/deliveryOrchestrator.service.js` — in_app creates inbox items, external channels create delivery attempts
- `controller/publishEvent.controller.js` — full pipeline: validate → insert event → resolve recipients → evaluate preferences → render → deliver
- `controller/getInbox.controller.js` — fetches inbox with parallel hydration (events + rendered + inbox state), auto-mark-seen
- `controller/inboxState.controller.js` — markSeen, markRead, dismiss, archive
- `controller/countUnread.controller.js` — count unseen non-dismissed non-archived inbox items
- `adapters/index.js` — public API surface exporting all controllers, services, and models
