# 12-27

> Feed Pipeline Duplicate Execution Fix

## Audit Findings

### Root Cause: Two compounding issues

**1. Unstable `fetchPosts` callback reference (useFeed.js)**

The `fetchPosts` useCallback depended on `[viewerActorId, realmId, upsertActors]`. While Zustand store methods are typically stable, including `upsertActors` in the dependency array meant that any theoretical reference change would recreate `fetchPosts`, which would re-trigger the initial fetch effect (`[viewerActorId, realmId, fetchPosts]`). More critically, this dependency pattern couples the callback's stability to the store subscription model, making it fragile.

**2. IntersectionObserver churn in CentralFeedScreen.jsx**

The `observeMore` callback depended on `[posts.length, hasMore, loading, fetchPosts]`. Every time the initial fetch completed:
- `posts.length` changed (0 to 3)
- `loading` changed (true to false)
- `observeMore` recreated
- `useEffect(() => observeMore(), [observeMore])` fired, creating a NEW IntersectionObserver
- The new observer immediately fired because the sentinel was within the 600px rootMargin
- This triggered `fetchPosts(false)` — an immediate pagination request right after the initial fetch

Each pipeline call fires 9+ DAL queries. Two pipeline executions per load = 18+ duplicate query groups, matching the observed 22 duplicate groups (with additional hydration and identity queries).

### Other factors ruled out
- **React StrictMode**: Disabled by default (`VITE_REACT_STRICT_MODE` not set)
- **Auth token refresh cascade**: AuthProvider already guards against TOKEN_REFRESHED re-renders
- **Identity double-commit**: Identity goes null->resolved, which correctly triggers one reset+fetch cycle
- **Multiple useFeed consumers**: CentralFeedScreen and PostFeedScreen use separate routes, never mounted simultaneously
- **PostCard re-triggering**: PostCards don't call fetchPosts

## Implementation

### Fix 1: Stabilize fetchPosts via refs (useFeed.js)

- Moved `upsertActors` and `getMissingOrStale` to refs (`upsertActorsRef`, `getMissingOrStaleRef`)
- Removed `upsertActors` from `fetchPosts` dependency array: `[viewerActorId, realmId]`
- Removed `realmId` from initial fetch effect deps: `[viewerActorId, fetchPosts]` (reset effect already handles realmId changes)
- Added `lastFetchAtRef` timestamp tracking in finally block

### Fix 2: Eliminate IntersectionObserver churn (CentralFeedScreen.jsx)

- Replaced the `observeMore` useCallback + effect pattern with a single effect
- Moved all rapidly-changing state (`posts.length`, `hasMore`, `loading`, `fetchPosts`) to refs
- Observer now only reconnects on `firstBatchReady` change (mount + first batch ready)
- Observer callback reads current values from refs instead of closed-over state

### Files modified
- `apps/VCSM/src/features/feed/hooks/useFeed.js`
- `apps/VCSM/src/features/feed/screens/CentralFeedScreen.jsx`

### Expected impact
- Initial route load: 1 pipeline execution instead of 2-3
- DB queries reduced from ~520 to ~260 per feed load
- IntersectionObserver reconnections reduced from N (every state change) to 1 (on first batch ready)
