## Execution Summary

### What Was Completed

All 3 slices executed. Follow notification events now publish through the notification engine.

**Slice 1 — Follow:**
Added `publishVcsmNotification()` call to `ctrlSubscribe` in `follow.controller.js`. After `dalInsertFollow` succeeds, publishes `follow` event with:
- recipient: followed actor
- source: follower actor
- objectType: `actor`, objectId: follower's actor ID
- linkPath: `/profile/{followerActorId}`

**Slice 2 — Follow request:**
Added publish to `ctrlSendFollowRequest` in `followRequests.controller.js`. After `dalUpsertPendingRequest` succeeds, publishes `follow_request` event with:
- recipient: target actor (the one being requested)
- source: requester actor
- linkPath: `/profile/{requesterActorId}`

**Slice 3 — Follow accept:**
Added publish to `ctrlAcceptFollowRequest`. After status updated to `accepted`, publishes `follow_request_accepted` event with:
- recipient: original requester (who asked to follow)
- source: target actor (who accepted)
- linkPath: `/profile/{targetActorId}`

### Dual-Write Safety

DB triggers on `vc.actor_follows` and `vc.social_follow_requests` continue to fire, writing to `vc.notifications`. The new engine writes go to `notification.*`. Since the read path now reads from `notification.*` only, users will see engine-published notifications. Legacy `vc.notifications` writes are now invisible to the UI but harmless.
