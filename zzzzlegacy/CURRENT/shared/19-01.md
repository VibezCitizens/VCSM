## Execution Summary — Canonical Menu QR URL

**Task:** All public menu QR entry points now use canonical slug-based URLs. The QR screen URL in the browser no longer exposes a raw UUID.

**Root issue:** The April 18 canonical URL work covered the menu page (`/actor/:actorId/menu` → `/profile/:slug/menu`) and the QR *payload* (QR code, Open Menu, Copy Link all generated slug URLs), but the QR *screen itself* remained at `/actor/:actorId/menu/qr` — browser URL still showed a raw UUID. No canonical QR route existed, and the dashboard + manage header both navigated to actor-ID paths for the QR screen.

**Fix Applied:**

- `VportPublicMenuQrScreen.jsx` — added `useActorCanonicalSlug` redirect: once slug resolves, issues `<Navigate to={/profile/${slug}/menu/qr} replace />`; renders QR view while loading (no flash)
- NEW `VportPublicMenuQrBySlugScreen.jsx` — canonical QR screen at `/profile/:slug/menu/qr`; resolves slug via `resolveMenuSlugDAL`, renders `VportPublicMenuQrView`
- `vportMenu.routes.jsx` — added `/profile/:slug/menu/qr` route; updated function signature to accept `VportMenuQrBySlugScreen`
- `routes/index.jsx` — added lazy import for `VportPublicMenuQrBySlugScreen`; passed `VportMenuQrBySlugScreen` into route builder
- `VportDashboardScreen.jsx` — `openQr` uses `dashboardDetails.slug` → `/profile/${slug}/menu/qr` (fallback to actor path); `openOnlineMenuPreview` uses slug directly → `/profile/${slug}/menu`
- `VportActorMenuManageHeader.jsx` — added `useActorCanonicalSlug(actorId)` hook; QR button navigates to `/profile/${canonicalSlug}/menu/qr` instead of `/vport/${actorId}/menu/qr`

**Files Changed:**
