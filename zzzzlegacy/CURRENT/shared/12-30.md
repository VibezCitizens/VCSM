### 2. Critical Breaks — Phase 2 (new VPORT render)

**Break 1 — Identity hydration (CRITICAL)**

File: `apps/VCSM/src/state/identity/identity.read.dal.js:105`

```
readVportIdentityDAL(vportId):
  vc.from('vports').select(VPORT_COLUMNS).eq('id', vportId).single()
```

For a new VPORT, `actor.vport_id` points to `vport.profiles.id`. This query hits `vc.vports` — returns null. Downstream in `vcsmActorHydrator.js`, the actor hydrates with `null` for name, avatar, slug. The profile renders blank.

**Break 2 — VPORT public details page (CRITICAL)**

File: `apps/VCSM/src/features/profiles/dal/vportPublicDetails.read.dal.js:3`

```
fetchVportPublicDetailsByActorId(actorId):
  supabase.schema('vc').from('actors').select(`id, kind, vport:vports(id, name, ..., public_details:vport_public_details(...))`)
```

For a new VPORT, the PostgREST join `vport:vports` resolves via FK `actors.vport_id → vc.vports.id`. That FK now points to a `vport.profiles.id` that does not exist in `vc.vports` — so `data.vport` is null. The profile page renders empty.

---

### 4. Safest Hybrid Approach

**Strategy: Try `vport.profiles` first, fall back to `vc.vports`.**

- For reads: query `vport.profiles` first. If null → query `vc.vports`. Return unified shape.
- For updates: detect which table owns the row at call time, update the correct one.
- For `listMyVports`: query actors via actor_owners for user, then batch-try both tables, merge results.

This avoids all schema-detection hacks. Two DB calls in worst case for legacy VPORTs. One for new VPORTs (the common case going forward).
