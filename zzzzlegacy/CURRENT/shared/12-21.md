## Profile Performance Audit

### Executive Summary

The Profile route triggers **3 separate reads of `vc.actor_privacy_settings`** for the same actor, **zero caching on the privacy DAL**, and the identity engine's `resolveAuthenticatedContext` has **no result cache** — meaning every call from `identityContext.jsx` fires the full 8-step platform query chain (session, app, access, account, state, preferences, actor links, active actor selection). On the vport path, an additional intermediate `VportProfileKindScreen` causes a waterfall: kind resolution -> vport type resolution -> profile view, each step blocked on the prior, with the profile tree unmounting/remounting when kind resolves.

### Profile Route Entry Chain

```
ProtectedRoute                        (apps/VCSM/src/app/guards/ProtectedRoute.jsx)
  -> useAuth()                         reads session (already cached by AuthProvider)
  -> useLegalConsent()                 (potential extra read, but not in profile-specific chain)
  -> <RootLayout>                      (apps/VCSM/src/app/layout/RootLayout.jsx)
       -> <TopNav>                     (always mounted)
       -> <BottomNavBar>               calls useIdentity() — CONSUMER #1
       -> <Outlet>
            -> <ActorProfileScreen>    (apps/VCSM/src/features/profiles/screens/ActorProfileScreen.jsx)
                 calls useIdentity()   — CONSUMER #2
                 calls useActorKind()  — queries vc.actors (cached via TTL 10min)
                 resolves PROFILE_KIND_REGISTRY[kind]
                   -> user: <ActorProfileViewScreen>
                   -> vport: <VportProfileKindScreen>
                              calls useVportType()  — EXTRA waterfall step
                              -> <VportProfileViewScreen>
                                   calls useIdentity() — CONSUMER #3 (line 86)
```

### Identity/Bootstrap Duplicate Sources

The identity engine `resolveAuthenticatedContext` at `engines/identity/src/controller/resolveAuthenticatedContext.controller.js` has **no caching layer**. Every call executes:

| Step | Table/View | DAL File |
|------|-----------|----------|
| 1 | `supabase.auth.getUser()` | `engines/identity/src/dal/session.read.dal.js` |
| 2 | `platform.apps` | `engines/identity/src/dal/app.read.dal.js` |
| 3 | `platform.user_app_access` | `engines/identity/src/dal/access.read.dal.js` |
| 4 | `platform.v_user_app_context` | `engines/identity/src/dal/account.read.dal.js` (line 110) |
| 5 | `platform.user_app_state` + `platform.user_app_preferences` (parallel) | `engines/identity/src/dal/state.read.dal.js` + `preferences.read.dal.js` |
| 6 | `platform.user_app_actor_links` | `engines/identity/src/dal/actorLinks.read.dal.js` |

**Where it fires from VCSM app code:**

1. **`identityContext.jsx` line 264** — `loadDefaultIdentityForUser()` on mount (or user change). Calls `resolveAuthenticatedContext` once. The controller at `identity.controller.js` line 141 does have in-flight dedup (`_identityInflight` map, line 15-16), but only deduplicates concurrent calls with the same userId+resolveAttempt key. Sequential calls are NOT deduped.

2. **`identityContext.jsx` line 63** — `switchActor()` calls `resolveAuthenticatedContext` AGAIN. This happens every actor switch.

3. **`identity.controller.js` line 281** — `loadOwnedActorChoices()` calls `resolveAuthenticatedContext` a THIRD time if actor picker is invoked.

4. **`identity.controller.js` line 192-210** — DEV-only RLS diagnostic at line 192 queries `vc.actor_privacy_settings` directly during identity hydration. This fires on every identity load in dev mode.

**Key finding:** The IdentityProvider at `identityContext.jsx` fires the resolve effect on `[authLoading, user?.id]` (line 487). This means any auth state change (including token refresh) re-triggers the full engine pipeline. There is no short-circuit if identity is already resolved and the userId hasn't changed.

### Profile/Public Data Duplicate Sources

**`public.profiles` is NOT directly queried in the profile path.** The profile data comes from:

1. **`readActorProfileDAL`** at `apps/VCSM/src/features/profiles/dal/readActorProfile.dal.js` — calls `vc.read_actor_profile` RPC (which internally reads from `vc.actors`, `public.profiles`, `vc.vports`). Has a **30-second TTL cache** (line 4). This is called by `getProfileView.controller.js` line 24.

2. **`readActorKindDAL`** at `apps/VCSM/src/features/profiles/dal/readActorKind.dal.js` — queries `vc.actors.kind`. Has a **10-minute TTL cache** (line 5). This fires BEFORE the profile load (in `ActorProfileScreen` line 28).

3. **`readIdentityActorByIdDAL`** at `apps/VCSM/src/state/identity/identity.read.dal.js` — reads `vc.actors` for the identity hydration. No explicit check if it duplicates what `readActorKind` already fetched.

4. **Hydration engine** at `engines/hydration/src/hydrate.js` — `getProfileView.controller.js` line 51 manually upserts actor summary into hydration store AFTER already having fetched the profile. This is write-only (good), but the RPC that fetched the data may overlap with what hydration would fetch.

**The `public.profiles` duplicate in perf snapshots** likely comes from:
- The `vc.read_actor_profile` RPC internally hitting `public.profiles`
- The identity engine's hydration path (`hydrateIdentityActor` at `identity.controller.js` line 246) potentially also reading actor/profile data
- Both of these fire for the SAME user when navigating to self-profile

### Privacy Duplicate Sources

**`vc.actor_privacy_settings` is queried 3 times for the same actor on profile load:**

| # | Location | File | Cached? |
|---|----------|------|---------|
| 1 | `readActorProfileDAL` line 22 | `apps/VCSM/src/features/profiles/dal/readActorProfile.dal.js` | YES (30s TTL on the full result) |
| 2 | `useProfileGate` -> `useActorPrivacy` -> `ctrlGetActorPrivacy` -> `dalGetActorPrivacy` | `apps/VCSM/src/features/social/privacy/dal/actorPrivacy.dal.js` | **NO CACHE** |
| 3 | DEV-only: `identity.controller.js` line 196 | `apps/VCSM/src/state/identity/identity.controller.js` | **NO CACHE** |

**Source #2 is the worst offender.** The `dalGetActorPrivacy` at `apps/VCSM/src/features/social/privacy/dal/actorPrivacy.dal.js` has zero caching — every call hits the database. And `useProfileGate` (line 11 of `useProfileGate.js`) fires independently from the profile load, meaning the same privacy data is fetched twice in parallel: once by `readActorProfileDAL` and once by `dalGetActorPrivacy`.

For the **vport path**, `VportProfileViewScreen.jsx` also calls `useProfileGate` (line 64), adding yet another privacy read for the same actor.

### Cache Bypasses

| Cache | Exists? | Problem |
|-------|---------|---------|
| `readActorKindDAL` TTL (10min) | YES | Working correctly. Cache hit on revisit. |
| `readActorProfileDAL` TTL (30s) | YES | Working, but 30s is short. Navigating away and back within 30s hits cache; after 30s, full refetch. |
| `dalGetActorPrivacy` | **NONE** | Always hits DB. No TTL cache, no in-flight dedup. |
| `ctrlGetActorPrivacy` controller | **NONE** | Pass-through to DAL with no caching layer. |
| Identity engine `resolveAuthenticatedContext` | **NONE** | No result cache. Every call fires 6+ platform queries. Only has in-flight dedup for concurrent calls with identical key. |
| Hydration store (`engines/hydration/src/store.js`) | YES (5min) | Working for actor summaries. But identity hydration does NOT check this store before fetching. |
| `useBlockStatus` | **NONE** | No cache on block status check. Fires on every mount. |

### Remount/Re-render Suspects

**1. VportProfileKindScreen waterfall (MAJOR)**
File: `apps/VCSM/src/features/profiles/kinds/vport/screens/VportProfileKindScreen.jsx`

When navigating to a vport profile:
- `ActorProfileScreen` mounts, calls `useActorKind(actorId)` -> shows skeleton
- Kind resolves to `'vport'`, `PROFILE_KIND_REGISTRY['vport']` = `VportProfileKindScreen`
- `VportProfileKindScreen` mounts, calls `useVportType(actorId)` -> shows "Loading..."
- Vport type resolves, renders `VportProfileViewScreen`
- `VportProfileViewScreen` mounts, calls `useProfileGate`, `useProfileView`, `useBlockStatus`, `useVportPublicDetails`

This is a **3-level waterfall**: kind -> vportType -> profile data. Each level blocks the next.

**2. useIdentity consumer cascade**
`BottomNavBar` (line 13 of BottomNavBar.jsx) and `ActorProfileScreen` (line 16) both call `useIdentity()`. When identity changes (e.g., after initial resolve), both re-render. This is expected behavior but means the profile tree re-renders when identity commits.

**3. `VportProfileViewScreen` calls `useIdentity()` AGAIN (line 86)**
This is the third `useIdentity()` consumer in the profile tree. It only uses it for `identity` in the gas prices view (line 389), but the hook subscription means any identity change triggers a re-render of the entire VportProfileViewScreen.

**4. canViewContent gate triggers double-load**
In both `ActorProfileViewScreen` and `VportProfileViewScreen`:
- `useProfileGate` fires with `version: gateVersion`
- `canViewContent` starts as `undefined` while gate loads
- `useProfileView` has `canViewContent` as a dependency (line 52 of `useProfileView.js`)
- When gate resolves, `canViewContent` changes from `undefined` to `true/false`, causing `useProfileView` to re-fire

This means the profile data is fetched TWICE: once with `canViewContent=undefined` and once with the resolved value.

**5. Effect dependency on `canViewContent` in `useProfileView`**
File: `apps/VCSM/src/features/profiles/hooks/useProfileView.js` line 52
The effect depends on `[viewerActorId, profileActorId, canViewContent, version]`. When `canViewContent` transitions from `undefined` -> `true`, the entire profile fetch (including posts, reactions, roses) re-executes.

### Exact Files To Fix First

1. `apps/VCSM/src/features/social/privacy/dal/actorPrivacy.dal.js` — Add TTL cache (eliminates duplicate #2 of privacy reads)
2. `apps/VCSM/src/features/profiles/hooks/useProfileView.js` — Stabilize `canViewContent` dependency to avoid double-fetch
3. `apps/VCSM/src/features/profiles/hooks/useProfileGate.js` — Use the privacy data already fetched by `readActorProfileDAL` instead of calling `dalGetActorPrivacy` separately
4. `apps/VCSM/src/features/profiles/kinds/vport/screens/VportProfileKindScreen.jsx` — Eliminate intermediate waterfall by merging kind+vportType resolution
5. `engines/identity/src/controller/resolveAuthenticatedContext.controller.js` — Add short TTL result cache (e.g., 5s) to prevent duplicate platform reads on rapid navigation
6. `apps/VCSM/src/features/profiles/kinds/vport/screens/VportProfileViewScreen.jsx` line 86 — Remove unnecessary `useIdentity()` call; pass identity data from props or narrow the subscription
7. `apps/VCSM/src/state/identity/identity.controller.js` lines 192-210 — Gate DEV privacy diagnostic behind a flag or remove it

### Recommended Fix Sequence

**Slice 1: Privacy cache + gate dedup (immediate win, low risk)**
- Add `createTTLCache(60_000)` to `dalGetActorPrivacy` in `actorPrivacy.dal.js`
- Optionally: have `useProfileGate` consume privacy from profile data instead of separate DAL call
- Expected impact: eliminates 2 of 3 privacy reads per profile load

**Slice 2: canViewContent stabilization (medium risk, high impact)**
- In `useProfileView.js`, treat `canViewContent === undefined` as "don't fetch yet" instead of fetching with undefined
- Add early return in `getProfileView.controller.js` when `canViewContent` is undefined
- Expected impact: eliminates the double profile fetch (profile + posts + reactions + roses)

**Slice 3: Vport waterfall collapse (medium risk, medium impact)**
- Merge `useActorKind` and `useVportType` into a single hook or load them in parallel
- Or: have `VportProfileKindScreen` start loading profile data while waiting for vportType
- Expected impact: removes 1 waterfall step (~200-400ms on slow connections)

**Slice 4: Identity engine caching (higher risk, systemic impact)**
- Add a short-lived cache (5-10s) to `resolveAuthenticatedContext` keyed on userId+appKey
- Or: cache individual DAL results (app row, access row) since they rarely change
- Expected impact: eliminates 6+ platform queries on rapid navigation between routes

- **After:** Profile data fetched once, only after gate resolves

### Slice 3 — Identity context result cache
**Files:**
- `engines/identity/src/controller/resolveAuthenticatedContext.controller.js` — Added 60-second module-level TTL cache keyed by `userId:appKey`. Cache check sits after session resolution but before the heavy 6-step platform query chain. Exported `invalidateIdentityResultCache()`.
- `engines/identity/src/controller/switchActiveActor.controller.js` — Calls `invalidateIdentityResultCache()` after actor switch
- `engines/identity/src/controller/logoutCleanup.controller.js` — Calls `invalidateIdentityResultCache()` on logout

### Slice 4 — Vport profile waterfall removal
**Files:**
- `apps/VCSM/src/features/profiles/screens/ActorProfileScreen.jsx` — Added `useVportType()` call in parallel with `useActorKind()`, passing `prefetchedVportType` and `vportTypeLoading` as props to the kind screen
