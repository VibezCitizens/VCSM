# 12-24 — Database Optimization Pass

## Task
Audit remaining duplicate queries and fix the highest-value ones.

## Audit Findings

### 1. Identity Engine Result Cache (resolveAuthenticatedContext)
- **Status:** Working correctly with 60s TTL + in-flight dedup
- **Callers:** identityContext.jsx (initial + switchActor), identity.controller.js (loadDefaultIdentityForUser + loadOwnedActorChoices)
- **Cache key:** `userId:appKey` — correct
- **Issue found:** 60s TTL causes re-queries during normal browsing. Extended to 120s.
- **N+1 found:** `loadOwnedActorChoices` did N individual `readIdentityActorByIdDAL` calls per actor link. Replaced with single batched `.in()` query.

### 2. notification.recipients Duplicates (CRITICAL)
- **Root cause:** Three independent hooks all call engine `countUnread()` for the same actor on page load:
  - `useNotiCount` (BottomNavBar — notification bell badge)
  - `useUnreadBadge` (BottomNavBar — labeled "chatUnread" but actually queries notification.recipients)
  - `useNotificationsHeader` (notification page header)
- Each `countUnread` call fires 2 DB queries (recipients + inbox_items), so 3 consumers = 6 DB queries for the same data.
- **Additional finding:** `inboxUnread.controller.js` and `notificationsCount.controller.js` are identical functions.

### 3. public.profiles Duplicates
- **Status:** Well-handled. Feed `readActorsBundle` has 30s per-actor cache. `readActorProfileDAL` has 30s TTL cache. No major overlap found.

### 4. Top N+1 Patterns
- `loadOwnedActorChoices` — N individual actor reads per actor link (fixed)
- Feed pipeline — already batched via `readActorsBundle` (no action needed)
- Hydration engine — already uses batch `.in()` queries with store freshness check (no action needed)

## Fixes Implemented

### Fix 1: Engine-level countUnread cache + in-flight dedup
- `engines/notifications/src/controller/countUnread.controller.js` — Added 5s TTL cache + in-flight promise dedup
- All 3 UI consumers now share the same cached result within any 5s window
- Eliminates 4 of 6 redundant DB queries per page load (notification.recipients + inbox_items)

### Fix 2: Cache invalidation on state changes
- `engines/notifications/src/controller/inboxState.controller.js` — markSeen, markRead, dismiss, archive all bust countUnread cache
- `engines/notifications/src/controller/getInbox.controller.js` — autoMarkSeen busts countUnread cache
- Exported `invalidateCountUnreadCache` from engine public API

### Fix 3: Consolidated duplicate controllers
- `apps/VCSM/src/features/notifications/inbox/controller/inboxUnread.controller.js` — Now delegates to `notificationsCount.controller.js` instead of duplicating the same function

### Fix 4: Extended identity result cache TTL
- `engines/identity/src/controller/resolveAuthenticatedContext.controller.js` — 60s -> 120s TTL
- Safe because cache is busted on actor switch and logout

### Fix 5: Batched actor reads in loadOwnedActorChoices
- `apps/VCSM/src/state/identity/identity.controller.js` — Replaced N individual `readIdentityActorByIdDAL` calls with single `.in("id", actorIds)` query

## Expected Impact
- **notification.recipients queries:** ~67% reduction (6 queries -> 2 per page load)
- **platform schema queries:** ~50% reduction during 60-120s window (identity cache extension)
- **vc.actors N+1:** Eliminated in actor switcher (N queries -> 1)
- **Estimated total query reduction:** ~40-60 fewer queries per session

## Note: useUnreadBadge Bug
`useUnreadBadge` is used as "chatUnread" in BottomNavBar but actually counts notification unreads, not chat unreads. This is a functional bug, not a perf issue. Flagged for separate fix.
