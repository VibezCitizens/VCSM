## Execution Summary

### Issue A ‚Äî Root Cause (CONFIRMED)

`useIdentityResolutionEffect` calls `loadDefaultIdentityForUser` ‚Üí `hydrateVcsmActor` ‚Üí `mapProfileActor`/`mapVportActor` which returns a full identity object with `displayName`, `avatar`, `username`. But `commitIdentity(nextDetails)` only stored `{ actorId, kind, ownerActorId }` in the public context; the display data was never written to `useActorStore`.

`ActorPill` reads via `useActorSummary(actorId)` ‚Üí `useActorStore.actors[actorId]` ‚Üí `null` ‚Üí falls back to `'User'`/`'/avatar.jpg'` on first render. The pill only updated later when a separate batch hydration event (feed, profile view) populated the store for that actor.

On `/upload` and `/settings?tab=account` there is no such event, so the pill was stuck on the fallback indefinitely.

**Fix applied:** Added `useActorStore.getState().upsertActors([...])` inside `commitIdentity()` in `identityContext.jsx`. This covers both initial load and actor-switch because both code paths call `commitIdentity`. No new network requests ‚Äî data is already present in `nextDetails`.

### Issue B ‚Äî Root Cause (CONFIRMED)

In `useAccountController.deleteAccount()`, when the Edge Function returns `AUTH_DELETE_FAILED` (app data deleted, auth user cleanup failed server-side), `dalDeleteCitizenAccountFull` throws an error. The catch block showed the error message but never called `logoutFromAuth()`. The user remained in the app with a deleted account's data and a still-valid auth session.

**Fix applied (2 files):**
1. `account.write.dal.js` ‚Äî attached `err.code = 'AUTH_DELETE_FAILED'` to the thrown error so callers can identify partial success without string-matching.
2. `useAccountController.js` ‚Äî catch block now checks `error?.code === 'AUTH_DELETE_FAILED'` and force-logs the user out. Total failure path (network error, 500) unchanged.
