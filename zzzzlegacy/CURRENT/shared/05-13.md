## Execution Summary

### Root Cause
Two mechanisms can allow a stale identity to win:

1. **Async race**: Multiple `run()` calls overlap when `user?.id` changes rapidly. The `cancelled` flag from React's effect cleanup protects against stale commits, but there's a gap — if two effects start before the first completes, both have `cancelled=false` until cleanup runs.

2. **Missing ownership validation**: `setIdentity(nextIdentity)` had NO check that the resolved identity actually belongs to the current session user. If the engine returns a wrong identity (e.g., due to PGRST116 fallback returning a different account's data), it gets committed blindly.

### Fixes Applied

**Fix 1 — Monotonic version guard** (identityContext.jsx)
Added a module-level `_resolveVersion` counter. Each resolve effect increments it at start. Before commit, the effect checks `myVersion !== _resolveVersion` — if a newer resolve started, the old one is discarded. This prevents async overlap even in edge cases the React `cancelled` flag doesn't cover.

**Fix 2 — Ownership validation before commit** (identityContext.jsx)
Before `setIdentity(nextIdentity)`, the code now checks:
```js
if (identityUserId && identityUserId !== user.id) {
  // IDENTITY_OWNERSHIP_MISMATCH — reject and clear
  setIdentity(null);
  return;
}
```
This guarantees the final identity belongs to the current session user.

**Fix 3 — Engine metadata includes userId** (identity.controller.js)
Added `userId: ctx.userId` to `_engineMeta` so the ownership check has the engine-proven userId to compare against the session.
