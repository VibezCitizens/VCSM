## Problem Summary

The security audit found five classes of vulnerabilities across the Supabase database:

1. **SECURITY DEFINER functions with no `SET search_path`** — CVE-2018-1058 vector. Without a fixed search_path, an attacker who controls any schema earlier in the search path can intercept calls to functions like `auth.uid()` inside a function running with superuser privileges.

2. **Functions callable by anon/authenticated with no internal auth guard** — `public.admin_delete_user_everywhere` had the real deletion body but zero authorization check. Any authenticated user could call it with any target UUID. `mark_all_messages_seen(uuid)` and `get_unread_message_total(uuid)` accepted a user ID as a parameter with no `auth.uid()` verification.

3. **Broken or too-broad RLS policies** — `public.profiles` used `USING (true)` exposing all profiles including private ones to anon. `learning.courses` had two `USING (true)` permissive policies overriding all org-scoped logic. `vc.friend_ranks` exposed all actor rank scores globally.

4. **Tables with RLS enabled but zero policies** — 21 tables were fully locked out. `platform.legal_documents` was silently blocking users from reading consent documents they were required to accept. Tables in `vport.*`, `vc.*`, `platform.*`, `learning.*` were all inaccessible.

5. **Tables with RLS entirely disabled** — 15 tables (1 in `platform`, 14 in `traffic`) had no RLS at all — every row readable and writable by any authenticated or anon caller with no restriction.

## Step 2 — RLS Policy Repairs

**File:** `step2_rls_policy_repairs.sql`
**Risk:** MEDIUM

### public.profiles — Private profile leakage
Dropped `profiles_public_read USING (true)` — exposed all profiles (including private) to anon. Replaced with three scoped policies:
- `profiles_select_via_actor` — owner self-read via `vc.actor_owners` join
- `profiles_discoverable_read` — public read only if `publish = true AND discoverable = true`
- `profiles_self_read` — authenticated self-read via `id = auth.uid()`

Note: `private` is NOT a column on `public.profiles` — privacy is in `vc.actor_privacy_settings`. The real columns controlling public visibility are `publish` and `discoverable`.

### learning.courses — Multi-tenant isolation
Dropped `"authenticated can insert courses"` and `"authenticated can view courses"` — both used `USING (true)` / `WITH CHECK (true)`, allowing any authenticated user to read all orgs' courses and insert into any organization. Replaced with:
- `courses_org_member_select` — `learning.can_current_user_access_course(id)`
- `courses_org_admin_insert` — `learning.can_current_user_manage_organization(organization_id)`

### wanders.cards — anon INSERT
`cards_insert_sender_user` incorrectly included `anon` role. Replaced with `cards_insert_sender_authenticated` scoped to `authenticated` only.

### vc.friend_ranks — Global rank exposure
`"friend_ranks visible to authenticated"` used `USING (true)` — any user could read all friend rank scores for all actors. Replaced with `friend_ranks_self_select`:
```sql
USING (
  owner_actor_id = vc.current_actor_id()
  OR friend_actor_id = vc.current_actor_id()
)
```

**File:** `step5_harden_security_definer.sql`

### public.admin_delete_user_everywhere
Added `learning.platform_admins` admin guard at function entry:
```sql
IF NOT EXISTS (
  SELECT 1 FROM learning.platform_admins
```

## secdef_a — search_path Hardening (33 functions)

**File:** `secdef_a_search_path_hardening.sql`
**Risk:** LOW — `ALTER FUNCTION ... SET search_path` only. No body, signature, or behavior changes. Idempotent.

**Why this matters (CVE-2018-1058):** Without a fixed `search_path`, an attacker who creates a shadow schema with identical function names can intercept calls inside a `SECURITY DEFINER` function running with superuser-level privileges.

**Technical note:** `ALTER FUNCTION` requires `pg_get_function_identity_arguments(oid)` (no DEFAULT values) — not `pg_get_function_arguments(oid)` which includes defaults and causes a syntax error.

33 functions hardened across schemas:

| identity | `refresh_actor_directory_all`, `refresh_actor_directory_row` |
| learning | `create_tenant_bootstrap`, `generate_student_login_id`, `generate_unique_tenant_slug` |
| omd | `get_photos_count`, `get_user_trips_count` |
| public | `admin_delete_user_everywhere`, `unfollow_user` (×2), `unfriend` |

| vport | `add_profile_actor_access`, `add_weekly_availability_rule_greenfield`, `create_profile_greenfield`, `create_resource_greenfield` (×2), `create_service_for_vport_actor`, `create_service_greenfield` (×2), `create_vport_for_actor` (×2), `current_actor_can_manage_location`, `current_actor_can_manage_organization`, `current_actor_can_manage_resource`, `hard_delete_vport`, `link_resource_service_greenfield`, `restore_vport`, `soft_delete_vport` |

## Moderation Batch 1 — can_manage_domain Bug Fix

`moderation.can_manage_domain(domain text, actor_id uuid)` had a broken `'vc'` branch:

**Before:**
```sql
WHEN 'vc' THEN
  RETURN EXISTS (SELECT 1 FROM vc.actors WHERE id = actor_id);
```
This returned `true` for any user who had a `vc.actors` row — i.e., every platform user. Effectively granted moderator-level RLS access to everyone.

**After:**
```sql
WHEN 'vc' THEN
  RETURN EXISTS (
    SELECT 1 FROM learning.platform_admins
    WHERE user_id = auth.uid() AND is_active = true
  );
```
Now matches the `'learning'` branch — requires platform admin membership.

## Moderation Batch 3 — moderators Table + Circular RLS Fix

Created `moderation.moderators` table:
- Columns: `id`, `actor_domain`, `actor_id`, `role`, `granted_by_actor_id`, `granted_at`, `is_active`, `expires_at`
- Roles: `viewer`, `moderator`, `admin`

**Circular RLS dependency discovered and fixed:**

`can_manage_domain` (SECURITY INVOKER) queries `moderation.moderators` to check role → RLS on `moderators` only allowed `platform_admins` to read → non-admin moderators couldn't see their own row → function always returned false for them → no moderator access.

Fixed by adding `moderators_self_read` policy:
```sql
CREATE POLICY moderators_self_read ON moderation.moderators
  FOR SELECT TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM vc.actor_owners
      WHERE actor_id = moderation.moderators.actor_id
        AND user_id = auth.uid()
    )
  );
```
Users can now read their own moderator grant row, allowing `can_manage_domain` to resolve correctly for non-admin moderators.

## Moderation Batch 4 — block_actor RPC Bidirectional Fix

**Problem:** `moderation.block_actor` RPC only deactivated `vc.actor_follows` in one direction (blocker→blocked). If the blocked actor was following the blocker, that row remained active.

**Fix:** RPC now deactivates follows in BOTH directions:
```sql
-- blocker→blocked (existing)
UPDATE vc.actor_follows SET is_active = false
WHERE follower_actor_id = p_blocker_actor_id AND followed_actor_id = p_blocked_actor_id;

-- blocked→blocker (added)
UPDATE vc.actor_follows SET is_active = false
WHERE follower_actor_id = p_blocked_actor_id AND followed_actor_id = p_blocker_actor_id;
```

**Backfill applied:** Existing `vc.actor_follows` rows where a block relationship exists but the reverse-direction follow was still active were deactivated.

**friend_ranks moved into RPC:** `vc.friend_ranks` bidirectional deletion moved from client-side (`deleteFriendRankRowsBetweenActors()` with swallowed try/catch) into the `moderation.block_actor` RPC body atomically. Deletion now happens in the same transaction as the block write — no risk of stale ranks from a failed client call.

**Pending:** `deleteFriendRankRowsBetweenActors()` call in `blockActor.controller.js` is now redundant and should be removed from the app code.
