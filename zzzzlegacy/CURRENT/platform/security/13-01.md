## Execution Summary

**What was completed:**

`createVport()` in `apps/VCSM/src/features/vport/dal/vport.core.dal.js` fully rewritten to target the new `vport.create_vport` RPC.

| Change | Detail |
|---|---|
| RPC target | `vc.rpc("create_vport")` → `vportSchema.rpc("create_vport")` |
| Slug generation | Client-side: name → lowercase → space→hyphen → strip non-alnum → 40-char cap → append 4-char random suffix |
| Category param | `p_vport_type` removed; `p_primary_category_key: cleanType` replaces it |
| RETURNS TABLE | `data[0]` row access; guard on `row?.profile_id` |
| Error handling | Named exceptions mapped to user-facing messages |
| console.log | Removed (2 blocks) — dev instrumentation no longer needed |
| Return shape | `vport_id, profile_id, actor_id, slug, profileId, vportId, actorId` |
| Removed from return | `actor_link_id, user_app_account_id, handle, name, vport_type` — not returned by new RPC |

**Why it was done this way:**
- The new `vport.create_vport` function resolves the user's existing personal actor (kind='user') rather than creating a new vport actor. Actor creation now happens as a separate DB step. The DAL return shape reflects this: `actorId` is now the user's actor, not a newly-minted vport actor.
- `vport.profile_actor_access` presence was confirmed in the full schema DDL provided this session — the blocking issue from the prior context is resolved.
- Slug is mandatory in the new RPC. Client-side generation with a random suffix keeps creation frictionless without requiring a unique name from the user.

**Blockers / Follow-up:**
- `vc.get_actor_summaries` still joins only `vc.vports` — new VPORTs will show blank display names in feeds until that function is updated to COALESCE from `vport.profiles`.
- `vport.create_vport` inserts `actorId = user's personal actor` into `vport.profile_actor_access`. The VPORT profile is now owned by the user's own actor, not a dedicated vport actor. Any code that looks up the actor for a vport via `vc.actors WHERE kind='vport' AND vport_id = ...` will not find a match for newly created VPORTs.
