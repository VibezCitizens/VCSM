## Applied Fixes

### Fix 1 — MAX_EMPTY_PAGES_PER_FETCH 3 → 2

**File:** `apps/VCSM/src/features/feed/queries/fetchCentralFeedPage.js`

```js
// Before
const MAX_EMPTY_PAGES_PER_FETCH = 3;

// After
const MAX_EMPTY_PAGES_PER_FETCH = 2;
```

**Impact:** Worst-case initial feed load reduced from ~33 to ~22 serial DB operations on sparse feeds.

---

### Fix 2 — Privacy Query Scoped to Uncached Actors

**File:** `apps/VCSM/src/features/feed/dal/feed.read.actorsBundle.dal.js`

The `vc.actor_privacy_settings` query was running against all `uniqueActorIds` on every pipeline call, including actors whose bundle data was already in the 30s TTL cache. Changed to scope the query to `uncachedIds` only (actors not yet in cache).

**Before:**
```js
if (uniqueActorIds.length > 0) {
  // ... privacy query using uniqueActorIds
}
```

**After:**
```js
if (uncachedIds.length > 0) {
  // ... privacy query using uncachedIds
}
```

**Impact:** On subsequent feed pages where most actors are cached, the privacy query does not fire at all.

### Fix 3 — deleted_by_actor_id Removed from Feed DAL + Normalize

**Files:**
- `apps/VCSM/src/features/feed/dal/feed.read.posts.dal.js`
- `apps/VCSM/src/features/feed/model/normalizeFeedRows.model.js`

Removed `deleted_by_actor_id` from the `readFeedPostsPage` SELECT list. Removed the corresponding `deleted_by_actor_id: r.deleted_by_actor_id ?? null` from the normalized output object. This column was fetched but never consumed by any UI component.

**Impact:** Reduced per-post wire payload. Eliminates unused column read from `vc.posts`.

### Fix 4 — LIMIT 50 Added to listPostComments

**File:** `apps/VCSM/src/features/post/commentcard/dal/postComments.read.dal.js`

```js
// Before: no limit
.order("created_at", { ascending: true })

// After
.order("created_at", { ascending: true })
.limit(50)
```

**Impact:** Posts with fewer than 50 comments are unaffected. High-volume threads no longer return unbounded row sets. 50 is a safety cap — cursor pagination is the long-term solution.

---

### Fix 5 — sendRose Post-Insert Reads Parallelized

**File:** `apps/VCSM/src/features/post/postcard/controller/sendRose.controller.js`

Before: `fetchPostByIdDAL` (notification routing) ran first, then `fetchReactionSummaryDAL` ran after the notification publish — 2 serial reads post-insert.

After: Both reads run in `Promise.all` after the insert. Notification publish happens after `post` data resolves, but both reads start concurrently.

```js
const [{ data: post }, { data: rows, error: readErr }] = await Promise.all([
  fetchPostByIdDAL(postId),
  fetchReactionSummaryDAL(postId),
]);
if (post?.actor_id) { publishVcsmNotification({ ... }); }
if (readErr) throw readErr;
```

**Impact:** Saves one serial round-trip per rose send.

---

### Fix 6 — toggleCommentLike Like-Path Reads Parallelized

**File:** `apps/VCSM/src/features/post/commentcard/controller/commentReactions.controller.js`

On the like path (not the unlike path): `readCommentActorAndPostIdDAL` (notification routing) + `getCommentLikeCount` now run in `Promise.all`.

```js
const [comment, likeCount] = await Promise.all([
  readCommentActorAndPostIdDAL(commentId),
  getCommentLikeCount(commentId),
]);
```

**Impact:** Saves one serial round-trip per new comment like.
