## Execution Summary

### What was completed
All `vc.vports` table references removed from the VCSM app. Every DAL that previously queried `vc.vports` now queries `vport.profiles` (via `vportSchema` client). The architecture shift key: new VPORTs use `vport.profiles.actor_id` as the join key (not `vc.actors.vport_id`).

### Files Changed

- `apps/VCSM/src/features/vport/dal/vport.core.dal.js` — removed `vc` import, removed `normalizeNewProfile()`, updated `getVportById`, `getVportBySlug`, `getVportsByIds`, `updateVport` to query `vport.profiles` only. `updateVport` now gets `actor_id` from the updated row directly for directory refresh.
- `apps/VCSM/src/features/vport/dal/vport.read.vportRecords.dal.js` — already updated (prior session)
- `apps/VCSM/src/features/vport/dal/readVportServiceCatalogByType.dal.js` — fixed bug: column is `category_key` not `vport_type`

**Settings:**

### Why
The VPORT migration moved all new VPORTs to `vport.profiles` schema. Legacy `vc.vports` was no longer the source of truth. Removing legacy code prevents split-brain queries and ensures the new `actor_id`-based join pattern is used consistently.

### Blockers / Follow-up
- DB-side: both `vport.create_vport` function variants still have `pg_temp` in `set search_path` — security risk for SECURITY DEFINER functions. Requires DB change (out of scope here).
- The `feed.read.actorsBundle.dal.js` `vportMap` is now keyed by `actor_id`. Any callers that previously read `vportMap[actor.vport_id]` need to be updated to use `vportMap[actor.id]`. The internal bundle consumers should be verified.
