## 2. Wentrex Chat Architecture

**Entry Points:**
- `InboxScreen.jsx` → uses `useInbox()` engine hook → renders conversation list
- `ConversationView.jsx` → uses `useConversation()`, `useConversationMessages()`, `useConversationMembers()` engine hooks
- `chatEngine.adapter.js` → wraps `createAllowedConversation()` and `createAnnouncementConversation()`

**Backend Flow:**
- Wentrex `setup.js` injects identity resolver, policy resolver, realm resolver, block relation stub, and actor search into engine
- Engine controllers call DAL functions which execute Supabase queries against `chat.*` schema
- Atomic message send uses `send_message_atomic` RPC (conversation update + message insert + inbox fan-out + outbox event in single transaction)

**Frontend Flow:**
- UI screens import only engine hooks (useInbox, useConversation, useConversationMessages, useConversationMembers)
- No direct Supabase queries to `chat.*` tables from UI
- Only Wentrex-specific queries are to `learning.actor_profiles` (actor search) and `learning.organizations` (realm resolution)

**Data Flow:**
```
Wentrex UI → engine hooks → engine controllers → engine DAL → Supabase → chat.* tables
                                                    ↓
                                               outbox_events → workers
```

## 6. Final Verdict

**Is Wentrex chat correctly using the `chat.*` tables?**

**MOSTLY YES — with 3 critical gaps.**

The architecture is sound. The chat engine is well-layered with proper DAL/controller/service separation. Wentrex's adapter correctly injects identity, policy, and realm resolution without bypassing the engine. UI screens exclusively use engine hooks. No legacy schema contamination detected. Column names and table names are correct across 14 of 16 tables.

**What must be fixed first (priority order):**

1. **Implement `chat.audit_log` writes** — Zero audit trail for any operation. This is a compliance and security gap. Start with: hard delete, moderation, conversation creation, member leave.

2. **Implement `chat.participant_snapshots` writes** — No membership state capture. Critical for audit, compliance, and "who was in the room" queries.

3. **Add membership validation to `inboxActions.controller.js` and `getInboxEntryForConversation.controller.js`** — Authorization bypass allows any actor to modify/read inbox entries.

4. **Fix stale inbox pointers after unsend/delete** — When last_message_id is soft-deleted or hard-deleted, inbox shows ghost messages.

5. **Fix `moderationActions` insert to include `conversation_id`** — Denormalized FK is NULL, making moderation queries by conversation impossible.

6. **Resolve `visibility = 'members'` enum mismatch** — Code uses a value not defined in the schema constraint.
