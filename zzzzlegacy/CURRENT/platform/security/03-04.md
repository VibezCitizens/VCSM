## Execution Summary

### Root Cause (expanded during execution)

Multiple layered failures, each revealed after the previous was fixed:

1. **`revalidate` export** — 5 pages had `export const revalidate = 900`. ISR is banned
   under `output: 'export'`. Next.js reports this as "missing generateStaticParams".
   Fixed: removed from all 5 pages.

2. **Async `generateStaticParams` in route handler** — `sitemaps/[chunk]/route.js` had
   `async function generateStaticParams()` awaiting a Supabase call. If the Supabase call
   fails at build time, Next.js silently treats the export as missing.
   Fixed: made `generateStaticParams` synchronous using `listPageCandidates()` (taxonomy-only).

3. **Top-level `await` async module** — `[detail]/page.jsx` and `[specialty]/page.jsx`
   imported `LIVE_DATA_STATUS` directly from `unifiedDataset.js`, which uses top-level
   `await loadVportRows()`. This makes the page module async, breaking generateStaticParams
   detection. Fixed: replaced import with `const LIVE_DATA_STATUS = "unavailable"`.

4. **Empty `generateStaticParams` from Supabase-only data** — All remaining dynamic routes
   (`[city]/[segment]`, `[city]/[segment]/[service]`, `[detail]`, `[specialty]`,
   `pro/[providerSlug]`) call provider-based functions that depend on `unifiedDataset`
   (MOCK_PROVIDERS). At build time without Supabase, all return `[]`. Next.js treats
   empty `generateStaticParams` as missing under `output: 'export'`.
   Fixed: created `taxonomyParams.repo.js` with sync taxonomy-only fallback functions.
   Routes now fall back to taxonomy-enumerated params when Supabase data is unavailable.
