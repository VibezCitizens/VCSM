## Execution Plan

### Design constraints discovered

`vc.actors` cannot be hard-deleted because:
- `vc.post_comments.actor_id` is NOT NULL + RESTRICT → cannot null or delete without first deleting all comments
- `vc.messages.sender_actor_id` is RESTRICT → same
- Cascade-deleting the actor would nuke posts, comments, messages, reactions, follows, wanders, etc.

**Decision: void the actor, delete the vport data.**

The actor row stays — `is_void = true` is set. This preserves all social content integrity (posts, comments, messages still reference a valid actor row). The vport.profiles row and all vport-specific child data are hard-deleted.

### Deletion order (respects FK constraints)

```
vport.bookings           WHERE profile_id = p_vport_id
vport.content_pages      WHERE profile_id = p_vport_id
vport.fuel_price_submission_reviews (via submission join)
vport.fuel_price_submissions WHERE profile_id = p_vport_id
vport.fuel_price_history WHERE profile_id = p_vport_id
vport.fuel_prices        WHERE profile_id = p_vport_id
--- then actor-scoped vport data ---
vc.vport_station_contact_routes WHERE contact_actor_id = v_actor_id
vc.vport_reviews         WHERE target_actor_id = v_actor_id
vc.vport_services / vport_service_addons / vport_rates WHERE actor_id = v_actor_id
vc.vport_actor_menu_items WHERE actor_id = v_actor_id
vc.vport_actor_menu_categories WHERE actor_id = v_actor_id
--- identity / platform ---
identity.actor_directory WHERE actor_id = v_actor_id
platform.user_app_actor_links WHERE actor_id = v_actor_id
vc.actor_onboarding_steps WHERE actor_id = v_actor_id
vc.actor_owners          WHERE actor_id = v_actor_id
--- void the actor (do NOT delete) ---
UPDATE vc.actors SET is_void = true WHERE id = v_actor_id
--- finally ---
DELETE vport.profiles WHERE id = p_vport_id
```

### Slice 1 — Migration (`20260420020000_vport_hard_delete_rpc.sql`)

  - Executes deletion chain above
  - Returns jsonb { ok, vport_id, actor_id }
- GRANT to authenticated, REVOKE from public

## Execution Summary

### What was completed

**Migration** — `apps/VCSM/supabase/migrations/20260420020000_vport_hard_delete_rpc.sql`

Created `vport.hard_delete_vport(p_vport_id uuid)`:
- Precondition: `auth.uid()` must own the vport AND `is_deleted = true` (soft delete first)
- Deletes data in 11-phase FK-safe order across vport.* and vc.vport_* tables
- Sets `vc.actors.is_void = true` — does NOT delete the actor row (preserves referential integrity for posts/comments/messages which have NOT NULL RESTRICT FKs)
- Returns `jsonb { ok, vport_id, actor_id }`
- SECURITY DEFINER, REVOKE from public, GRANT to authenticated

**Supabase workaround:** `SELECT INTO variable` is rejected by Supabase SQL editor with `42P01: relation "v_actor_id" does not exist`. Used `UPDATE ... SET updated_at = now() ... RETURNING actor_id INTO v_actor_id` for the precondition check instead. Same fix applied to soft delete in 20-01.

**DAL — `account.write.dal.js`**
- `dalHardDeleteVport(vportId)`: new function wrapping `vport.hard_delete_vport` RPC

**DAL — `vport.core.dal.js`**

- Exported on default object
