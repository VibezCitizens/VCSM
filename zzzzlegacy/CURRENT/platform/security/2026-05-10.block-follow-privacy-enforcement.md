## Problem

The platform had no end-to-end enforcement of block, privacy, and follow state across the post detail route, profile posts prefetch, feed cache invalidation, and the database RLS layer.

**Root causes (VENOM CRITICAL / HIGH):**
- `/post/:postId` fetched any post regardless of viewer — no block check, no privacy check, no follow check. Any actorId could be plugged into the URL to read any post.
- `useActorPosts` had `enabled: !!actorId` — it prefetched profile posts even when a privacy or block gate should have suppressed them. React Query then cached those posts client-side.

**Root causes (cache write paths):**
- Four cache invalidation functions (`invalidateFeedBlockCache`, `invalidateFeedFollowCache`, `invalidateActorPrivacyCache`, `invalidateActorBundleEntry`) were exported with zero call sites. Blocks, unfollows, privacy toggles, and follow accepts/declines never busted their associated caches. Stale state persisted until TTL expiry.

**Root causes (session binding):**
- Block and follow-request accept/decline controllers accepted any `actorId` from the caller. No enforcement that the actor performing the action matched the authenticated session.

**Root causes (performance / correctness):**
- Follow cache was scoped per page — every scroll page triggered a separate DB round-trip for the same viewer's follow graph.
- `readFollowState.dal.js` was a permanent stub returning `{ is_active: false }` — the `isFollowing` field on profile cards was always wrong.
- `dalGetFollowStatus` in `actorFollows.dal.js` had no cache — concurrent calls for the same pair each hit the DB.
- `actorPrivacy.dal.js` had no in-flight deduplication — rapid concurrent reads for the same actor issued redundant parallel DB queries.

**Root causes (DB / RLS — Carnage):**
- `vc.posts` RLS SELECT policy had no block exclusion. A viewer could query posts directly via Supabase and bypass app-layer block filtering.
- `moderation.blocks` SELECT policy only allowed the blocker to read their block rows. The blocked actor could not read blocks where they were the target — required for bidirectional enforcement in `vc.posts` RLS.

## Changes Applied

### Phase 1 — PostDetail and Profile Posts Gates (CRITICAL + HIGH)

#### Item 1: PostDetail bypass gate

**Problem:** `/post/:postId` fetched any post with no viewer context. Block, privacy, and follow state were never checked.

**New file:** `apps/VCSM/src/features/post/postcard/dal/postVisibility.dal.js`

`checkPostVisibilityDAL({ postActorId, viewerActorId })` — parallel block + privacy query, serial follow query only when the post actor is private. Block query explicitly filters `blocker_domain='vc'` and `blocked_domain='vc'` (required — `moderation.blocks` has no `id` column; the table uses a 4-column composite PK).

Returns `{ canView: true|false, reason }` where reason is one of: `'blocked'`, `'private_not_following'`, `'visible'`.

**Modified:** `getPostById.controller.js` — accepts `viewerActorId`, calls `checkPostVisibilityDAL`, returns `null` when `!canView`.

**Modified:** `usePostDetailPost.js` — accepts and forwards `viewerActorId` to the controller.

**Modified:** `PostDetail.view.jsx` — reads `actorId` from `useIdentity()` and passes it to the hook.

#### Item 2: Profile posts prefetch leak

**Problem:** `useActorPosts` had `enabled: !!actorId` — it fired even when the profile gate had already determined the viewer could not see content. React Query cached those posts client-side in violation of the block/privacy gate.

**Modified:** `useActorPosts.js` — added `canViewContent` param. `enabled: !!actorId && canViewContent !== false`.

**Modified:** `ActorProfilePostsView.jsx` — accepts and forwards `canViewContent` prop.

**Modified:** `ActorProfileViewScreen.jsx` — passes `canViewContent={gate.canView}`.

### Phase 2 — Cache Invalidation on All Write Paths (HIGH)

All four cache invalidation functions had zero call sites before this session.

**New file:** `apps/VCSM/src/features/feed/adapters/feedCache.adapter.js`

Cross-feature cache invalidation adapter. Re-exports `invalidateFeedBlockCache`, `invalidateFeedFollowCache`, and `invalidateActorBundleEntry` from their respective DAL files. All write-path controllers import from this adapter — never directly from the feed DAL internals.

#### Item 3: Block cache never invalidated

**Modified:** `blockActor.controller.js` — calls `invalidateFeedBlockCache(blockerActorId)` after block, unblock, and toggle operations.

#### Item 4: Follow cache never invalidated on follow / unfollow

**Modified:** `follow.controller.js` — calls `invalidateFeedFollowCache(followerActorId)` after a successful public follow and after unsubscribe.

**Modified:** `unsubscribe.controller.js` — two sequential writes (`dalDeactivateFollow` + `dalUpdateRequestStatus`) converted to `Promise.all`. Added `invalidateFeedFollowCache(followerActorId)`.

#### Item 5: Privacy toggle doesn't bust caches

**Modified:** `actorPrivacy.controller.js` (settings) — calls `invalidateActorPrivacyCache(actorId)` and `invalidateActorBundleEntry(actorId)` after `dalSetActorPrivacy`.

`invalidateActorBundleEntry(actorId)` is a new targeted per-actor bust added to `feed.read.actorsBundle.dal.js`. It calls `bundleCache.invalidate('actor:${actorId}')` rather than `invalidateActorsBundleCache()` which clears the entire bundle cache. The full-cache version remains for block invalidation; the targeted version is for privacy changes that only affect a single actor's entry.

### Phase 3 — Session Binding (MEDIUM)

#### Item 6: Block session binding

**Modified:** `blockActor.controller.js`, `unblockActorController`, `toggleBlockActorController` — added `assertingActorId` as a third parameter. Each throws if `assertingActorId !== blockerActorId`.

**Modified:** `useBlockActions.js` — calls `useIdentity()` internally and passes `sessionActorId` as `assertingActorId` on every block/unblock/toggle call.

**Modified:** `useBlockActorAction.js` — same pattern.

#### Item 7: Unblock ownership check (prior session — documented here for completeness)

`unblockActorController` calls `checkBlockStatus()` first and returns `{ blocked: false }` idempotently if `blockedByMe` is false. Prevents forcibly clearing another actor's block record.

#### Item 8: Follow-request accept/decline session binding

**Modified:** `ctrlAcceptFollowRequest`, `ctrlDeclineFollowRequest` — added `assertingActorId` param. Throws if `assertingActorId !== targetActorId`. Prevents a follower from accepting or declining their own request as if they were the target.

**Modified:** `useFollowRequestActions.js` — calls `useIdentity()` internally and passes `sessionActorId` as `assertingActorId`.

### Phase 4 — Performance / Correctness (MEDIUM)

#### Item 9: Follow cache was page-scoped

**Problem:** `feed.read.followRows.dal.js` used `.in("followed_actor_id", uniqueActorIds)` — a per-page DB query scoped to only the actors on that page. Every scroll page caused a cache miss and a new DB round-trip.

**Modified:** `feed.read.followRows.dal.js` — removed the `.in()` filter. Now fetches the viewer's full active follow graph once, caches the entire result set, and filters in-memory for each page. First page load triggers one full fetch; all subsequent pages are in-memory lookups against the cached graph.

#### Item 10: readFollowState.dal.js was a permanent stub

**Problem:** `readFollowState.dal.js` returned `{ is_active: false }` unconditionally — a permanent stub left in place. The `isFollowing` field on profile cards was always `false` regardless of actual follow state.

**Modified:** `readFollowState.dal.js` — replaced stub with a real `vc.actor_follows` query using `viewerActorId` and `targetActorId`. Used by `getProfileView.controller.js` to populate `isFollowing` on the profile card.

#### Item 11: TTL cache for dalGetFollowStatus

**Modified:** `actorFollows.dal.js` — added an 8-second TTL cache (`createTTLCache(8_000)`) keyed by `follower:followed`. A `has()` check gates the DB query. Cache entries are invalidated in the `dalInsertFollow` and `dalDeactivateFollow` write paths to prevent stale hits after state changes.

#### Item 12: In-flight dedup for actorPrivacy

**Modified:** `actorPrivacy.dal.js` — added a pending Promise Map. Concurrent callers for the same `actorId` share one in-flight DB round-trip rather than each issuing a separate query. `invalidateActorPrivacyCache()` also clears the pending map entry to prevent stale in-flight references from blocking future reads.

### Phase 5 — DB Migrations (Carnage)

Pre-migration snapshots taken from live DB via Supabase Management API before either migration was applied. Snapshots saved to:
- `zNOTFORPRODUCTION/db_snapshot/2026-05-10_pre-migration_moderation_blocks.snapshot.sql`
- `zNOTFORPRODUCTION/db_snapshot/2026-05-10_pre-migration_vc_posts.snapshot.sql`

**Key findings from snapshot:**
- `moderation.blocks` has no `id` column. Primary key is a 4-column composite: `(blocker_domain, blocker_actor_id, blocked_domain, blocked_actor_id)`. All block queries must filter on all four columns.
- `moderation.blocks` RLS was already comprehensive — existing policies used the `moderation.is_current_vc_actor()` helper function. Indexes already covered all lookup patterns. No index work needed.
- `vc.posts` RLS was enabled with `relforcerowsecurity=true` but the SELECT policy had no block exclusion. Direct Supabase queries bypassed the app-layer block filter entirely.

#### Migration 20260510010000 — moderation.blocks: blocks_select_blocked policy

```sql
CREATE POLICY "blocks_select_blocked" ON moderation.blocks
  FOR SELECT TO authenticated
  USING (blocked_domain = 'vc' AND moderation.is_current_vc_actor(blocked_actor_id));
```

Allows blocked actors to read blocks where they are the target. Required for bidirectional block enforcement in the `vc.posts` RLS policy — the posts policy joins against `moderation.blocks` and needs both sides to be readable by the viewer.

**File:** `apps/VCSM/supabase/migrations/20260510010000_moderation_blocks_rls_and_indexes.sql`

#### Migration 20260510020000 — vc.posts: Updated SELECT policy

Updated the `posts_select_actor_based` SELECT policy on `vc.posts`:
- Preserved existing logic: post owner always sees own posts; public actors visible to all; private actors visible only to followers.
- Added bidirectional block exclusion using `moderation.is_current_vc_actor()` and domain column filters — excludes posts from any actor who has blocked the viewer or whom the viewer has blocked.
- Added two performance indexes: `(actor_id, deleted_at)` and `(actor_id, created_at DESC)`.

**File:** `apps/VCSM/supabase/migrations/20260510020000_vc_posts_privacy_rls.sql`

## Enforcement Flow After Changes

```
User navigates to /post/:postId
  → usePostDetailPost(postId, viewerActorId)
    → getPostById.controller(postId, viewerActorId)
      → checkPostVisibilityDAL({ postActorId, viewerActorId })
        → parallel: block check (blocker_domain='vc', blocked_domain='vc') + privacy check
        → if private: serial follow check
        → returns { canView, reason }
      → controller returns null if !canView
    → hook returns null → PostDetail renders gate or 404
```

```
User writes a block or unblock
  → useBlockActions / useBlockActorAction
    → blockActor.controller(targetId, blockerActorId, sessionActorId)
      → assertingActorId === blockerActorId (throws if not)
      → DB write
      → invalidateFeedBlockCache(blockerActorId) — clears full block cache
      → invalidateActorBundleEntry(actorId) — targeted per-actor bust
```

```
User toggles privacy setting
  → actorPrivacy.controller → dalSetActorPrivacy
    → invalidateActorPrivacyCache(actorId)
    → invalidateActorBundleEntry(actorId)
  Feed next load: actor bundle re-fetched with correct is_private flag
```

```
Feed page load (any page)
  → feed.read.followRows.dal.js: fetches FULL viewer follow graph once, caches
  → subsequent pages: in-memory filter against cached graph
  → feedRowVisibility model: block + privacy checked per row
```

```
Direct Supabase query against vc.posts (bypassing app layer)
  → RLS: posts_select_actor_based enforced at DB level
    → block exclusion via moderation.blocks join (bidirectional)
    → private actor exclusion via actor_follows join
    → owner always passes
```

## Remaining Gaps

- **Cache lag (actor bundle):** Actor bundle TTL is 30s. A privacy change or block applied within that window may still show stale visibility state in the feed until `invalidateActorBundleEntry` propagates or TTL expires. The targeted per-actor bust reduces blast radius but does not eliminate the window.
- **Citizen profile route:** The block/privacy gate is enforced on PostDetail and ActorProfile posts. The Citizen profile view screen itself does not yet render a gate component for blocked viewers landing on a profile URL directly.
- **Follow-request pending state:** A private actor who blocks a pending follower does not auto-cancel the pending follow request at the DB level. The request remains in `pending` state until the actor manually declines it — the block gate prevents content visibility but the request row persists.
- **DB-level follow check on vc.posts:** The updated `vc.posts` SELECT policy enforces block exclusion at the DB layer. Privacy enforcement (following check) at the DB layer relies on the `vc.actor_follows` join, which covers the authenticated viewer. Service-role or admin access bypasses RLS entirely — this is expected and not a gap in the app threat model.

### SF-01 — ctrlSubscribe block guard (follow.controller.js)

Imported `ctrlGetBlockStatus` from `@/features/block`. Added bidirectional block check immediately after self-follow guard, before all writes and notifications. Throws `'Cannot follow a blocked actor'` if `isBlocked === true`.

Also removed dead `ctrlUnsubscribe` export (was incomplete — missing `dalUpdateRequestStatus` call, never imported by any caller). `dalDeactivateFollow` import also removed since it was only used by the dead function.

### SF-01 — ctrlSendFollowRequest block guard (followRequests.controller.js)

Same bidirectional block check pattern. Added after self-follow guard, before `dalGetRequestStatus` and all writes. Throws `'Cannot send follow request to a blocked actor'`. Guards independently from `ctrlSubscribe` since it can be called directly.

### SF-05 — Accept request feed cache invalidation (followRequests.controller.js)

`ctrlAcceptFollowRequest` now calls `invalidateFeedFollowCache(requesterActorId)` and `invalidateFeedFollowCache(targetActorId)` after `dalInsertFollow` + `dalUpdateRequestStatus` succeed. Both actors' feed follow caches are busted immediately on accept.

### SF-03 — ctrlCancelFollowRequest session binding (followRequests.controller.js)

Added `assertingActorId` parameter. Guard: `assertingActorId !== requesterActorId` throws `'ctrlCancelFollowRequest: session actor does not own this request'` before any DB access. Matches the existing accept/decline ownership pattern.

### SF-02 — useProfileGate block awareness (profiles/hooks/useProfileGate.js)

Imported `useBlockStatus` from `@/features/block`. Called positionally: `useBlockStatus(viewerActorId, targetActorId)`. `blockLoading` included in loading derivation to prevent content flash while block resolves.

`canView` rewritten: `isSelf || (!isBlocked && (!isPrivate || isFollowing))`
- Self-view always passes
- Any block in either direction → `canView = false` regardless of privacy/follow state
- `isBlocked` added to return object for UI to render `<BlockedState />` vs `<PrivateProfileNotice />`

### AF-01 — assertingActorId caller (useFollowActorToggle.js)

Passed `assertingActorId: followerActorId` to `ctrlCancelFollowRequest`. `followerActorId` is the session actor in this context.
