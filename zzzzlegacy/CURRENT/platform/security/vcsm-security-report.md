## CRITICAL FINDINGS

---

### VENOM FINDING #1 — CRITICAL
- **Location:** `apps/VCSM/src/features/upload/controllers/createPostController.js:29-32`
- **Application Scope:** VCSM
- **Current behavior:** Post creation validates auth via `getCurrentAuthUserDAL()` but does NOT validate that `identity.actorId` belongs to the authenticated user. The controller trusts the identity context without verifying ownership.
- **Risk:** A user could supply an `identity.actorId` from another user. If the React context can be manipulated client-side, posts could be attributed to the wrong actor.
- **Severity:** CRITICAL
- **Why it matters:** Posts created under another actor's identity is a privilege escalation and content attribution attack.
- **Recommended mitigation:** In controller, verify that `identity._engineMeta.userId` matches `user.id` from auth session before allowing post creation. Do not trust client-provided actorId without explicit ownership proof.
- **Follow-up command:** BUGSBUNNY

### VENOM FINDING #2 — CRITICAL
- **Location:** `apps/VCSM/src/features/dashboard/vport/controller/saveVportPublicDetailsByActorId.controller.js:47-57`
- **Application Scope:** VCSM
- **Current behavior:** Controller reads vport_id from actor via `readActorVportLinkDAL()` but does NOT verify that the current user OWNS the target actor. It trusts the `actorId` URL parameter.
- **Risk:** Any authenticated user can call this controller with any actorId and update that vport's public details (name, description, settings). Full vport takeover for settings is possible.
- **Severity:** CRITICAL
- **Why it matters:** Unauthorized modification of any vport's public profile data. Business identity corruption.
- **Recommended mitigation:** Add explicit ownership check before upsert: verify `identity.actorId === actorId` AND confirm actor ownership through `readActorOwnerUserDAL`. Reject if mismatch.
- **Follow-up command:** BUGSBUNNY

### VENOM FINDING #3 — CRITICAL
- **Location:** `apps/VCSM/src/features/moderation/controllers/moderationActions.controller.js`
- **Application Scope:** VCSM
- **Current behavior:** `hideReportedObjectController()` and `dismissReportController()` accept `moderatorActorId` as an input parameter with NO permission check. Any authenticated user can perform moderation actions by passing their own actorId.
- **Risk:** Any authenticated user can hide posts, dismiss reports, and perform moderation actions across the entire platform.
- **Severity:** CRITICAL
- **Why it matters:** Complete moderation bypass. Malicious actors can hide all reports against themselves and suppress content platform-wide.
- **Recommended mitigation:** Add explicit moderator role/permission check before any moderation action. Verify caller has `moderator` or `admin` role via platform engine. Do NOT trust the `moderatorActorId` parameter — derive it from the authenticated session.
- **Follow-up command:** BUGSBUNNY + DB (check if moderator role exists in schema)

### VENOM FINDING #7 — MEDIUM
