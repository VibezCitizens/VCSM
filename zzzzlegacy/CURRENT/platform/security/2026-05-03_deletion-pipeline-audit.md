## A. Actor / User Account Soft Delete

### Overview

The soft delete path is invoked NOT directly by the frontend in isolation. Since v3, the frontend **only** calls hard delete (the Edge Function), which internally calls the soft delete RPC as step 1. The standalone `dalSoftDeleteCitizenAccount()` still exists in the DAL but is no longer called by the frontend — it is called by the Edge Function via the anon client.

### Entry Point Chain

```
AccountTab.view.jsx (Citizen mode — "Delete account" button)
  → setShowConfirmAccount(true)
  → DeleteModal "Delete account" button
  → deleteAccount()               [useAccountController.js]
  → ctrlDeleteAccount()           [account.controller.js]
  → dalDeleteCitizenAccountFull() [account.write.dal.js]
  → supabase.functions.invoke('delete-citizen-account')  [POST]
  [Inside Edge Function:]
  → userClient.rpc('soft_delete_citizen_account')        [public schema]
```

### RPC: `public.soft_delete_citizen_account()`

- **Schema:** `public`
- **Security:** `SECURITY DEFINER`, called via anon client scoped to the user's JWT (inside the Edge Function)
- **Params:** None — identity resolved from `auth.uid()` inside the RPC
- **Tables updated:**

| Table | Column | Value |
|---|---|---|
| `public.profiles` | `is_deleted` | `true` |
| `public.profiles` | `deleted_at` | `now()` |
| `vc.actors` | `is_deleted` | `true` |
| `vc.actors` | `deleted_at` | `now()` |

- **Not touched:** `vc.actor_owners`, `is_void` (these are separate features)
- **Guard:** RPC checks `a.is_deleted = false` before proceeding — idempotent-safe (throws `CITIZEN_NOT_FOUND` if already deleted)

### DB Functions Updated by Soft Delete

- `vc.get_actor_summaries(uuid[])` — excludes actors where `coalesce(a.is_deleted, false) = false`
- `vc.read_actor_profile(uuid)` — excludes actors where `coalesce(a.is_deleted, false) = false`

Both functions were updated in migration `20260420040000_citizen_soft_delete_v2.sql`.

### Expected vs Current Session Behavior

**Expected:** After soft delete, app data is hidden. The session is then cleaned up by the hard delete path (step 3 of the Edge Function), which deletes the auth.users row. Soft delete alone does NOT clear the session.

**Current:** Soft delete is only invoked inside the Edge Function, never standalone. When soft delete succeeds but auth delete fails (`AUTH_DELETE_FAILED` code), `useAccountController.deleteAccount()` still force-logs the user out by clearing localStorage and calling `logoutFromAuth()`. This is the correct safety behavior — user has no valid app state to return to.

### Hydration / Cache Invalidation

After hard delete triggers the logout:
- `useAccountController.deleteAccount()` manually removes `actor_kind`, `actor_vport_id`, `actor_touch` from localStorage
- `logoutFromAuth()` in `AuthProvider` calls `clearAllIdentityStorage()` and dispatches `actor:changed` event
- `supabase.auth.signOut({ scope: 'local' })` clears local Supabase session tokens
- React Query cache is NOT explicitly invalidated — it is abandoned because the full unmount cycle happens as the user is redirected to `/login`

**Verdict:** Cache invalidation is adequate for this flow because the entire app re-initializes post-login. No stale reads are possible.

### Can User Log In Afterward?

- If soft delete only (partial Edge Function failure): No — the auth user row still exists in Supabase, but `vc.actors.is_deleted = true`. On next login, `identity.controller.js` would read the actor row, detect `actorRow.is_deleted === true`, and return `DELETED_ACCOUNT_SENTINEL`. The `useIdentityResolutionEffect` hook is expected to detect this sentinel and call `logout()`. The user cannot see any app content.
- If hard delete succeeds (auth.users deleted): The email can be re-used to sign up as a new Citizen. No app rows block re-registration.

## B. Actor / User Account Hard Delete

### Overview

This is the only path exposed to the user in the current UI. "Delete account" = hard delete. Soft delete is an internal implementation step, not a user-facing option for accounts.

### Entry Point Chain

```
AccountTab.view.jsx (Citizen mode — "Delete account" button)
  → setShowConfirmAccount(true)
  → DeleteModal "Delete account" confirmation
  → deleteAccount()                     [useAccountController.js]
  → ctrlDeleteAccount()                 [account.controller.js]
  → dalDeleteCitizenAccountFull()       [account.write.dal.js]
  → POST /functions/v1/delete-citizen-account  [Edge Function]
    → [1] rpc('soft_delete_citizen_account')   — app data first
    → [2] adminClient.auth.admin.deleteUser()  — auth user second
```

### Edge Function: `delete-citizen-account/index.ts`

- **Path:** `apps/VCSM/supabase/functions/delete-citizen-account/index.ts`
- **Auth:** Verifies Bearer JWT via `userClient.auth.getUser()` before any mutation
- **No client-supplied userId:** Identity comes from the JWT only — `auth.uid()` inside the RPC
- **Service role key:** Never exposed to the client — held server-side only

### Whether auth.users Is Deleted

Yes. The Edge Function calls `adminClient.auth.admin.deleteUser(user.id)` as step 2. If this step fails, `AUTH_DELETE_FAILED` is returned. The frontend detects this code and still forces logout. The auth.users row persists; admin recovery is possible via `userId` logged server-side.

### Whether App Rows Are Deleted / Anonymized

`soft_delete_citizen_account()` RPC marks:
- `public.profiles.is_deleted = true`
- `vc.actors.is_deleted = true`

Posts, comments, messages, follows, reactions, VPORTs, and all social content are **NOT deleted**. The citizen's actor row is marked deleted and hidden from all hydration reads. Social content orphans gracefully — actor cards render `is_void` / `is_deleted` placeholders.

### Storage / Media Cleanup

**None.** There is no storage bucket cleanup in the current deletion pipeline. Avatar files and banner files in Supabase Storage (or S3) are NOT deleted. This is a known gap.

### Email Re-Signup

After a successful hard delete (auth.users removed), the same email address can be used to create a new account. No constraint exists at the app level blocking re-registration by email.

## C. VPORT Soft Delete

### Entry Point Chain (Two surfaces)

**Surface 1 — Account tab (single active VPORT when in VPORT identity mode):**
```
AccountTab.view.jsx (VPORT mode — "Deactivate…" button on VportDangerRow)
  → setSoftTarget(activeVport)
  → SoftDeleteModal "Deactivate VPORT" confirmation
  → handleSoftConfirm()                   [AccountTab.view.jsx]
  → softDeleteVport(softTarget.id)        [useAccountController.js]
  → ctrlSoftDeleteVport({ vportId })      [account.controller.js]
  → dalDeleteMyVport(vportId)             [account.write.dal.js]
  → vportSchema.rpc('soft_delete_vport', { p_vport_id })
```

**Surface 2 — Account tab (citizen mode — all VPORTs listed):**
Same chain as Surface 1, sourced from `vports` list state.

**Note:** The VPORTs tab (`VportsTab.view.jsx`) does NOT have a soft delete button. The deactivate flow is account-tab-only.

### RPC: `vport.soft_delete_vport(p_vport_id uuid)`

- **Schema:** `vport`
- **Security:** `SECURITY DEFINER`, ownership enforced via `owner_user_id = auth.uid()` in WHERE clause
- **Params:** `p_vport_id uuid`
- **Tables updated:**

| Table | Column | Value |
|---|---|---|
| `vport.profiles` | `is_deleted` | `true` |
| `vport.profiles` | `is_active` | `false` |
| `vport.profiles` | `deleted_at` | `now()` |
| `vport.profiles` | `deleted_by_actor_id` | self (actor_id from the same row) |
| `vport.profiles` | `updated_at` | `now()` |

- **Not touched:** `vc.actors.is_void`, `vc.actors.is_deleted`, `vc.actor_owners`

### Whether vc.actors.vport_id Remains Linked

Yes. The `vc.actors` row is NOT modified by soft delete. `vc.actors.vport_id` still points to the `vport.profiles.id` row. The link is preserved for restore.

### Whether vport.profiles.is_deleted Changes

Yes — set to `true`. `is_active` is also set to `false`.

### Whether Public Views Hide It

Yes. The existing RLS policy on `vport.profiles` uses `USING (is_active = true AND is_deleted = false)`. This immediately makes the VPORT invisible to all public PostgREST reads. No view migration needed.

Additionally, `resolveVportSlugDAL` filters `is_deleted = false`, so deleted VPORTs cannot be reached via public slugs.

### Whether Owner Restore Screen Can Still Fetch It

Yes. The owner's `listMyVportsDAL` queries `vport.profiles` with `owner_user_id = auth.uid()` without an `is_deleted` filter — it returns all VPORTs including soft-deleted ones. The is_deleted flag is included in the SELECT and used to split `activeVports` / `deactivatedVports` in the view.

### Post-Delete Actor Switch Behavior

If the user is currently in VPORT identity mode when the soft delete succeeds:
1. `useAccountController.softDeleteVport()` checks `if (isVport)` and tries `switchActor()` to the citizen actor.
2. If `switchActor` succeeds, no redirect is needed.
3. If `switchActor` fails, `_switchToProfile()` is called (localStorage manual write) and `window.location.replace('/me')` is used as the fallback.

The `blockedVport` auto-switch in `identityContext.jsx` acts as a systemic safety net — if the identity is still showing a deleted VPORT after the switch, the effect fires and switches back to citizen mode.

## D. VPORT Restore / Reactivate

### Entry Point Chain (TWO independent surfaces — both broken in the same way)

**Surface 1 — Account tab `useAccountController`:**
```
AccountTab.view.jsx → handleRestoreConfirm()
  → restoreVport(restoreTarget.id)            [useAccountController.js line 115]
  → ctrlRestoreVport({ vportId: targetVportId })  [account.controller.js line 24]
  → restoreVportFn?.(vportId)                 [account.controller.js line 25]
  → undefined — SILENTLY DOES NOTHING
```

**Surface 2 — VPORTs tab `useVportsController`:**
```
VportsTab.view.jsx → handleRecoverConfirm()
  → restoreVport(restoreTarget.id)            [useVportsController.js line 73]
  → ctrlRestoreVport({ vportId: targetVportId })  [account.controller.js line 24]
  → restoreVportFn?.(vportId)                 [account.controller.js line 25]
  → undefined — SILENTLY DOES NOTHING
```

**Surface 3 — RestoreVportScreen (blocked VPORT route guard):**
```
RestoreVportScreen.jsx → handleRestore()
  → restore()                                 [useRestoreVport hook]
  → restoreVport({ vportId })                 [useVportAccountOps adapter]
  → ctrlRestoreVport({ ...params, restoreVportFn: restoreVport })  [useVportAccountOps.js]
  → restoreVportFn?.(vportId)                 [account.controller.js line 25]
  → restoreVport(vportId) from vport.core.dal.js
  → vportSchema.rpc('restore_vport', { p_vport_id })
  → WORKS CORRECTLY
```

### CRITICAL BUG: ctrlRestoreVport Signature Mismatch

`account.controller.js:24` defines:
```js
export async function ctrlRestoreVport({ vportId, restoreVportFn }) {
  await restoreVportFn?.(vportId)
}
```

The controller holds no direct import of `dalRestoreVport`. It is designed as an injectable delegate — callers must supply `restoreVportFn`.

**Both main callers (Account tab and VPORTs tab) call:**
```js
await ctrlRestoreVport({ vportId: targetVportId })
```
No `restoreVportFn` is passed. `restoreVportFn` is `undefined`. The optional chain `restoreVportFn?.(vportId)` evaluates to `undefined` with no exception thrown. The RPC is never called. The restore succeeds silently (returns `true` from the hook) but nothing happens in the database.

**Only Surface 3 (RestoreVportScreen via `useVportAccountOps`) passes `restoreVportFn` correctly** and actually executes the restore.

The DAL function `dalRestoreVport()` exists in `account.write.dal.js` but is imported nowhere in the current wiring — it is an orphan.

### RPC: `vport.restore_vport(p_vport_id uuid)` (never reached from Account/VPORTs tabs)

- **Schema:** `vport`
- **Params:** `p_vport_id uuid`
- **Tables affected if called:**

| Table | Column | Value |
|---|---|---|
| `vport.profiles` | `is_deleted` | `false` |
| `vport.profiles` | `is_active` | `true` |
| `vport.profiles` | `deleted_at` | `NULL` |
| `vport.profiles` | `deleted_by_actor_id` | `NULL` |
| `vport.profiles` | `updated_at` | `now()` |

### Hydration Refresh Requirements

After a successful restore, the app must:
1. Refresh `availableActors` — the identity engine's actor link list must include the restored VPORT for `switchToVport` to resolve it
2. Invalidate or refetch the `listMyVportsDAL` query so `items` reflects `is_deleted: false`
3. Allow the user to switch into the VPORT again

**Current behavior on VPORTs tab:** When `restoreVport` returns `true` (incorrectly), the view optimistically sets `is_deleted: false` on the list item and calls `refreshAvailableActors()`. The DB state is NOT updated — the optimistic update creates a UI/DB mismatch. Next load will re-show the VPORT as deleted.

**Current behavior on Account tab:** When `restoreVport` returns `true` (incorrectly), the view optimistically sets `is_deleted: false` and clears `restoreTarget`. Same DB desync.

### Current Broken Query Paths

- `AccountTab.view.jsx → handleRestoreConfirm → restoreVport → ctrlRestoreVport` → silently does nothing
- `VportsTab.view.jsx → handleRecoverConfirm → restoreVport → ctrlRestoreVport` → silently does nothing

## E. VPORT Hard Delete

### Entry Point Chain (TWO independent surfaces)

**Surface 1 — Account tab:**
```
AccountTab.view.jsx → handleHardConfirm()
  → hardDeleteVport(hardTarget.id)           [useAccountController.js line 100]
  → ctrlHardDeleteVport({ vportId })         [account.controller.js line 20]
  → dalHardDeleteVport(vportId)              [account.write.dal.js line 75]
  → vportSchema.rpc('hard_delete_vport', { p_vport_id })
```

**Surface 2 — VPORTs tab:**
```
VportsTab.view.jsx → handleHardDeleteConfirm()
  → hardDeleteVport(hardDeleteTarget.id)     [useVportsController.js line 88]
  → ctrlHardDeleteVport({ vportId })         [account.controller.js line 20]
  → dalHardDeleteVport(vportId)              [account.write.dal.js line 75]
  → vportSchema.rpc('hard_delete_vport', { p_vport_id })
```

Hard delete wiring is **correct** — both surfaces reach the RPC.

### RPC: `vport.hard_delete_vport(p_vport_id uuid)`

- **Precondition enforced at DB level:** `vport.profiles.is_deleted = true` — will throw `VPORT_NOT_FOUND_OR_NOT_DELETED` if called on an active VPORT
- **11-phase FK-safe deletion chain:**

| Phase | Tables Deleted |
|---|---|
| 1 | `vport.fuel_price_submission_reviews`, `vport.fuel_price_submissions`, `vport.fuel_price_history`, `vport.fuel_prices` |
| 2 | `vport.bookings`, `vport.content_pages` |
| 3 | `vport.locksmith_service_areas` |
| 4 | `vc.vport_station_contact_routes` (RESTRICT FK — must go first) |
| 5 | `vc.vport_actor_menu_item_media`, `vc.vport_actor_menu_items`, `vc.vport_actor_menu_categories` |
| 6 | `vc.vport_service_addons`, `vc.vport_services`, `vc.vport_rates` |
| 7 | `vc.vport_reviews`, `vc.vport_station_price_settings`, `vc.vport_station_amenities` |
| 8 | `vc.vport_fuel_price_submission_reviews`, `vc.vport_fuel_price_submissions`, `vc.vport_fuel_price_history`, `vc.vport_fuel_prices` |
| 9 | `identity.actor_directory`, `platform.user_app_actor_links`, `vc.actor_onboarding_steps`, `vc.actor_owners` |
| 10 | `UPDATE vc.actors SET is_void = true` (actor row kept, voided) |
| 11 | `DELETE vport.profiles WHERE id = p_vport_id` |

### Whether Actor Row Remains or Is Deleted

The `vc.actors` row is **NOT deleted**. It is voided: `is_void = true`. This is intentional — `vc.post_comments.actor_id` and `vc.messages.sender_actor_id` are RESTRICT FKs to `vc.actors`. Deleting the actor would cascade-block or destroy social content. Voiding preserves historical content integrity.

### Whether Media / Storage Cleanup Exists

**No.** Avatar and banner files in storage buckets are not removed by the hard delete RPC. This is a known gap documented in `vcsm.vport.delete-lifecycle.md` §9.

### Known FK Gaps in the Hard Delete Chain

Four tables are documented as NOT covered by the current deletion chain:
- `vport.resources`
- `vport.portfolio_items`
- `vport.availability_exceptions`
- `vport.availability_rules`

If these tables contain rows for the deleted VPORT, they will persist as orphaned rows after hard delete. The booking model also references `booking_resources` / `resources` — the `vport.resources` table is the canonical FK anchor here (via `vportClient` as `.from('resources')`).

### Whether Public Routes Break After Deletion

- VPORT profile page: `resolveVportSlugDAL` filters `is_deleted = false`. After soft delete, the slug resolves to nothing. The profile page shows 404.
- After hard delete: the `vport.profiles` row is removed entirely. The slug no longer exists. The public route 404s correctly.
- `vc.actors.is_void = true` causes `vc.get_actor_summaries` to exclude the actor from feed hydration (existing `is_void = false` filter).
- Social content (posts, comments) authored by the voided actor will render actor-placeholder UI.

### Post-Delete Behavior in the UI

**Account tab (`useAccountController.hardDeleteVport`):**
1. On success, calls `_switchToProfile()` — manually writes localStorage actor keys
2. Calls `window.location.replace('/me')` — full page reload
3. Does NOT call `refreshAvailableActors()` — relying on the page reload to re-initialize

**VPORTs tab (`useVportsController.hardDeleteVport`):**
1. On success, removes the VPORT from the list: `setItems(prev => prev.filter(...))`
2. Calls `refreshAvailableActors()`
3. Clears `hardDeleteTarget`
4. Does NOT trigger a page redirect — user stays on the VPORTs tab

## F. Relationship / Query Audit

### Valid Relationship Paths (Current)

| Relationship | Current Code | Status |
|---|---|---|
| `vc.actors.vport_id → vport.profiles.id` | Used in `vportReviewAuthors.read.dal.js` — actors joined then vport.profiles queried by actor_id | VALID (queries `vport.profiles` by `actor_id` column, not via embed) |

| `vport.bookings.resource_id → vport.resources.id` | Booking model uses `row.booking_resources` / `row.resources` field names | PARTIALLY VALID — field names are inconsistent |

### Noted Issues

**1. `vport.core.dal.js` — `listMyVports()` does not include `is_deleted`**
```js
const SELECT = "id,owner_user_id,name,slug,avatar_url,banner_url,bio,is_active,created_at,actor_id";
```
This function is used by `CreateVportForm.jsx.adapter.js` and similar surfaces — not by the settings list. The settings list uses `listMyVportsDAL` from `vports.read.dal.js` which does include `is_deleted`. However, any consumer of `vport.core.dal.listMyVports()` will receive rows without `is_deleted`, which could cause incorrect UI state.

**2. `booking.model.js` — inconsistent resource field name**
```js
ownerActorId: row.booking_resources?.owner_actor_id ?? row.resources?.owner_actor_id ?? null
```
The model tries both `booking_resources` and `resources` as field names. This suggests the DAL query returns `vport.resources` rows but the model was written when the table was called `booking_resources`. Only one name can be correct — the DAL files use `.from('resources')` meaning the column is available as `resources` on an embed.

**3. `vportReviewAuthors.read.dal.js` comment is stale**
```
*  - kind='vport' -> vc.actors.vport_id   -> vc.vports(name, slug, avatar_url)
```
The code actually queries `vport.profiles` (via `vportClient`), not `vc.vports`. The `vc.vports` table no longer exists as the active source. The comment is misleading but the code is correct.

**4. `useVportAccountOps.js` — ctrlRestoreVport adapter is the ONLY correct wiring**
The adapter in `useVportAccountOps` passes `restoreVportFn` correctly. It is consumed only by `useRestoreVport` → `RestoreVportScreen`. The two primary restore surfaces (Account tab, VPORTs tab) bypass this adapter entirely and call `ctrlRestoreVport` directly without `restoreVportFn`. This is the root cause of the restore bug.

### Corrected Relationship Paths

- `vc.actors.vport_id` → `vport.profiles.id` (join via `actor_id` column on profiles, not via embed)
- `vport.profiles.id` → `vport.profile_public_details.profile_id` (direct FK, sync pattern in `setVportDirectoryVisibleDAL`)
- `vport.bookings.resource_id` → `vport.resources.id` (canonical — model should use `row.resources`, drop `row.booking_resources` fallback)

## I. Recommended Fix Plan

The following is an ordered fix plan. No implementation is included here.

### Fix 1 — Add DEV-Only Logging (Instrumentation Before Fixes)

Add the log points described in Section G before changing any business logic. Specifically, add `RESTORE_NO_OP` logging to `ctrlRestoreVport` that fires whenever `restoreVportFn` is undefined. This will confirm the bug is actively hitting production and validate the fix once applied.

### Fix 2 — Fix ctrlRestoreVport: Remove Injectable Pattern, Wire DAL Directly

The injectable `restoreVportFn` pattern in `account.controller.js` is the root cause of the restore bug. Two options:

**Option A (recommended):** Import `dalRestoreVport` directly inside `ctrlRestoreVport`, just like `ctrlSoftDeleteVport` and `ctrlHardDeleteVport` already do with their respective DAL functions. Remove the `restoreVportFn` parameter entirely.
