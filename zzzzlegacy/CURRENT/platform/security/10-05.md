## Execution Summary

### Root Cause — "Forbidden." Error

`{"error":"Forbidden."}` is returned by the Supabase API gateway (not PostgREST) when the
`vport` schema is not listed in Project Settings → API → Exposed schemas.

- Neither `vport.create_vport` nor `vport.create_vport_for_actor` raise "Forbidden." in their bodies
- Both EXECUTE grants on `vport.create_vport` are confirmed `true` for `authenticated`
- The error format (lowercase `error` key, period) is the Supabase gateway response, not PostgREST

**Fix:** Supabase Dashboard → Project Settings → API → Exposed schemas → add `vport`

### Overload Ambiguity Fix

Two overloads of `vport.create_vport` existed:
- 6-arg (no `p_directory_visible`): dead, no app callers, `search_path` has `public` first (CVE-2018-1058)
- 7-arg (with `p_directory_visible`): canonical, called by `vport.core.dal.js`

Diagnostics use `supabase.schema('vc').rpc('create_vport', ...)` which calls `vc.create_vport` —
a separate function in a different schema, unaffected by this migration.

### Migration Applied

File: `_ACTIVE/migrations/2026-05-10_fix_vport_create_rpc_overload_ambiguity.sql`

```sql
DROP FUNCTION IF EXISTS vport.create_vport(text, text, text, text, text, text);

ALTER FUNCTION vport.create_vport(text, text, text, text, text, text, boolean)
  SET search_path TO 'vport', 'vc', 'public', 'auth', 'pg_temp';
```

### App Code Changes

None. `vport.core.dal.js` already sends 7 params including `p_directory_visible`.

### Remaining Action (not a migration)

Apply in Supabase Dashboard before creating a VPORT:
> Project Settings → API → Exposed schemas → add `vport`
