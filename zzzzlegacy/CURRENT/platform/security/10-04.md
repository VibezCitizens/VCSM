## Execution Summary

### Step 1 — Revoke Dangerous EXECUTE

Revoked EXECUTE from `anon` and `authenticated` on:
- `public.admin_delete_user_everywhere` — zero auth guard, any caller could delete any user's data
- `public.mark_all_messages_seen(uuid)` and `public.get_unread_message_total(uuid)` — accepted user ID as parameter, no auth.uid() check
- 10 legacy social functions writing to tables that no longer exist (`public.user_blocks`, `public.followers`, `public.friends`, `public.friend_requests`)
- Revoked anon from 4 functions that should never be called unauthenticated

### Step 2 — RLS Policy Repairs

- `public.profiles` — dropped `profiles_public_read USING (true)` (exposed all profiles including private to anon). Replaced with 3 scoped policies: owner self-read via actor_owners, discoverable-only public read (`publish=true AND discoverable=true`), authenticated self-read
- `learning.courses` — dropped two `USING (true)` permissive policies that let any user read/insert into any org's courses. Replaced with org-scoped `can_current_user_access_course()` and `can_current_user_manage_organization()`
- `wanders.cards` — fixed INSERT policy from `anon` to `authenticated` only
- `vc.friend_ranks` — dropped global `USING (true)`, replaced with self-only read (`owner_actor_id = vc.current_actor_id() OR friend_actor_id = vc.current_actor_id()`)

### Step 3 — App Code Migration (prior session)

`readActorProfile.dal.js` rewritten to use direct queries with explicit column selects instead of calling `vc.read_actor_profile(uuid)`. Required before step 4 could safely drop the RPC.

### Step 4 — Drop Dead Legacy Functions

Dropped 16+ SECURITY DEFINER functions:
- Legacy social: `public.block_user`, `public.follow_user`, `public.send_friend_request`, `public.cancel_friend_request`, `public.respond_friend_request`, `public.is_blocked`, `public.react_to_post`, `public.start_direct_conversation`
- Replaced by direct RLS queries: `public.search_profiles`, `public.get_profile_min_by_id`, `public.get_profile_min_by_username`, `public.profile_preview`, `public.publish_profile`, `public.unpublish_profile`
- Replaced in DAL: `vc.read_actor_profile`

### Step 5 — Harden Remaining SECURITY DEFINER Functions

`public.admin_delete_user_everywhere`:
- Added `learning.platform_admins` admin guard (previously had zero authorization check inside body)
- Locked to `service_role` only via REVOKE/GRANT
- Added `SET search_path`

`public.mark_all_messages_seen`:
- Rewrote to remove uid parameter, use `auth.uid()` internally
- Old uid-param version revoked and dropped

`public.get_unread_message_total(uuid)`:
- No new version created — table it referenced (`vc.inbox_entries`) no longer exists
- App uses `notification.recipients` direct queries instead

Revoked anon from: `vc.create_actor_for_user`, `vc.ensure_my_actor`, `vc.create_vport`, `vc.create_notification`, `chat.send_message_atomic`, `public.handle_new_user`, and others.

### Step 5B — Trigger + Wanders search_path

- `wanders.create_inbox_share_link`
- `wanders.drop_send_card`
- `wanders.redeem_inbox_share_link`

### secdef_a — search_path Hardening (33 functions)

Applied `ALTER FUNCTION ... SET search_path` to 33 SECURITY DEFINER functions across `identity`, `learning`, `omd`, `public`, `vc`, `vport`, `wanders`. No function body, signature, or return type changes. Fixes CVE-2018-1058 schema injection vector.

### secdef_b — Zero-Policy Tables (21 tables)

Added policies to tables with RLS enabled but zero policies. Categories:
- Service-only internal tables (chat.legacy_mappings, chat.outbox_events, learning.student_login_id_counters, etc.) → deny-all authenticated
- Reference/config data (notification.event_categories, platform.app_roles, vport.fuel_types, vport.station_brands, etc.) → public or authenticated read
- `platform.legal_documents` → anon + authenticated read for is_active=true (was fully locked, blocking consent gate)
- `platform.user_app_account_roles` → self-read via join to user_app_accounts
- `vc.professional_housing_*` → authenticated read only

### secdef_c — RLS-Disabled Tables (15 tables)

Enabled RLS and added policies:
- `platform.user_capabilities` → self-read via join to user_app_accounts
- 14 `traffic.*` tables (TRAZE directory) → public read for SEO data; 2 lead capture tables (business_claim_requests, business_intake_leads) → anon INSERT allowed, read denied (PII protection)
- Fixed during execution: `is_deleted` column doesn't exist on `traffic.providers` — corrected to `is_indexable`

### Moderation Batch 1 — can_manage_domain Bug

`moderation.can_manage_domain` had a broken 'vc' branch returning `true` for any user who had a vc actor — effectively granting every platform user moderator-level RLS access. Fixed to require `learning.platform_admins` membership, matching the 'learning' branch.

### Moderation Batch 3 — moderators Table + Circular RLS Fix

Created `moderation.moderators` table with roles: `viewer`, `moderator`, `admin`.

Discovered circular RLS dependency during creation: `can_manage_domain` (SECURITY INVOKER) queries `moderators` → RLS on `moderators` only allowed platform admins to read → non-admin moderators couldn't see their own row → function always returned false for them.

Fixed by adding `moderators_self_read` policy: users can read their own grant row via `vc.actor_owners` join.

### Moderation Batch 4 — block_actor RPC Bidirectional Fix

`moderation.block_actor` RPC previously only deactivated `vc.actor_follows` in one direction (blocker→blocked). The reverse direction (blocked→blocker follows) was missed.

Fixed to deactivate both directions. Backfill applied to fix existing data.

Also moved `vc.friend_ranks` deletion from client-side (`deleteFriendRankRowsBetweenActors()` with swallowed try/catch) into the RPC atomically — deletion now happens in the same transaction as the block write.
