## Proposed Schema

### traffic.cities

```sql
CREATE TABLE traffic.cities (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  state TEXT NOT NULL,
  country TEXT NOT NULL DEFAULT 'US',
  lat DECIMAL(9,6),
  lng DECIMAL(9,6),
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_traffic_cities_slug ON traffic.cities(slug);
CREATE INDEX idx_traffic_cities_active ON traffic.cities(is_active) WHERE is_active = true;
```

### traffic.providers

```sql
CREATE TABLE traffic.providers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug TEXT UNIQUE NOT NULL,

  tagline TEXT,
  description TEXT,

  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),

CREATE INDEX idx_traffic_providers_slug ON traffic.providers(slug);
CREATE INDEX idx_traffic_providers_city ON traffic.providers(city_id);
CREATE INDEX idx_traffic_providers_vcsm_actor ON traffic.providers(vcsm_actor_id) WHERE vcsm_actor_id IS NOT NULL;
CREATE INDEX idx_traffic_providers_claim ON traffic.providers(claim_status);
```

## RLS Policies

Traffic schema is read-only from the public internet. All RLS policies are SELECT-only.

```sql
-- Enable RLS on all traffic tables
ALTER TABLE traffic.cities ENABLE ROW LEVEL SECURITY;
ALTER TABLE traffic.segments ENABLE ROW LEVEL SECURITY;
ALTER TABLE traffic.services ENABLE ROW LEVEL SECURITY;
ALTER TABLE traffic.neighborhoods ENABLE ROW LEVEL SECURITY;
ALTER TABLE traffic.providers ENABLE ROW LEVEL SECURITY;
ALTER TABLE traffic.provider_services ENABLE ROW LEVEL SECURITY;
ALTER TABLE traffic.reviews ENABLE ROW LEVEL SECURITY;

-- Public read for active records only
CREATE POLICY "public_read_active_cities" ON traffic.cities
  FOR SELECT USING (is_active = true);

CREATE POLICY "public_read_active_segments" ON traffic.segments
  FOR SELECT USING (is_active = true);

CREATE POLICY "public_read_active_services" ON traffic.services
  FOR SELECT USING (is_active = true);

CREATE POLICY "public_read_neighborhoods" ON traffic.neighborhoods
  FOR SELECT USING (true);

CREATE POLICY "public_read_active_providers" ON traffic.providers
  FOR SELECT USING (is_active = true);

CREATE POLICY "public_read_provider_services" ON traffic.provider_services
  FOR SELECT USING (true);

CREATE POLICY "public_read_reviews" ON traffic.reviews
  FOR SELECT USING (true);
```

## VCSM Sync Strategy

Traffic is a read-only projection of VCSM provider data. Sync is one-way: VCSM → Traffic.

**Recommended: Event-driven sync via Supabase Edge Function**

```
VCSM actor updated / vport profile saved
  → trigger edge function: syncVcsmActorToTrafficProvider()
  → upsert traffic.providers WHERE vcsm_actor_id = actor.id
  → update: name, tagline, logo_url, vcsm_slug, claim_status
```

**Fallback: Nightly batch sync job**

If webhook architecture is not in place at launch, a nightly cron job via Supabase pg_cron:
```sql
SELECT cron.schedule('sync-traffic-providers', '0 3 * * *', $$
  -- upsert VCSM actor data into traffic.providers
$$);
```

**Never allow Traffic to write to VCSM.** Traffic is a consumer only.

## Build-Time Data Integration

When live data is wired, replace `mockData.ts` with build-time Supabase fetch:

```typescript
// src/lib/trafficData.ts (replaces mockData.ts)
import { createClient } from '@supabase/supabase-js'

const supabase = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_KEY!  // server-only key, never shipped to client
)

export async function getActiveProviders() {
  const { data } = await supabase
    .schema('traffic')
    .from('providers')
    .select('id, slug, name, tagline, logo_url, vcsm_slug, claim_status, city_id')
    .eq('is_active', true)
  return data ?? []
}
```

All `generateStaticParams()` calls consume these functions. Zero client-side DB access.

## Migration Rollback Strategy

All Traffic schema migrations should be:
1. Additive-only at first (new schema, new tables)
2. `DROP TABLE` only after confirmed data migration from mock data
3. No foreign key to VCSM `vc.*` schema — use `vcsm_actor_id UUID` soft reference only (cross-schema FK would create a deployment coupling)
