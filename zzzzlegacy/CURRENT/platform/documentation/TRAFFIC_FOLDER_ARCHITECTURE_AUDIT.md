## 1. Executive Summary

Traffic is a **Programmatic SEO Directory Engine** built with Next.js 14 (App Router). It is a standalone application within the VCSM monorepo that serves one purpose: publish indexable, search-engine-optimized directory pages for local service providers across multiple countries and cities, then funnel organic visitors back to the main Vibez Citizens (VCSM) platform via deep links with tracking parameters.

**What Traffic IS:**
- A public, unauthenticated directory site at `traffic.vibezcitizens.com`
- A static site generator producing country/city/locality/service/provider pages
- An SEO acquisition surface with schema.org structured data, sitemaps, and canonicals
- A conversion funnel routing users to VCSM for booking, following, and profile claiming

**What Traffic IS NOT:**
- Not a social app
- Not part of VCSM's React SPA
- Not part of Wentrex
- Not connected to any database (100% mock data)
- Not authenticated (no sessions, no identity)

**VCSM Connection:**
Traffic connects to VCSM exclusively through outbound deep links. Provider records carry bridge fields (`vcsmActorId`, `vcsmSlug`, `claimedAt`) that enable Traffic to link directly to claimed provider profiles on the main platform. The conversion layer builds parameterized URLs with `source=traffic` tracking for all handoff links.

**Current Scale:**
- 10 countries, 11 regions, 11 cities, 17 localities
- 4 service categories (barber, hair color, makeup, locksmith)
- 25 specialties across those services
- 28 providers with stats, services, and price aggregates
- 2 live verticals (barbers, locksmiths), 3 launching (restaurants, gas stations, money exchange)

## 3. Routing System

Traffic uses Next.js App Router with a `(seo)` route group for all directory pages. The routing system supports two URL schemes simultaneously:

**Canonical (country-prefixed):** `/{country}/{city}`, `/{country}/{city}/{service}`, `/{country}/{city}/{locality}/{service}`, `/{country}/pro/{providerSlug}`

**Legacy (city-first):** `/{city}`, `/{city}/{service}`, `/{city}/{locality}/{service}`, `/pro/{providerSlug}`

### Route Files

#### `src/app/page.jsx` -- Homepage
- **Type:** Static page (no dynamic segments)
- **Data:** Loads all cities, providers, services, localities, stats. Builds featured provider cards, popular city cards, category grid.
- **Renders:** Hero section, service categories grid, popular cities grid, featured providers grid, how-it-works steps, claim CTA, footer.
- **SEO:** Inline metadata export with OG and Twitter cards.
- **Conversion:** Links to `/claim-profile` on VCSM platform, `Explore Providers` link to `/us/san-francisco/locksmith`.
- **Assessment:** This is the richest page in the app. It is well-structured but contains significant inline data assembly logic (~220 lines of helper functions).

#### `src/app/(seo)/[city]/page.jsx` -- Legacy City Page
- **Type:** Legacy route. Handles `/{city}` URLs (e.g., `/san-francisco`).
- **Static Params:** `listCityStaticParams()` -- all active city slugs.
- **Data:** Resolves city by slug, loads country/region, lists providers and services.
- **Renders:** `DirectoryPageTemplate` with breadcrumbs, provider list, related links.
- **SEO:** `buildDirectoryMetadata()`, `buildBreadcrumbSchema()`, `buildDirectoryItemListSchema()`.
- **Assessment:** LEGACY route. The canonical equivalent is `/{country}/{city}` handled by the `[city]/[segment]` route. This page still generates static params and renders content for backward compatibility.

#### `src/app/(seo)/[city]/[segment]/page.jsx` -- Dual-Segment Page
- **Type:** MULTI-RESOLUTION route. Handles both `/{country}/{city}` (canonical) and `/{city}/{service}` (legacy).
- **Resolution Order:** `resolveCountryCity(params)` first, then `resolveLegacyCityService(params)`.
- **Static Params:** Merges `listCountryCityStaticParams()` and `listCityServiceStaticParams()` with deduplication.
- **Data:** Depends on resolution mode. Country-city mode loads all providers/services/localities. Legacy city-service mode loads filtered providers + price aggregates.
- **Renders:** `DirectoryPageTemplate` in both modes, but with different breadcrumbs, related links, and context.
- **SEO:** Mode-specific metadata and schema generation.
- **Assessment:** This is the most complex route file (310 lines). The dual-resolution pattern is clever but creates significant code duplication with the `[city]/page.jsx` for country-city rendering and with the `[city]/[segment]/[service]/page.jsx` for city-service rendering.

#### `src/app/(seo)/[city]/[segment]/[service]/page.jsx` -- Triple-Segment Page
- **Type:** MULTI-RESOLUTION route. Handles `/{country}/{city}/{service}` (canonical), `/{country}/pro/{providerSlug}` (canonical provider), and `/{city}/{locality}/{service}` (legacy).
- **Resolution Order:** `resolveCountryProvider()` first (checks `segment === "pro"`), then `resolveCountryCityService()`, then `resolveLegacyLocalityService()`.
- **Static Params:** Merges three param lists with deduplication.
- **Data:** Provider mode loads full provider graph. City-service mode loads filtered providers + price aggregates. Legacy locality-service mode loads locality-filtered providers.
- **Renders:** `ProviderPageTemplate` for provider mode, `DirectoryPageTemplate` for both directory modes.
- **SEO:** Three distinct metadata builders and schema generators.
- **Assessment:** The most complex route file in the app (518 lines). Handles three fundamentally different page types. While functionally correct, this is the primary candidate for decomposition.

#### `src/app/(seo)/[city]/[segment]/[service]/[detail]/page.jsx` -- Four-Segment Page
- **Type:** CANONICAL-ONLY route. Handles `/{country}/{city}/{locality}/{service}`.
- **Static Params:** `listCountryLocalityServiceStaticParams()`.
- **Data:** Resolves country > city > locality > service chain. Loads locality-filtered providers + price aggregates.
- **Renders:** `DirectoryPageTemplate`.
- **SEO:** `buildDirectoryMetadata()`, breadcrumb + item list schemas.
- **Assessment:** CLEAN. Single-purpose route with no multi-resolution complexity. This is what all route files should aspire to be.

#### `src/app/(seo)/pro/[providerSlug]/page.jsx` -- Legacy Provider Page
- **Type:** LEGACY route. Handles `/pro/{providerSlug}`.
- **Static Params:** `listProviderStaticParams()`.
- **Data:** Builds full provider graph (provider + country + city + locality + region + services + stats).
- **Renders:** `ProviderPageTemplate`.
- **SEO:** `buildProviderMetadata()`, breadcrumb + provider schemas.
- **Assessment:** LEGACY route. The canonical equivalent is `/{country}/pro/{providerSlug}` handled by the triple-segment route. Maintained for backward compatibility.

#### `src/app/sitemap-index.xml/route.js` -- Sitemap Index

- **Data:** `listSitemapChunks()` from `pageCandidate.repo`.

## 10. Global Directory Architecture

### Country-Aware URL Model

Traffic implements a global directory with country-prefixed canonical URLs:

```
/{country}/{city}                              -- city directory
/{country}/{city}/{service}                    -- city+service directory
/{country}/{city}/{locality}/{service}         -- locality+service directory
/{country}/pro/{providerSlug}                  -- provider profile
```

### Geographic Hierarchy

```
Country (10)
  Region (11) -- state/province/department/county/emirate
    City (11) -- tied to country + optional region
      Locality (17) -- neighborhood/district/borough/zone
```

Each entity carries:
- `isActive` flag for soft-delete/launch gating
- Slug for URL generation
- Country/city/region IDs for parent resolution
- Cities carry `countryCode`, `stateCode`, `timezone`
- Localities carry `localityType` for display context

### Currency and Locale
- Countries define `defaultLocale` and `defaultCurrencyCode`
- Providers carry their own `currencyCode`
- Price aggregates carry `currencyCode`
- `Intl.NumberFormat` handles locale-aware formatting with error fallback

### Legacy Path Support
Traffic maintains full backward compatibility with pre-global URLs:

| Legacy Pattern | Canonical Equivalent |
|----------------|---------------------|
| `/{city}` | `/{country}/{city}` |
| `/{city}/{service}` | `/{country}/{city}/{service}` |
| `/{city}/{locality}/{service}` | `/{country}/{city}/{locality}/{service}` |
| `/pro/{providerSlug}` | `/{country}/pro/{providerSlug}` |

This is implemented through multi-resolution route files that try canonical resolution first, then fall back to legacy resolution. Both legacy and canonical URLs render the same content but canonical URLs are used for `rel=canonical` and sitemaps.

### What Still Needs Improvement
1. **No HTTP redirects from legacy to canonical.** Both URL patterns render content. For SEO, legacy URLs should 301 redirect to canonical equivalents.
2. **Sitemaps only include canonical URLs by default.** `listPageCandidates()` only generates global candidates unless `includeLegacy: true` is passed. This is correct for SEO but means legacy URLs are not discoverable via sitemap.
3. **No country-level landing pages.** There is no `/{country}` route. Users jump from homepage directly to `/{country}/{city}`.
4. **No region-level pages.** The region model exists in data but has no route representation (e.g., `/us/california`).

## 12. Duplications / Architecture Problems

### P1: Route File Duplication (HIGH)
The `[city]/[segment]/page.jsx` duplicates the entire city page rendering logic from `[city]/page.jsx`. The `[city]/[segment]/[service]/page.jsx` duplicates city-service rendering from `[city]/[segment]/page.jsx` AND provider rendering from `pro/[providerSlug]/page.jsx`. Approximately 400 lines of rendering code are duplicated across route files.

### P2: Multi-Resolution Route Complexity (HIGH)
The triple-segment route (`[city]/[segment]/[service]/page.jsx`) handles three fundamentally different page types (provider, city-service, locality-service) in 518 lines. This makes the file hard to reason about, test, and maintain.

### P3: Naming Debt -- neighborhood vs locality (MEDIUM)
The rename from "neighborhood" to "locality" is incomplete:
- Mock data array: `MOCK_NEIGHBORHOODS` (old name)
- Type: `Neighborhood` aliased to `Locality`
- Repositories: backward-compatible aliases exist (`listNeighborhoodsByCity`, etc.)
- Quality guards: function names use `neighborhood` (`isNeighborhoodServiceIndexable`)
- Paths: `neighborhoodServicePath()`, `neighborhoodServiceSpecialtyPath()`
- Page candidate repo: `listNeighborhoodServiceStaticParams()`
- Price aggregates: `neighborhoodId` field name
This creates cognitive load and will cause confusion as the codebase grows.

### P4: Homepage Inline Logic (MEDIUM)
The homepage (`page.jsx`) contains ~220 lines of helper functions for data assembly (`getCityPopularityMap`, `getServiceMixForCity`, `getFeaturedProviderCards`, `getPopularCityCards`, `buildClaimLandingLink`, `buildMainPlatformLink`, `getClaimBadge`). These should be extracted to a homepage-specific data module.

### P5: Monolithic Mock Dataset (LOW-MEDIUM)
`mockDataset.js` at 1291 lines is unwieldy. When the dataset grows to hundreds of providers across more verticals, this file will become unmanageable. Should be split by entity type (countries.mock.js, providers.mock.js, etc.) or eventually replaced by database queries.

### P6: TypeScript DevDependencies in JS Project (LOW)
`package.json` includes `@types/node`, `@types/react`, `@types/react-dom`, and `typescript` as devDependencies. The project uses `jsconfig.json` and `.jsx` files exclusively. The `typecheck` script (`tsc --noEmit`) will not work meaningfully without TypeScript files.

### P7: No Legacy-to-Canonical Redirects (MEDIUM)
Legacy URLs (`/san-francisco/locksmith`) and canonical URLs (`/us/san-francisco/locksmith`) both render content. This creates duplicate content risk for search engines. Canonical tags are set correctly, but 301 redirects would be stronger signals.

### P8: Quality Thresholds at Minimum (LOW)
All indexability thresholds in `qualityGuards.js` are set to 1. This means every city/service combination with even a single provider generates a page. For production, this should be raised to prevent thin content penalties.

### P9: Single Responsive Breakpoint (LOW)
`globals.css` has only one breakpoint at 760px. The grid layouts use `auto-fit` with `minmax()` which provides some responsiveness, but tablet and large-mobile ranges are not specifically addressed.

### P10: No Error Boundaries (LOW)
No route page has error boundary or fallback UI. If data resolution fails unexpectedly (not `notFound()` but an actual error), the page will crash without graceful degradation.

## 13. Refactor Recommendations

### Priority 1 -- High Value, Architecture Debt

**R1: Extract shared page renderers from route files.**
Create `src/app/(seo)/_renderers/` with shared functions like `renderCountryCityPage()`, `renderCityServicePage()`, `renderLocalityServicePage()`, `renderProviderPage()`. Each route file would become a thin resolver that calls the appropriate renderer. Eliminates ~400 lines of duplication.

**R2: Complete the neighborhood-to-locality rename.**
Rename `MOCK_NEIGHBORHOODS` to `MOCK_LOCALITIES`, update all `neighborhood*` function names and parameters, update `neighborhoodId` in `PriceAggregate`, remove backward-compatible aliases. This is mechanical but prevents growing confusion.

**R3: Add 301 redirects for legacy paths.**
Add a Next.js middleware or `next.config.mjs` redirects that 301 legacy URLs to their canonical equivalents. Critical for SEO before production launch.

### Priority 2 -- Medium Value, Cleanup

**R4: Extract homepage data assembly.**
Move `getFeaturedProviderCards()`, `getPopularCityCards()`, and related helpers to `src/data/repositories/homepage.repo.js` or `src/app/homepage.data.js`.

**R5: Raise quality thresholds for production.**
Implement environment-aware thresholds in `qualityGuards.js` (e.g., `cityServiceMinProviders: 2` in production, `1` in development).

**R6: Remove TypeScript devDependencies.**
Remove `@types/node`, `@types/react`, `@types/react-dom`, `typescript` from `package.json`. Remove the `typecheck` script.

### Priority 3 -- Low Value, Polish

**R7: Split mockDataset.js.**
Break into `countries.mock.js`, `cities.mock.js`, `providers.mock.js`, etc. with a barrel `index.js`.

**R8: Add responsive breakpoints.**
Add 480px (mobile), 1024px (tablet), 1280px (desktop) breakpoints to `globals.css`.

**R9: Add error boundaries.**
Add `error.jsx` files to route groups for graceful error handling.

**R10: Add country-level landing pages.**
Create `/{country}` routes that list cities within a country.

### Priority 4 -- Future (post-launch)

**R11: Replace mock data with Supabase connector.**
The repository pattern makes this straightforward -- replace `MOCK_*` array scans with Supabase queries. The repository API surface stays the same.

**R12: Add ISR (Incremental Static Regeneration).**
Replace full static generation with ISR for frequently updated pages (provider stats, new providers).
