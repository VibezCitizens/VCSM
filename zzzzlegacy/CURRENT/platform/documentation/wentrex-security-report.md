## HIGH FINDINGS

---

### VENOM FINDING #1 — HIGH
- **Location:** Multiple DAL files — exemplified by `apps/wentrex/src/learning/administration/dal/courses/getCourseById.dal.js`
- **Application Scope:** WENTREX
- **Current behavior:** Multi-tenant isolation is enforced at the application (controller) layer only. DAL queries fetch by courseId, then the controller validates `courseRow.realm_id !== realmId`. The database query itself has no realm_id filter.
  ```javascript
  // DAL — no realm filter
  .eq("id", courseId)
  .maybeSingle()

  // Controller — post-fetch validation
  if (!courseRow || courseRow.realm_id !== realmId) {
    return { ok: false, error: { code: "COURSE_NOT_FOUND" } };
  }
  ```
- **Risk:** If the controller validation is bypassed, miscalled, or has a logic bug, cross-tenant data leaks. Any DAL called directly without a controller wrapper has no isolation guarantee.
- **Severity:** HIGH
- **Why it matters:** This is the core SaaS contract — one school must never see another school's data.
- **Recommended mitigation:**
  1. Add `eq("realm_id", realmId)` directly to DAL queries that are accessed by courseId
  2. Alternatively, enforce realm scoping via DB-level views that filter by current user's authorized realm
  3. Add integration tests specifically for cross-realm access rejection
- **Follow-up command:** BUGSBUNNY + DB

### VENOM FINDING #2 — HIGH
- **Location:** `apps/wentrex/src/learning/administration/dal/diagnostics/adminDiagnostics.dal.js`
- **Application Scope:** WENTREX
- **Current behavior:**
  - `listVisibleOrganizationsDal()` — returns ALL organizations globally with no realm filtering
  - `listVisibleRealmsDal()` — returns ALL realms globally with no realm filtering
  - Both called from the admin diagnostics controller
- **Risk:** Any admin from any school can see ALL tenants, ALL organizations, ALL realms in the system. Full platform tenant enumeration.
- **Severity:** HIGH
- **Why it matters:** A bad-faith admin at one school could enumerate all other schools on the platform.
- **Recommended mitigation:**
  1. Gate diagnostics endpoint to platform admins only (not school admins)
  2. Add realm scoping to both queries
  3. Consider removing `listVisibleRealms` / `listVisibleOrganizations` entirely — school admins only need their own realm's data
- **Follow-up command:** BUGSBUNNY

### VENOM FINDING #3 — MEDIUM
- **Location:** `apps/wentrex/src/learning/screens/ChangePasswordScreen.jsx:89`
- **Application Scope:** WENTREX
- **Current behavior:** Email address is logged to browser console even inside a DEV check. Access token last 8 characters are also logged (lines 68, 87).
  ```javascript

  ```
- **Risk:** PII (email) exposed in browser console, potentially captured in screen recordings, Sentry integrations, or shared sessions.
- **Severity:** MEDIUM

### VENOM FINDING #4 — MEDIUM
- **Location:** `apps/wentrex/src/features/identity/WentrexIdentityContext.jsx` + `provisionWentrexIdentity.controller.js`
- **Application Scope:** WENTREX
- **Current behavior:** `provisionWentrexIdentity()` resolves a learning actor by user_id globally. It does NOT verify the actor belongs to the requested realm slug. Context is returned with `activeActor` and `organizationId` but no explicit realm cross-check during session init.
- **Risk:** A user with actors in multiple tenants could access the wrong tenant if URL slug doesn't match their actor's realm.
- **Severity:** MEDIUM
- **Recommended mitigation:** Add realm validation in `provisionWentrexIdentity()` — verify that the resolved actor's realm matches the `realmSlug` from the URL before returning context.

### VENOM FINDING #5 — MEDIUM

- **Current behavior:** Controller called by any admin — returns all memberships, all visible organizations, all visible realms. No realm scoping on the controller call.

### VENOM FINDING #6 — MEDIUM
- **Location:** `apps/wentrex/src/learning/administration/controller/admin/linkParentToStudent.controller.js:24-47`
- **Application Scope:** WENTREX
- **Current behavior:** Parent-to-student link controller verifies `courseRow.realm_id !== realmId` post-fetch, but the link creation itself uses `courseId` as the tenant anchor. If courseId is spoofed from a different realm, a cross-tenant observer link could be created.
- **Risk:** Low exploitability (UUIDs not guessable) but defense-in-depth is weak.
- **Severity:** MEDIUM (LOW-MEDIUM)
- **Recommended mitigation:** Add realm_id validation to the DAL before creating the link. Verify course exists in the provided realm BEFORE creating the observer_student_link row.
