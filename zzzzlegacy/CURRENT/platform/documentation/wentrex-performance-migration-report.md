## KRAVEN FINDING #2 — LOW

**Serial Controller Chain — Student Course Load**

`buildStudentCourseData()` controller calls multiple DALs sequentially:

```
getCourseById()
  → listModulesByCourseId()
    → listLessonsByModuleIds()
      → listLessonProgressByCourseAndActor()
        → listSubmissionsByAssignmentId()
```

These are sequential — each waits for the previous. `listModulesByCourseId` and `listLessonProgressByCourseAndActor` could be parallelized with `Promise.all()`.

**Recommended:** Fan out independent fetches in parallel inside the controller.

## CARNAGE FINDING #1 — HIGH

**lesson_progress Queries Missing organization_id Filter**

| DAL | Current Filters | Gap |
|-----|----------------|-----|
| `listLessonProgressByCourseAndActor.dal.js` | actor_id + course_id | No organization_id |
| `getLessonProgressByLessonAndActor.dal.js` | lesson_id + actor_id | No organization_id |

These rely on FK chain (lesson → course → organization) and RLS policies alone.

**Risk:** If RLS on `learning.lesson_progress` does not explicitly filter by organization_id, cross-organization leakage is possible when actors have overlapping courses across orgs.

**Recommended action:**
1. Run: `SELECT * FROM pg_policies WHERE tablename = 'lesson_progress';`
2. Verify RLS contains `organization_id` or `realm_id` filter
3. If not: add `eq("organization_id", organizationId)` to both DAL queries as defense-in-depth
