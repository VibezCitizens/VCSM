## Navigation Architecture Overview

The VCSM navigation system uses a floating bottom nav bar (`BottomNavBar`) with 7 icons, always mounted inside `RootLayout` (CSS-hidden on chat sub-screens, auth, learning, and dev routes to preserve realtime subscriptions and polling).

**Nav bar component:** `apps/VCSM/src/shared/components/BottomNavBar.jsx`
**Layout wrapper:** `apps/VCSM/src/app/layout/RootLayout.jsx`
**Route definitions:** `apps/VCSM/src/app/routes/protected/app.routes.jsx`
**Route index:** `apps/VCSM/src/app/routes/index.jsx`

The nav bar uses `NavLink` (React Router) for all tabs except Upload, which uses `useNavigate()` via a `<button>` click handler.

Two badge hooks run continuously from `BottomNavBar` while mounted:
- `useNotiCount` (notifications badge, 45s poll)
- `useUnreadBadge` (chat unread badge, 15s poll)

Both use identity store (`useIdentity`) to resolve the current `actorId`.

## 1. Profile Navigation

Updated: 2026-04-18 — Profile URLs are now UUID-free. Bottom nav links to `/profile/self`.

### Icon Location
`apps/VCSM/src/shared/components/BottomNavBar.jsx`
```
<Tab to={profilePath} label="Citizen" icon={<User size={18} />} />
```

### Route Triggered
- Always: `/profile/self` when identity is resolved
- Fallback: `/feed` when `personaActorId` is null
- Computed: `const profilePath = personaActorId ? '/profile/self' : '/feed'`
- Uses `NavLink` (declarative navigation)

The `/profile/self` route is a redirect shortcut — `ActorProfileScreen` immediately
redirects to the viewer's canonical slug URL (e.g. `/profile/architect1`). No profile
UUID is ever visible in the browser URL.

### Page Entry
`apps/VCSM/src/features/profiles/screens/ActorProfileScreen.jsx`

### URL Resolution Flow (ActorProfileScreen)

Three strict resolution paths — no other source is accepted:

| Input | Resolution | Example |
|-------|-----------|---------|
| `"self"` | `identity.actorId` → canonical slug → redirect | `/profile/self` → `/profile/architect1` |
| UUID in param | Extract UUID → canonical slug → redirect | `/profile/{uuid}` → `/profile/tyba-restaurant` |
| Slug/username | `useResolveActorBySlug` → DB lookup → actorId | `/profile/architect1` → render |

### Controllers Called
1. **`buildActorCanonicalSlugController`** - `features/profiles/controller/buildActorCanonicalSlug.controller.js`
   - Computes the canonical UUID-free slug for a resolved actorId
   - Called via `useActorCanonicalSlug` hook for "self" and UUID paths
2. **`getActorKindController`** - `features/profiles/controller/getActorKind.controller.js`
   - Resolves whether the actor is `user` or `vport` kind
   - Called via `useActorKind` hook once canonical URL is matched

### DAL Queries
1. **`resolveActorBySlugOrUsernameDAL`** - `features/profiles/dal/readActorSeoData.dal.js`
   - Reverse lookup: slug/username → actorId
   - Tries `vport.profiles.slug`, then `public.profiles.username → vc.actors`
   - Only called for slug-based URLs (not "self" or UUID)
2. **`readActorDirectoryRowDAL`** - `features/profiles/dal/readActorSeoData.dal.js`
   - Reads `vc.actors.kind + profile_id`, then `public.profiles.display_name + username`
3. **`readVportProfileByActorDAL`** - `features/profiles/dal/readActorSeoData.dal.js`
   - Reads `vport.profiles.name + slug` for vport actors
4. **`readActorKindDAL`** - `features/profiles/dal/readActorKind.dal.js`
   - Reads `kind` from `vc.actors` for kind-based screen dispatch

### Engines Involved
- **Identity engine** (`@identity`) - via `useIdentity()` for viewer actorId and "self" resolution
- **Hydration engine** (`@hydration`) - via `useActorStore` for slug fallback when DB fields are absent

### State Updates
- `identityContext` (useIdentity) - reads actorId for "self" resolution
- Profile kind registry dispatches to the correct screen component after canonical URL is matched

### Side Effects
- **Redirect (replace history):** `useActorSlugRedirect` fires `navigate('/profile/{canonical}', { replace: true })` when routeParam ≠ canonicalSlug
- DEV only: `_DevProbe` overlay renders pipeline state at every render phase (shows resolution path, redirect pending state, slug not found, etc.)

### User vs Vport Branch (after canonical URL matched)
- `kind === 'user'` → `ActorProfileViewScreen`
- `kind === 'vport'` → `VportProfileKindScreen` → `VportProfileViewScreen`

### Execution Flow
```
Icon (NavLink) -> /profile/self
  -> ActorProfileScreen
    -> isSelf=true → actorIdForSelf = identity.actorId
    -> useActorCanonicalSlug(identity.actorId) -> buildActorCanonicalSlugController
        -> readActorDirectoryRowDAL (vc.actors + public.profiles)
        -> readVportProfileByActorDAL (vport.profiles)
        -> ActorSeoModel -> canonicalSlug e.g. "architect1"
    -> useActorSlugRedirect("self", "architect1", identity.actorId)
        -> navigate('/profile/architect1', { replace: true })
  -> [now at /profile/architect1]
    -> slugToResolve = "architect1"
    -> useResolveActorBySlug("architect1") -> resolveActorBySlugOrUsernameDAL
        -> public.profiles WHERE username = "architect1" -> actorId
    -> routeParam === canonicalSlug → render
    -> useActorKind(actorId) -> getActorKindController -> readActorKindDAL
    -> PROFILE_KIND_REGISTRY[kind] -> ActorProfileViewScreen | VportProfileKindScreen
```

### Vport Profile Tab Auto-Selection

`VportProfileViewScreen` manages an `effectiveTabs` array that is type-specific once `publicDetails` resolves. The default tab is determined by the first key of `effectiveTabs`.

**Problem that existed:** Two effects ran — the first locked onto a generic fallback tab ("about") before `publicDetails` loaded. The second effect couldn't override it because it checked `prev === "vibes"` which was no longer true.

**Current behavior (2026-04-18):**

One effect using `autoAppliedFirstKeyRef`:
- Tracks the last `firstKey` that was auto-applied
- When `effectiveTabs` recomputes and `firstKey` changes (e.g. fallback "about" → gas-specific "gas"), advances to the new `firstKey` regardless of current tab value
- Only fires when `!userHasSelectedTabRef.current` (user hasn't manually picked a tab)
- Once user manually selects any tab, auto-advance is permanently disabled for the session

**Refs used:**
| Ref | Purpose |
|-----|---------|
| `autoAppliedFirstKeyRef` | Last firstKey that was auto-applied — compared on each effectiveTabs change |
| `userHasSelectedTabRef` | Set to `true` by `handleTabSelect`; disables all auto-advance permanently |

**Result:** Gas station vports land on the "Gas" tab immediately after `publicDetails` resolves, even if the generic fallback briefly set "about".

## 8. Public Menu Routing

Updated: 2026-04-18

### Canonical URL
```
/profile/:slug/menu
```
`slug` = actor's canonical human-readable slug (e.g. `yankee-gasolina-laredo-tx`). Raw UUIDs never appear in public menu URLs.

### Route Table

| Path | Screen | Purpose |
|------|--------|---------|
| `/profile/:slug/menu` | `VportPublicMenuBySlugScreen` | Canonical public menu — slug-based |
| `/m/:actorId` | `VportPublicMenuRedirectScreen` | Short link → resolves slug → redirects to `/profile/:slug/menu` |
| `/actor/:actorId/menu` | `VportPublicMenuScreen` | Legacy → resolves slug → redirects to `/profile/:slug/menu` |
| `/actor/:actorId/menu/qr` | `VportPublicMenuQrScreen` | Owner QR view (protected) |
| `/actor/:actorId/menu/flyer` | `VportActorMenuFlyerScreen` | Owner flyer builder (protected, feature-flagged) |

### Slug Resolution DAL Note

The canonical route `/profile/:slug/menu` resolves slugs via `resolveMenuSlugDAL`, which queries `vport.public_menu_read_model_v`. This is intentionally **not** `resolveActorBySlugOrUsernameDAL`, which queries `vport.profiles` directly and is not accessible to anonymous users.

### Splash Screen Behavior

The splash screen bypasses immediately on `/profile/:slug/menu`. The bypass regex was added to `apps/VCSM/index.html` on 2026-04-18:
```js
/^\/profile\/[^/]+\/menu\/?$/.test(path)
```

### Resolution Flow

**New canonical path:**
```
/profile/:slug/menu
  → VportPublicMenuBySlugScreen
    → useResolveActorBySlug(slug) → actorId  [via resolveMenuSlugDAL → vport.public_menu_read_model_v]
    → VportPublicMenuView(actorId)
```

**Short link legacy path:**
```
/m/:actorId
  → VportPublicMenuRedirectScreen
    → buildActorCanonicalSlugController(actorId) → canonicalSlug
    → navigate(/profile/${canonicalSlug}/menu, replace)
    → fallback: navigate(/actor/${actorId}/menu, replace)  [if slug fails]
```

**Old actorId path:**
```
/actor/:actorId/menu
  → VportPublicMenuScreen
    → useActorCanonicalSlug(actorId) → canonicalSlug
    → <Navigate to=/profile/${canonicalSlug}/menu replace />  [once slug resolves]
    → renders VportPublicMenuView while slug loads (no flash)
```

### URL Generation (Owner Tools)
`VportPublicMenuQrView` (QR modal) and `VportActorMenuFlyerView` (flyer builder) resolve `useActorCanonicalSlug(actorId)` before constructing the share URL:
```js
menuUrl = canonicalSlug
  ? `${origin}/profile/${canonicalSlug}/menu`
  : `${origin}/m/${actorId}`  // fallback only while slug loading
```

**Note:** `VportActorMenuQrView` and `VportActorMenuQrScreen` (`features/dashboard/qrcode/menu/`) were confirmed dead code — never imported by the router. Deleted 2026-04-18.

### Hard Rule
Raw `actorId` UUIDs must never appear in public-facing menu URLs. This applies to QR payloads, Open Menu buttons, Copy Link, share links, and any URL constructed for customer-facing consumption.
