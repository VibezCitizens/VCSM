### KRAVEN PERFORMANCE FINDING #1 — MEDIUM
- **Location:** `apps/VCSM/src/features/profiles/controller/photos/photoReactions.controller.js:27-29`
- **Detected pattern:** Serial Chain
- **Estimated impact:** MEDIUM
- **Root cause:** Three metadata queries execute sequentially:
  ```js
  const reactions = await listPostReactions(...)
  const commentCounts = await listPostCommentsCount(...)
  const roseCounts = await listPostRoseCount(...)
  ```
  These are fully independent reads. Sequential execution triples the latency for the photos tab.
- **Recommended optimization:**
  ```js
  const [reactions, commentCounts, roseCounts] = await Promise.all([
    listPostReactions(...),
    listPostCommentsCount(...),
    listPostRoseCount(...)
  ])
  ```
- **Expected improvement:** ~2/3 reduction in metadata fetch time for the photos tab.

### KRAVEN PERFORMANCE FINDING #2 — MEDIUM
- **Location:** `apps/VCSM/src/features/profiles/kinds/vport/dal/services/readVportTypeByActorId.js:21-40`
- **Detected pattern:** Sequential two-step query (N+1 risk in bulk context)
- **Estimated impact:** MEDIUM
- **Root cause:** Per-actorId call: first query fetches actor to get vport_id, then second query fetches profile_categories for that vport_id. When called per-item in a vport list context, this becomes N+1 (2 queries per vport × N vports).
- **Recommended optimization:** For bulk operations, fetch all vport_ids from actors in one query, then batch-fetch all profile_categories with `.in("profile_id", vportIdArray)`. Consolidate into one round trip.
- **Expected improvement:** Reduces from 2N queries to 2 queries for any bulk vport type resolution.

### KRAVEN PERFORMANCE FINDING #3 — MEDIUM
- **Location:** `apps/VCSM/src/features/booking/dal/listBookingsInRange.dal.js:43-75`
- **Detected pattern:** Index-Aware Range Query (potential missing index)
- **Estimated impact:** MEDIUM (grows with data volume)
- **Root cause:** Calendar query uses `lt("starts_at", endIso)` and `gt("ends_at", startIso)` filtered by `resource_id`. If no composite index on `(resource_id, starts_at, ends_at)`, this will full-scan the bookings table as booking count grows.
- **Recommended optimization:** Verify and add index (see CARNAGE report for SQL). No code change needed.
- **Expected improvement:** Sub-linear query time at scale.
