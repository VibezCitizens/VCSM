## 1. Executive Summary

Traffic is a Next.js 14 programmatic SEO directory that generates city/service/provider pages for search engine indexing. It currently runs on 100% mock data with no database. VCSM has a mature vport (business storefront) system with actor-based identity, public details, services, portfolio, reviews, menu, and booking surfaces.

The two systems have **zero runtime integration today**. Traffic providers are hardcoded JS objects. VCSM vports live in PostgreSQL behind Supabase RLS. However, Traffic already has architectural hooks that anticipate VCSM integration: a `claimStatus` field on providers, deep link builders that route to VCSM with `source: "traffic"` attribution, and CTA modules that render "Claim Profile" buttons for unclaimed providers.

The critical missing piece is a **provider identity bridge** — Traffic providers have no `vcsmActorId` or `vcsmVportId` field, and no projection layer exists to sync VCSM vport data into Traffic's rendering pipeline.

**Recommended architecture: Option C (Hybrid)** — Traffic maintains its own provider records (including unclaimed/seed listings) with an optional `vcsmActorId` foreign key. Claimed providers pull enriched data from VCSM vport public surfaces. Unclaimed providers render from Traffic's own records until claimed.

## 3. VCSM Current Vport Architecture

### 3.1 Actor-Based Identity

VCSM uses an actor-based identity system. All entities are scoped to `vc.actors`:

| Table | Purpose |
|---|---|
| `vc.actors` | Central identity table. Fields: `id`, `kind` (`'user'` or `'vport'`), `profile_id`, `vport_id`, `is_void` |
| `vc.vports` | Business storefront records. Fields: `id`, `owner_user_id`, `name`, `slug`, `avatar_url`, `bio`, `is_active`, `banner_url`, `vport_type` |
| `vc.actor_owners` | Ownership link: `actor_id` -> `user_id`. A user can own multiple actors (personal + vport actors) |
| `vc.actor_privacy_settings` | Privacy settings per actor |

**Key relationship:** `vc.actors.vport_id` -> `vc.vports.id`. An actor of kind `'vport'` has a linked vport row.

## 5. Identity / Ownership Model

### 5.1 VCSM Actor Model

```
vc.actors
  id          — UUID, primary identity key
  kind        — 'user' | 'vport'
  profile_id  — FK to public.profiles (for kind='user')
  vport_id    — FK to vc.vports (for kind='vport')
  is_void     — realm partition flag

vc.actor_owners
  actor_id    — FK to vc.actors.id
  user_id     — FK to auth.users.id
  (A user can own multiple actors: their personal + N vport actors)

vc.vports
  id          — UUID
  owner_user_id — FK to auth.users.id
  name        — Business name
  slug        — URL slug (unique)
  vport_type  — Enum string (77+ types)
  avatar_url  — Logo/avatar
  bio         — Description
  is_active   — Active flag
```

### 5.2 How Ownership Works

1. User authenticates via Supabase Auth
2. Identity engine resolves: `auth.users.id` -> `vc.actor_owners` -> `vc.actors`
3. User selects active actor (personal or vport)
4. All operations scoped to `actorId`

**File:** `apps/VCSM/src/features/settings/vports/dal/vports.read.dal.js`

The `readMyVports()` DAL demonstrates the ownership chain:
```
auth.users.id -> vc.actor_owners.user_id -> vc.actors (kind='vport') -> vc.vports
```

### 5.3 Actor ID Resolution

**File:** `apps/VCSM/src/features/profiles/kinds/vport/dal/readVportActorIdByVportId.dal.js`

Can resolve `vportId` -> `actorId` via: `vc.actors WHERE kind='vport' AND vport_id = ?`

### 5.4 Identity for Traffic

For Traffic to link to VCSM, it needs one of:
- `actorId` (preferred — the universal identity key)
- `vportSlug` (resolvable to actorId via `vc.vports.slug`)
- `vportId` (resolvable but less preferred — VCSM prefers actor-first)

**Currently Traffic has none of these.**

## 6. Missing Integration Bridge

### 6.1 No VCSM Identity Field on Traffic Providers

The Traffic `Provider` type has no field that references a VCSM actor, vport, or slug. The `id` field (`"pro-tripoint-lock"`) is Traffic-internal.

**Missing field:** `vcsmActorId: string | null` on the Provider type.

### 6.2 No Public Projection Layer

There is no service, Edge Function, or API that:
- Returns a list of active VCSM vports suitable for directory indexing
- Returns public vport data in a shape Traffic can consume
- Maps VCSM vport types to Traffic service taxonomy
- Provides aggregate stats (rating, review count) in a Traffic-compatible format

### 6.3 No Slug Resolution

Traffic deep links use `providerSlug` (Traffic slug: `"tripoint-lock-keys-sf"`). VCSM vports have their own slugs (e.g., from `vc.vports.slug`). There is no cross-reference or shared slug namespace.

### 6.4 No Sync Pipeline

No mechanism exists to:
- Push VCSM vport data to Traffic
- Pull VCSM vport data into Traffic at build time
- Notify Traffic when a vport profile changes
- Keep Traffic provider stats in sync with VCSM review aggregates

### 6.5 No Claim Flow Backend

The claim CTA button exists in Traffic UI (`buildPlatformClaimLink`), but:
- VCSM has no `/claim-profile` route handler
- No claim verification flow exists (email verification, ownership proof)
- No mechanism to attach a Traffic provider record to an existing VCSM vport
- No mechanism to create a new VCSM vport from a Traffic claim

### 6.6 No Geo Taxonomy Bridge

Traffic has its own geo taxonomy (countries, regions, cities, neighborhoods) with Traffic-internal IDs. VCSM vports store location in `vport_public_details.address` (jsonb) and `location_text` (freetext). There is no shared geo taxonomy.

### 6.7 No Service Taxonomy Bridge

Traffic services (`svc-barber`, `svc-locksmith`) are a separate catalog from VCSM vport types (`"barber"`, `"locksmith"`). The concepts align semantically but have no formal mapping.

## 7. Claim Flow Readiness

### 7.1 What Exists

- Traffic renders a "Claim Profile" CTA for unclaimed providers
- The CTA links to `vibezcitizens.com/claim-profile?provider={slug}&source=traffic`
- The `claimStatus` field on Traffic providers supports three states: `"claimed"`, `"unclaimed"`, `"pending"`

### 7.2 What Is Missing

| Component | Status | Effort |
|---|---|---|
| VCSM `/claim-profile` route | Does not exist | Medium |
| Claim verification flow (email/SMS/document) | Does not exist | High |
| Traffic provider -> VCSM actor resolution | Does not exist | Low |
| Claim status sync (VCSM -> Traffic) | Does not exist | Medium |
| Duplicate detection (prevent multiple claims) | Does not exist | Medium |
| Admin moderation queue for claims | Does not exist | Medium |

### 7.3 Recommended Claim Flow

1. Provider clicks "Claim Profile" on Traffic -> opens VCSM `/claim-profile?provider=tripoint-lock-keys-sf`
2. VCSM requires authentication (login or signup)
3. VCSM checks if a vport with matching slug/name already exists
4. If yes: link existing vport actor to the Traffic provider record
5. If no: create new vport, populate from Traffic provider data, link
6. Verification step: business email, phone, or document proof
7. On approval: update Traffic provider `claimStatus` to `"claimed"` and set `vcsmActorId`
8. Traffic re-renders with enriched VCSM data (reviews, portfolio, booking links)

## 8. Recommended Architecture Model

### Recommendation: Option C — Hybrid Model

**Traffic maintains its own provider records with an optional VCSM actor link.**

```
Traffic Provider Record
├── id (Traffic-internal)
├── slug
├── displayName
├── claimStatus ("unclaimed" | "pending" | "claimed")
├── vcsmActorId (nullable — set when claimed)
├── ...provider fields (address, phone, bio, services)
└── ...stats (rating, reviews — from Traffic or VCSM)

When vcsmActorId is NULL:
  → Render from Traffic's own data
  → Show "Claim Profile" CTA

When vcsmActorId is set:
  → Enrich from VCSM public surfaces at build time
  → Show live booking/follow CTAs
  → Link schema.org sameAs to VCSM profile
```

### Why Not Option A (Traffic as Standalone DB)

- Traffic would need to duplicate all of VCSM's vport data management (services, reviews, portfolio, booking)
- Two sources of truth for the same business data
- No benefit from VCSM's existing review, portfolio, and booking systems
- Claim flow becomes meaningless if Traffic owns its own data

### Why Not Option B (Traffic as Pure VCSM Projection)

- Traffic needs to list providers that do NOT yet exist in VCSM (seed/directory listings for SEO)
- Pure projection means zero directory coverage until vports are created in VCSM
- SEO requires breadth — thousands of provider pages — before real businesses sign up
- Cannot build SEO gravity with an empty directory

### Why Option C Wins

- Traffic can seed thousands of unclaimed listings immediately (SEO gravity)
- Claimed providers get enriched with real VCSM data (reviews, portfolio, booking)
- Provider ownership is clear: unclaimed = Traffic-owned seed, claimed = VCSM-linked
- Build-time data pipeline: at `next build`, fetch VCSM public data for claimed providers
- Graceful degradation: if VCSM is down, Traffic falls back to its own records
- Mirrors real-world directory patterns (Google My Business, Yelp)

## 9. Risks If Implemented Incorrectly

### 9.1 Data Sovereignty Violation

If Traffic writes directly to VCSM's `vc.*` tables, it violates the app isolation contract. Traffic must only READ from VCSM — never write. All writes (claim flow, review sync) must go through Edge Functions or VCSM-owned APIs.

### 9.2 Slug Collision

If Traffic generates slugs independently and a VCSM vport has a different slug for the same business, deep links will break. Slug resolution must be bidirectional or use a shared slug registry.

### 9.3 Stale Data

If Traffic caches VCSM data at build time and the VCSM vport updates between builds, users see stale data. Mitigation: ISR (Incremental Static Regeneration) with reasonable revalidation windows, or a webhook from VCSM on vport updates.

### 9.4 RLS Bypass

Traffic should never use a Supabase service role key in client-side code. All VCSM data access must go through Edge Functions or a build-time data pipeline that runs server-side.

### 9.5 Claim Fraud

Without verification, anyone could claim any business listing. The claim flow must include identity verification (business email, phone verification, or document upload).

### 9.6 Review Count Divergence

Traffic currently has mock `ProviderStats.reviewCount`. If claimed providers show VCSM review counts but unclaimed show zero, the disparity looks suspicious. Consider synthetic review counts or "No reviews yet" UI for unclaimed providers.

### 9.7 Geo Taxonomy Drift

If Traffic and VCSM evolve separate geo taxonomies, the city/neighborhood matching for claimed providers may break. Consider a shared geo table or a mapping table.

### 9.8 Cross-App Import Violation

Per `CLAUDE.md` and the boundary contract: apps never depend on each other. Traffic must never import VCSM code. The integration must be data-only (API/database reads), never code-level imports.

## 10. Recommended Next Implementation Phases

### Phase 0 — Provider Schema Extension (Traffic-Only)

Add to `Provider` type:
```js
vcsmActorId: string | null    // VCSM actor UUID, set when claimed
vcsmSlug: string | null        // VCSM vport slug, for deep linking
claimedAt: string | null       // ISO timestamp of claim approval
```

Add to Traffic's database schema (when real DB is created):
```sql
CREATE TABLE traffic.providers (
  ...existing fields...,
  vcsm_actor_id UUID NULL,
  vcsm_slug TEXT NULL,
  claimed_at TIMESTAMPTZ NULL,
  CONSTRAINT uq_vcsm_actor UNIQUE (vcsm_actor_id)
);
```

**Effort:** Low. No VCSM changes needed.

### Phase 1 — VCSM Public Read API (Edge Functions)

Build Supabase Edge Functions that expose public vport data:

```
GET /v1/public/vport/{actorId}/summary
GET /v1/public/vport/{actorId}/services
GET /v1/public/vport/{actorId}/stats
GET /v1/public/vport/by-slug/{slug}
GET /v1/public/vport/directory?type=locksmith&city=san-francisco
```

These functions read from existing VCSM tables using the service role (server-side only) and return clean JSON. They enforce the same RLS-level filtering (active only, public only).

**Effort:** Medium. Requires Edge Function infrastructure.

### Phase 2 — Build-Time Data Pipeline (Traffic)

At `next build` time:
1. For each claimed provider (where `vcsmActorId` is set):
   - Fetch VCSM public summary, services, stats via Phase 1 API
   - Merge into Traffic provider record
   - Override Traffic stats with live VCSM stats (rating, review count)
2. Generate pages with enriched data
3. Add `sameAs` to schema.org structured data pointing to VCSM profile URL

**Effort:** Medium. Requires build-time fetch pipeline.

### Phase 3 — Claim Flow

1. Build VCSM `/claim-profile` route (protected, requires auth)
2. Accept `?provider={trafficSlug}&source=traffic` params
3. Resolve Traffic provider to VCSM vport (by slug match or manual entry)
4. Verification step (email/phone/document)
5. On approval: create or link vport actor, update Traffic provider record via webhook
6. Traffic re-renders with enriched data on next build

**Effort:** High. Requires new VCSM route, verification flow, and webhook system.

### Phase 4 — Real Database Migration (Traffic)

Replace mock data with a real database (Supabase or standalone Postgres):
- Migrate all mock arrays to database tables
- Build seed pipeline for unclaimed listings (web scraping, data partnerships)
- Add admin panel for listing moderation

**Effort:** High. Foundational infrastructure change.

### Phase 5 — Live Data Enrichment (Optional)

For claimed providers, bypass build-time cache and fetch VCSM data at request time:
- ISR with `revalidate` on provider pages
- Webhook from VCSM on vport profile update -> trigger Traffic page revalidation
- Real-time review count and rating updates

**Effort:** Medium. Requires webhook infrastructure.
