# Kraven — 2026-04-12 — Badge Duplicate Read Hunt

## Command Run

Kraven (performance hunter and bottleneck analysis)

## Session Date

2026-04-12

## Problem

`chat.inbox_entries` and `vc.notifications` unread-count reads repeating heavily at runtime. Duplicate query groups observed in dev profiler.

## Investigation

### Caller Chain Traced

```
BottomNavBar.jsx
  ├─ useNotiCount({ actorId })
  │    ├─ getUnreadNotificationCount(actorId)     → vc.notifications COUNT
  │    ├─ subscribeNotificationBadge({ actorId })  → realtime INSERT/UPDATE/DELETE
  │    └─ polling interval: 60s
  │
  └─ useUnreadBadge({ actorId })
       ├─ getInboxUnreadBadgeCount(actorId)        → chat.inbox_entries SUM
       ├─ subscribeInboxBadge({ actorId })          → realtime INSERT/UPDATE/DELETE
       └─ polling interval: 20s
```

### Root Cause Identified

1. **BottomNavBar unmounts/remounts** on route changes to chat sub-screens (`/chat/:id`), learning routes (`/learning/*`), and auth routes
2. `RootLayout.jsx` line 62: `{!hideBottomNav && <BottomNavBar />}` — conditional render causes full unmount
3. Each remount:
   - Creates 6 new realtime subscriptions (3 events × 2 hooks)
   - Fires `fetchCount(true)` with `force=true` (bypasses 15s/10s TTL cache)
   - Starts new polling intervals
4. Rapid navigation creates subscription churn + redundant force queries

### Hooks Involved (only 1 consumer each)

| Hook | Consumer | Cache TTL | Poll | Realtime Events |
|------|----------|-----------|------|-----------------|
| `useNotiCount` | BottomNavBar only | 15s | 60s | 3 (INSERT/UPDATE/DELETE on vc.notifications) |
| `useUnreadBadge` | BottomNavBar only | 10s | 20s | 3 (INSERT/UPDATE/DELETE on chat.inbox_entries) |

### Why It's NOT Multiple Consumers

Both hooks are used in **only** `BottomNavBar.jsx`. The duplication comes from the mount/unmount cycle, not from multiple components.

## Fix Plan

### Slice 1 — Keep BottomNavBar Mounted (CSS hide)

**File:** `apps/VCSM/src/app/layout/RootLayout.jsx`

Change:
```jsx
// BEFORE: unmounts on route change
{!hideBottomNav && <BottomNavBar />}

// AFTER: always mounted, CSS-hidden
<div style={{ display: hideBottomNav ? 'none' : undefined }}>
  <BottomNavBar />
</div>
```

**Impact:** Prevents subscription teardown/recreation. Hooks stay alive, cache stays warm, polling continues uninterrupted. Single subscription pair for entire session.

### Slice 2 — Debounce Realtime Force-Refresh

**Files:**
- `features/notifications/inbox/hooks/useNotiCount.js`
- `features/notifications/inbox/hooks/useUnreadBadge.js`

**Change:** Realtime `onChange` currently calls `fetchCount(true)` (force = bypasses cache). Add 2s debounce so rapid realtime events (e.g., multiple inbox_entries updates) don't each trigger a full DB query.

### Slice 3 — Validate

Check dev profiler for:
- `chat.inbox_entries` query count per navigation cycle
- `vc.notifications` query count per navigation cycle
- Realtime subscription count (should be 2 channels total, not N×2)

## Expected Impact

| Metric | Before | After |
|--------|--------|-------|
| Subscriptions per session | N×6 (per mount cycle) | 6 (once) |
| Force-refresh queries per navigation | 2 (bypass cache each time) | 0 (cache stays warm) |
| Badge query count per minute | ~6-10 (poll + realtime + remount) | ~2-3 (poll only, cache-first) |
