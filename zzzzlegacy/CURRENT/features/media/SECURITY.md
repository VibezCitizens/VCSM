## Findings

### VENOM-F1 ‚Äî RLS Policy Status (RESOLVED)

**Severity at audit:** CRITICAL ‚Äî BLOCKING
**Status: RESOLVED** (confirmed by DB audit `2026-05-19_12-00_db_media-assets-rls-audit.md` per THOR release gate)

At time of VENOM audit, the `media_assets_deny_all` policy in `2026-05-10_secdef_b_zero_policy_tables.sql` was marked PROPOSAL ONLY. DB audit subsequently confirmed RLS is live. THOR gate passed: "VENOM-F1 RESOLVED (RLS confirmed)."

---

### VENOM-F2 ‚Äî `owner_actor_id` Not Session-Verified at Controller Layer (MITIGATED)

**Severity:** CRITICAL
**Status: MITIGATED** ‚Äî DB RLS enforces `owner_actor_id IN (SELECT actor_id FROM vc.actor_owners WHERE user_id = auth.uid())`. Controller-layer gap remains as defense-in-depth gap. THOR gate: "VENOM-F2 MITIGATED at DB."

`createMediaAssetController` accepts `ownerActorId` and `createdByActorId` as caller-supplied parameters with no ownership check ‚Äî only presence validation. DB RLS provides the actual enforcement barrier.

---

### VENOM-F3 ‚Äî Inconsistent App-Layer Validation Across Callers (OPEN)

**Severity:** HIGH (downgraded to LOW per THOR ‚Äî see VENOM-F3)
**Status: OPEN**

Caller ownership validation is inconsistent:

| Caller | Auth Validation |
|---|---|
| `designStudio.assetsExports.controller.js` | YES ‚Äî explicit `requireOwnerActorAccess(ownerActorId)` |
| `useUploadSubmit` (posts) | YES ‚Äî `identity.actorId` from `useIdentity()` |
| `recordChatAttachment.controller.js` | NO ‚Äî accepts `ownerActorId` as raw parameter |
| `flyerEditor.controller.js` | NO ‚Äî assumes caller owns `vportId` |
| `submitCreateVport.controller.js` | PARTIAL ‚Äî flow-level trust |
| `addPortfolioMediaWithRecord.controller.js` | PARTIAL ‚Äî dashboard context implies ownership |
| `cards.controller.js` | PARTIAL ‚Äî `senderActorId` from `ensureGuestUser()` |
| `recordPostMedia.controller.js` | YES ‚Äî identity system |

Primary risk surface: `recordChatAttachment.controller.js` passes `ownerActorId` with no validation. If RLS were not enforced, this would allow media asset registration attributed to any actor. RLS enforcement is the current mitigation.

---

### VENOM-F4 ‚Äî `insertMediaAssetDAL` Has No Auth Check (OPEN)

**Severity:** MEDIUM ‚Äî defense in depth gap
**Status: OPEN**

DAL is a raw Supabase INSERT with no ownership validation. Per architecture contract, business rules belong in the controller ‚Äî DAL is technically correct. However, as a defense-in-depth measure, no second line of defense existed until RLS was confirmed. Now that DB RLS is confirmed, this is a residual depth gap only.

---

### VENOM-F5 ‚Äî Module-Level Cache Risk (ACCEPTABLE)

**Severity:** LOW
**Status: ACCEPTABLE**

`resolveVcsmAppIdDAL` module-level `_cachedAppId`. In a server-side multi-user context this would be a cross-user data leak. In VCSM's Vite client-side architecture, each browser session has its own JS module scope ‚Äî no user data leaks across sessions. Risk accepted.
