## Overview

The platform has **27 views** across **8 schemas**. None are materialized — all are standard SQL views computed at query time. No view caching layer exists at the DB level.

| Schema | View count | Primary purpose |
|---|---|---|
| `vport` | 8 | Public-facing VPORT data for business card, menu, TRAZE, SEO |
| `omd` | 7 | Pin/map feature — stats, votes, profiles |
| `reviews` | 4 | Public review data for VPORTs |
| `notification` | 2 | Inbox read model (full + slim) |

| `vc` | 2 | Actor presentation + user privacy |

## `vc` Schema — Core Actor Views

### `vc.actor_presentation`
**Source tables:** `vc.actors`, `public.profiles`, `vc.vports`

The primary read model for rendering any actor (user or vport) in the UI — display name, avatar, banner, slug. All fields use `COALESCE` to fall back from profile to vport fields. Used heavily in the feed, posts, and anywhere actor identity is displayed.

```
actor_id, kind, display_name, username,
photo_url, banner_url,
vport_name, vport_slug, vport_avatar_url, vport_banner_url,
profile_id, vport_id
```

### `vc.user_profiles`
**Source tables:** `public.profiles`, `vc.actors`, `vc.actor_privacy_settings`

Minimal view that exposes `profile.id` + computed `private` boolean. Used for privacy gating — does not expose profile data, only whether the profile is private.

## `vport` Schema — VPORT Public Views

### `vport.public_provider_details_v`
**Source tables:** `vport.profiles`, `vport.profile_public_details`
**Filter:** `is_active = true`, `is_deleted = false`, `slug IS NOT NULL`

Core public VPORT details — name, bio, avatar, banner, website, email, phone, location, hours, address. Used as the base read model for business card and public profile rendering.

### `vport.public_provider_services_v`
**Source tables:** `vport.services`, `vport.profiles`, `vport.service_booking_profiles`
**Filter:** active profile, active service (`enabled = true`)

Exposes all enabled services for a VPORT including booking metadata — duration, price, bookable flag, booking mode. Used by the public service catalog and booking flow.

### `vport.public_menu_read_model_v`
**Source tables:** `vport.profiles`, `vport.profile_public_details`, `vport.menu_categories`, `vport.menu_items`
**Filter:** active profile, active categories, active items

Flat join of profile + menu structure. Every row is one menu item with its category and profile context. Also embeds the canonical `public_menu_url` as `/profile/${slug}/menu`. This is the authoritative source for the public menu screen (`VportMenuBySlugScreen`).

### `vport.public_categories_v`
**Source tables:** `vport.categories`
**Filter:** `is_active = true`

Simple filtered list of active VPORT categories (key + label).

### `vport.public_content_read_model_v`
**Source tables:** `vport.content_pages`, `vport.profiles`, `vport.profile_public_details`, `vport.profile_categories`, `vport.categories`
**Filter:** `is_published = true`, `is_indexable = true`, active profile

Flat SEO content read model. Embeds the canonical `public_url` as `/profile/${slug}/content/${content_slug}`. Used by TRAZE and any content page renderer.

### `vport.public_actor_seo_v`
**Source tables:** `vc.actors`, `public.profiles`, `vport.profiles`, `vport.profile_public_details`, `vport.profile_categories`, `vport.categories`
**Filter:** `is_void = false`

Broad SEO view joining actor → profile/vport → location → primary category. Used for directory and discovery pages.

### `vport.public_traze_profiles_v`
**Source tables:** `vport.profiles`, `vport.profile_public_details`, `vport.profile_categories`
**Filter:** `is_active = true`, `is_deleted = false`, slug not null/empty, `directory_visible = true`, `directory_status = 'listed'`
**Security:** `security_invoker = true`

The TRAZE-specific provider profile view. Only VPORTs that are directory-listed appear here. Exposes city/country/state extracted from the JSONB `address` field. This is the live data source for TRAZE provider pages once the mock data is replaced.

Key fields: `slug`, `category_key`, `city`, `city_slug`, `state_code`, `country_code`, `city_country_code`, `directory_visible`, `directory_status`

### `vport.public_traze_services_v`
**Source tables:** `vport.services`, `vport.profiles`
**Filter:** active profile, enabled service, slug not null

TRAZE-specific services list. Lighter than `public_provider_services_v` — no booking profile data.

## `reviews` Schema

### `reviews.public_vport_reviews_v`
**Source tables:** `reviews.reviews`
**Filter:** `target_kind = 'vport'`, `active_card = true`, `is_deleted = false`

All visible reviews for VPORTs. Exposes author/target snapshot fields (display_name, username, avatar_url taken at time of review) so the view is safe to read without joining back to profiles.

### `reviews.public_vport_review_summary_v`
**Source tables:** `reviews.reviews`
**Filter:** same as above + `overall_rating IS NOT NULL`

Aggregated per `target_actor_id` — review count, average rating, first/last activity timestamps. The business card reads from this view for the star rating display.

### `reviews.public_vport_review_dimensions_v`
**Source tables:** `reviews.review_dimension_ratings`, `reviews.reviews`, `reviews.review_dimensions`
**Filter:** active card, not deleted, active dimension

Per-dimension aggregates — `rating_count` and `average_rating` grouped by `(target_actor_id, dimension_id)`.

### `reviews.public_vport_review_dimension_ratings_v`
**Source tables:** `reviews.review_dimension_ratings`, `reviews.reviews`
**Filter:** same as above

Raw per-review dimension ratings (not aggregated). Used when you need individual rating rows rather than averages.

## `notification` Schema

### `notification.inbox_view`
**Source tables:** `notification.recipients`, `notification.events`, `notification.inbox_items`

Slim inbox view. Joins delivery recipient + event + inbox item (is_seen, seen_at, is_read, read_at). Used for the notification bell and inbox list.

### `notification.inbox_full_view`
**Source tables:** `notification.recipients`, `notification.events`, `notification.rendered_deliveries`

Full inbox view. Adds rendered delivery metadata — `template_id`, `locale`, `title`, `body`, `actions_json`. Used when full notification content needs to be displayed.

## `identity` Schema

### `identity.actor_directory_view`
**Source tables:** `identity.actor_directory`
**Filter:** `is_hydratable = true`, `is_active = true`, `is_void = false`

All hydratable active actors. Broader than `public_actor_directory` — includes non-discoverable actors. Used for internal lookup and hydration pipelines.

### `identity.public_actor_directory`
**Source tables:** `identity.actor_directory`
**Filter:** `actor_domain = 'vc'`, hydratable, active, not void, `is_listable_in_app = true`, `discoverable = true`, `publish = true`, `is_private = false`

Strict public-facing actor directory. Only actors that are fully published, discoverable, and not private appear here. Used for explore/search surfaces.

## `omd` Schema — Map/Pin Feature

All `omd` views use `security_invoker = true` or `security_invoker = 'on'` — they execute with the permissions of the calling user, not the view owner.

| View | Purpose |
|---|---|
| `omd.active_pins` | Public + own pins. Filter: `is_private = false OR user_id = auth.uid()` |
| `omd.pin_view_counts` | Aggregated view counts per pin |
| `omd.pin_vote_counts` | Up/down vote counts per pin |
| `omd.pin_stats` | Combined stats per pin (up, down, unique views excluding owner) |
| `omd.my_pins_list` | All pins for current user with stats joined |
| `omd.pins_with_profile` | Public pins with author display_name/username/photo_url from `public.profiles` |
| `omd.top_posters` | Ranked list of users by pin count with profile data |

## `platform` Schema

### `platform.current_user_consents`
**Source tables:** `platform.user_consents`

`DISTINCT ON (user_id, app_id, consent_type)` ordered by `accepted_at DESC` — always returns the most recent consent record per user/app/type combination. Used for consent verification gates.

### `platform.public_legal_documents_v`
**Source tables:** `platform.legal_documents`, `platform.apps`
**Filter:** `app.is_active = true`, `legal_document.is_active = true`

Active legal documents with their app key. Used for the public legal document display screen.

### `platform.v_user_app_context`
**Source tables:** `platform.user_app_accounts`, `platform.apps`

Joins app account rows with their app metadata (key, name). Used to resolve app context for a user session.

## Key Rules When Working with Views

1. **No `select('*')` on views** — same rule as tables. Always specify columns.
2. **Views are not RLS-aware by default** — unless the view has `security_invoker = true`, it runs as the view owner (typically `postgres`/`service_role`). The `omd` views and `vport.public_traze_profiles_v` explicitly opt into security_invoker.
3. **Canonical URL fields are baked into views** — `public_menu_read_model_v` embeds `/profile/${slug}/menu` and `public_content_read_model_v` embeds the content URL. If the route pattern ever changes, these view definitions must be updated.
4. **TRAZE data source** — `vport.public_traze_profiles_v` is the live DB source for TRAZE provider data once mock data is replaced. It filters on `directory_visible = true` and `directory_status = 'listed'` — VPORTs must meet both to appear in TRAZE.
5. **No materialized views** — all views are computed at query time. High-traffic views (like `vc.actor_presentation`) benefit from upstream table indexes, not view-level caching.
