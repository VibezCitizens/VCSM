### Slices

1. **Foundation** ‚Äî Create `platform/services/` root, `index.js`, define service conventions and export boundaries
2. **identityService** ‚Äî Wrap identity/platform context loading behind a stable service API. Expose app/account/actor context loaders without modifying app consumers yet.
3. **actorService** ‚Äî Establish canonical actor bundle service API. Reuse existing proven loaders internally where appropriate. Prepare future migration away from scattered actor/profile/privacy reads.
4. **Service stubs** ‚Äî Create `notificationService` and `chatService` stubs with documented API shapes. No premature implementation beyond foundation and signatures.

### Assumptions

- `platform/` is a new platform-level root
- Services consume engines only through public APIs or stable shared loaders
- No database changes needed
- No app consumers modified in this slice
- `actorService` establishes canonical API shape now; app migration comes later

### Blocking Risks

- Over-abstraction risk if service API doesn't match real consumer needs
- Risk of mixing app-specific logic into platform services
- Risk of hidden dependency on VCSM-only actor shapes if not carefully normalized

### Architecture Rationale

- `identityService` first: identity/platform bootstrap is the largest duplicated orchestration point
- `actorService` second: actor/profile/privacy loading is fragmented across multiple paths
- `notificationService` + `chatService` stubs: define future platform boundaries without forcing premature migration

## Execution Summary

### What Was Completed

All 4 slices executed successfully. The Platform Service Layer foundation is built.

**Slice 1 ‚Äî Foundation:**
Created `platform/services/` root with `index.js` barrel export. Defined export conventions: services never export DAL, never import from apps, consume engines through public APIs only. Aliased overlapping names (`getInbox` ‚Üí `getNotificationInbox` / `getChatInbox`).

**Slice 2 ‚Äî identityService:**
Implemented 5 functions wrapping the identity engine and hydration engine:
- `getPlatformContext()` ‚Äî full session ‚Üí app ‚Üí account ‚Üí actor resolution
- `getActiveActor()` ‚Äî convenience active actor extraction
- `getActorLinks()` ‚Äî all available actor links for current user
- `switchActor()` ‚Äî delegate to identity engine's switchActiveActor
- `getUserAppState()` ‚Äî context + hydrated actor summaries (warm cache)

All functions grounded in real code paths from `identity.controller.js` and the identity/hydration engine APIs.

**Slice 3 ‚Äî actorService:**
Implemented 7 functions wrapping the hydration engine store and fetch pipeline:
- `getActorBundle()` ‚Äî single actor with cache-aware fetch
- `getActorBundles()` ‚Äî batch fetch, only stale/missing actors hit network
- `getActorBundleMap()` ‚Äî keyed map convenience wrapper
- `getActorSummaries()` ‚Äî raw rows + hydrate (for engine DI callbacks)
- `getActorFromStore()` ‚Äî synchronous store read (no fetch)
- `isActorStale()` ‚Äî freshness check
- `invalidateActors()` ‚Äî force-stale specific actors for cache busting

**Slice 4 ‚Äî Service stubs:**
Created `notificationService.js` (5 stubs) and `chatService.js` (5 stubs) with documented API shapes. All stubs throw explicit `[PSL] not yet implemented` errors. No premature wiring.
