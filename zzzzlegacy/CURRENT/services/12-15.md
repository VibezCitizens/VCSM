## Execution Summary

### What Was Completed

All 4 slices executed successfully. VCSM notifications read path now uses the `notification.*` schema via the notification engine.

**Slice 1 — Engine setup:**
Created `apps/VCSM/src/features/notifications/setup.js` following the established DI pattern (same as chat/identity/reviews/portfolio). Injects `supabaseClient` and `resolveActorCard` (via hydration engine). No `resolveRecipients` injected yet — write path remains legacy.

**Slice 2 — Bootstrap wiring:**
Added `setupVcsmNotificationsEngine()` to `apps/VCSM/src/main.jsx` after existing engine setups.

**Slice 3 — Inbox read migration:**
- Updated `Notifications.controller.js` to call `getInboxNotifications()` from the engine instead of legacy `fetchNotificationsPage()` DAL
- Engine handles `autoMarkSeen` internally — removed manual mark-seen logic from controller
- Updated `notification.mapper.js` to map engine's `InboxNotification` shape (`recipientId`, `eventKey`, `sourceActorId`, `payload`, `title`, `body`) instead of legacy `vc.notifications` flat row
- Updated `blockFilter.js` to accept configurable `getActorId` extractor for engine shape compatibility
- Updated `extractRequesterActorId` to read from `payload` (engine) instead of `context` (legacy)
- Follow-request filter now checks `eventKey` instead of `kind`

**Slice 4 — Badge migration:**
- Updated `inboxUnread.controller.js` to call engine `countUnread({ recipientActorId })` instead of legacy `fetchInboxUnreadCounts` DAL
- Updated `notificationsCount.controller.js` to call engine `countUnread({ recipientActorId })` instead of legacy `countUnreadNotifications` DAL
- Both controllers wrapped in try/catch returning 0 on failure for resilience
