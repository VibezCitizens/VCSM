# join / modules / join — SECURITY

## THOR Status

NO THOR BLOCKERS. Highest severity: MEDIUM.

## Findings

### JOIN-SEC-001 — Pre-Auth Token Oracle
| Field | Value |
|---|---|
| ID | JOIN-SEC-001 |
| Source Findings | VEN-JOIN-001 |
| Severity | MEDIUM |
| Surface | barberVport.read.dal.js — unauthenticated resource fetch |
| Description | The resource lookup performed before authentication implicitly validates the token. An unauthenticated caller can probe whether a token is valid by observing the response (resource found vs not found). This is a validity oracle — not a direct exploit, but leaks token existence to unauthenticated actors. |
| Status | OPEN |
| THOR | Not blocked |

### JOIN-SEC-002 — Side-Effects Before Resource State Verification
| Field | Value |
|---|---|
| ID | JOIN-SEC-002 |
| Source Findings | VEN-JOIN-002 |
| Severity | MEDIUM |
| Surface | joinBarbershopAccount.controller.js → autoResumeInviteOnboarding |
| Description | autoResumeInviteOnboarding creates side-effects (VPORT creation or modification) before verifying the invite resource state. If the resource has already been claimed or is in a terminal state, side-effects have already occurred and cannot be easily rolled back. |
| Status | OPEN |
| THOR | Not blocked |

### JOIN-SEC-003 — Client-Controlled Metadata in Auth JWT
| Field | Value |
|---|---|
| ID | JOIN-SEC-003 |
| Source Findings | VEN-JOIN-003 |
| Severity | MEDIUM |
| Surface | dal/joinInvite.dal.js → signUpForInviteDAL → supabase.auth.signUp metadata |
| Description | Client embeds pending_invite_token, vport_name, and category_key directly into the Supabase auth signup metadata. These fields are embedded in the JWT and used to drive auto-resume onboarding. No server-side length or character validation confirmed. Malformed or injected values could corrupt the onboarding state machine. |
| Status | OPEN — UNVERIFIED (server validation not confirmed) |
| THOR | Not blocked |

### JOIN-SEC-004 — Invite Path Missing Controller-Layer State Pre-Check
| Field | Value |
|---|---|
| ID | JOIN-SEC-004 |
| Source Findings | BW-JOIN-003 |
| Severity | MEDIUM |
| Surface | joinBarbershopAccount.controller.js — createBarberVportAndAccept, useExistingBarberVportAndAccept, autoResumeInviteOnboarding |
| Description | Invite-path controllers do not have a controller-layer resource state pre-check before calling acceptJoinResourceDAL. Only the QR path has ELEK-001 defense-in-depth (controller + DAL double guard). Invite path relies solely on the DAL-layer conditional UPDATE. If a bug bypasses the DAL guard, there is no controller backstop on the invite path. |
| Status | OPEN — PARTIAL (DAL guard applies) |
| THOR | Not blocked |

### JOIN-SEC-005 — acceptJoinResourceDAL No SQL Ownership Filter
| Field | Value |
|---|---|
| ID | JOIN-SEC-005 |
| Source Findings | BW-JOIN-001 |
| Severity | MEDIUM |
| Surface | dal/joinInvite.dal.js → acceptJoinResourceDAL |
| Description | The UPDATE SQL does not include a caller-identity ownership filter. Ownership is enforced only at the controller layer via assertActorOwnsVportActorController. If RLS on vport.resources is absent or misconfigured AND the controller gate is bypassed, the write proceeds without identity verification at the DB layer. |
| Status | OPEN — RLS on vport.resources unverified |
| THOR | Not blocked |
