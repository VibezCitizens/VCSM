## Purpose

This report maps every function that accepts caller-supplied identity parameters:
`*ActorId | *UserId | *ProfileId | *VportId | *OwnerId | *RealmId`

An "injection surface" is any function where an external caller can supply an identity value
that is used to scope data access, authorize mutations, or determine ownership.

This is a discovery document — it does not assess whether surfaces are secured or insecure.
Security assessment is the responsibility of VENOM / ELEKTRA / BLACKWIDOW.

---

## Surface Classification

### Tier 1 — High Authority
Functions that accept identity parameters and use them to authorize mutations.
These are the most sensitive surfaces — a wrongly-trusted parameter = privilege escalation.

### Tier 2 — Read-Scoped
Functions that accept identity parameters and use them to filter read queries.
Lower risk — typically returns empty rather than unauthorized data.

### Tier 3 — Propagation
Functions that accept identity parameters and pass them downstream unchanged.
Risk depends on the eventual consumer.

### Tier 4 — Component/UI
Functions that accept identity parameters as props for display only.
Lowest risk — no database operations.

## Special Surfaces: realmId

The `realmId` parameter (76 occurrences) is a trust-critical surface because it controls
which content realm a post/feed/chat is scoped to. Supplying the wrong realmId can route
content into the wrong realm (public vs void).

**Functions accepting realmId as a mutation parameter:**
| Feature | Function | Impact |
|---|---|---|
| post | createPostController | Post routed to wrong realm |
| feed | fetchFeedPagePipeline | Wrong realm's content returned |
| chat | resolveActorRealmContext | Conversation misrouted |
| wanders | useWandersCreateCardExperience | Card created in wrong realm |

**Existing safeguard:** The VCSM contracts specify that system posts must always use
`resolvePublicRealmIdDAL()` — never a caller-supplied realmId.

---

## Special Surfaces: userId vs actorId Boundary

`userId` (72 uses) is the raw Supabase auth UID. It appears in:
- `auth` feature — user creation / session bootstrap
- `legal` feature — consent recording
- `state` feature — actor resolution from userId
- `identity` feature — platform bootstrap

The `userId → actorId` resolution is handled at the `state` and `identity` feature boundaries.
Above those layers, all identity should be expressed as `actorId`. Any controller outside
`auth`, `legal`, `state`, or `identity` that accepts `userId` directly is a potential
architecture violation.

**Currently clean:** No production controller outside those 4 features accepts a raw `userId`.
