## Notable Parameter Patterns

### Pattern: Dual-Actor (Caller + Target)
Functions accepting both a caller identity and a target identity. High-trust boundary.

```
callerActorId + vportActorId     — settings controllers
requestActorId + targetActorId   — booking ownership controller
myActorId + targetActorId        — block hooks/controllers
reporterActorId + targetActorId  — moderation controllers
viewerActorId + actorId          — feed/profile read controllers
followerActorId + followedActorId — social DAL
```

### Pattern: User → Actor Bridge
Functions that receive a raw `userId` (Supabase auth UID) and resolve to `actorId`.

```
userId → actorId  — auth, state, identity features
userId → userAppAccountId → legal consent flow
```

### Pattern: actorId + realmId
Feed and post operations scope content to a specific realm:

```
actorId + realmId  — post creation, feed pages, chat context
realmId           — wanders card creation (critical for void realm routing)
```

### Pattern: actorId + courseId
Pervasive through the learning feature:

```
actorId + courseId  — enrollment, progress, lesson completion
```

### Pattern: ownerActorId vs actorId
Two distinct ownership semantics exist:
- `actorId` — the acting identity (who is making the call)
- `ownerActorId` — the ownership identity (who owns the resource)

In most features these coincide. In media/flyer, they are passed separately:
```
createMediaAssetController({ ownerActorId, createdByActorId, ... })
```

### Parameter `supabase` (116 uses)
The Supabase client object is passed as a parameter to 116 functions, primarily in DAL and service layers. This is an injection pattern — the client is provided by the caller. Used in test environments to inject mock clients.

### Parameter `shared` (98 uses)
Used exclusively in diagnostics/dev group — `runDiagnosticsTests` passes a `shared` context object across test groups. Not a production identity surface.
