Title:
Device Trust & Session Intelligence Architecture

Goal:
Design and implement a complete device/session intelligence system for VCSM that supports:

- Device recognition
- New device login detection
- Login notifications
- Session visibility
- Session revocation
- Sign Out Everywhere
- Security auditing
- Future suspicious-login detection

This is NOT a browser fingerprinting system.

This is a user-security and session-management system.

--------------------------------------------------
PROBLEM
--------------------------------------------------

Today VCSM knows:

- User authenticated
- Session exists
- Refresh token exists

But VCSM does NOT know:

- Which device created the session
- Whether device is trusted
- How many devices are active
- Which device was last used
- Whether a login is new
- Whether a device should be revoked

As a result:

- Sign Out Everywhere has no visibility layer
- Login notifications cannot be generated
- Security dashboard cannot exist
- Compromise investigations are difficult

--------------------------------------------------
DESIGN PRINCIPLE
--------------------------------------------------

Track sessions.

Do NOT attempt invasive browser fingerprinting.

Use stable identifiers generated and controlled by VCSM.

The system should survive:

- Refreshes
- Browser restarts
- Token rotation

while respecting privacy.

--------------------------------------------------
PROPOSED MODEL
--------------------------------------------------

Layer 1:
Device Identity

Layer 2:
Session Identity

Layer 3:
Login Events

Layer 4:
Security Actions

--------------------------------------------------
LAYER 1 — DEVICE IDENTITY
--------------------------------------------------

Generate a local device identifier:

vcsm_device_id

Stored locally.

Example:

device_7f2d5a....

Purpose:

Represents a trusted browser installation.

Not tied to auth.

Not tied to cookies.

Not tied to Supabase.

Survives logout.

A device may have:

- many sessions
- many logins
- many users over time

--------------------------------------------------
LAYER 2 — SESSION IDENTITY
--------------------------------------------------

Every successful login creates:

session_id

Represents:

User + Device + Login Event

Tracks:

- created_at
- last_seen_at
- revoked_at
- device_id
- user_id

A user can have:

MacBook Firefox
MacBook Chrome
iPhone Safari

Each becomes its own session.

--------------------------------------------------
LAYER 3 — LOGIN EVENTS
--------------------------------------------------

Store every authentication event.

Examples:

LOGIN_SUCCESS
LOGIN_FAILED
PASSWORD_RESET
SIGN_OUT
SIGN_OUT_EVERYWHERE
SESSION_REVOKED

Purpose:

Security audit trail.

Allows:

- timeline reconstruction
- compromise investigation
- anomaly detection

--------------------------------------------------
LAYER 4 — DEVICE TRUST
--------------------------------------------------

Track:

trusted
untrusted
revoked

Examples:

trusted:
User's regular laptop

untrusted:
New browser seen for first time

revoked:
User explicitly removed device

--------------------------------------------------
NEW DEVICE DETECTION
--------------------------------------------------

Login Flow:

1. User signs in
2. Device ID submitted
3. Lookup existing trusted device

IF FOUND:

IF NOT FOUND:

create device

trigger notification
