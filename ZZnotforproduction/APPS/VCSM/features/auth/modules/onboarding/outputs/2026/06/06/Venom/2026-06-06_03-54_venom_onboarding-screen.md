- **Location:** apps/VCSM/src/features/auth/controllers/onboarding.controller.js:13
- **Application Scope:** VCSM (apps/VCSM)
- **Platform Surface:** PWA · Supabase Table/View
- **Trust Boundary:** Authenticated Citizen (newly registered)
- **Boundary Violated:** Feature module isolation contract — auth controller imports initiation DAL directly
- **Contract Violated:** Boundary Isolation Contract
- **Current behavior:** `onboarding.controller.js` imports `acceptVibeInviteByCodeDAL` from `@/features/initiation/dal/vibeInvites.dal` at line 13. This is a direct cross-feature DAL import. The `initiation` feature's adapter (`onboarding.adapter.js`) exports only `OnboardingCardsView` and `CitizenVibesScreen` — no DAL proxy exists.
- **Risk:** Two risk dimensions: (1) Architectural — any change to `initiation/dal/vibeInvites.dal` signature or location silently breaks the auth onboarding path. Since the dependency bypasses the adapter, it is invisible to refactoring tooling. (2) Security surface expansion — the auth controller is directly coupled to a third-party DAL write surface without an independent ownership check for the invite write. The invite attribution fires post-actor-creation with the session-verified `actor.id`, but the DAL's internal ownership logic is opaque from this feature's perspective.
- **Severity:** MEDIUM
- **Exploitability:** LOW — invite attribution is best-effort (.catch()); exploit of this specific path requires access to the initiation DAL internals; does not allow forging writes under another identity because `actor.id` is RPC-generated and session-verified.
- **Attack Preconditions:** Authenticated Citizen; would require DAL-level manipulation not available from the client.
- **Blast Radius:** Single actor — invite attribution only; does not affect profile or actor creation path
- **Identity Leak Type:** None — actor.id is generated by the system, not user-supplied
- **Cache Trust Type:** None
- **RLS Dependency:** UNVERIFIED — vibe_invites table RLS status unknown
- **Why it matters:** Cross-feature DAL imports are banned by the architecture contract. They create hidden dependencies that weaken the security boundary review coverage. If `vibeInvites.dal` is modified to accept broader inputs or relax ownership checks, this silently propagates into the auth onboarding path. Future VENOM/ELEKTRA runs on the initiation feature will not catch the coupling.
- **Recommended mitigation:** Add `acceptVibeInviteByCode` as an exported function in `initiation/adapters/onboarding.adapter.js` and update the import in `onboarding.controller.js` to use the adapter. Alternatively, move the invite attribution to the identity platform bootstrap layer (`ensureVcsmPlatformBootstrap`) which already handles post-actor-creation setup.
- **Rationale:** Adapter boundaries exist to make security surface enumeration tractable. Every cross-feature DAL import adds an unmonitored write surface to the consuming feature.
- **Follow-up command:** WOLVERINE (boundary fix); ELEKTRA (patch advisory)
- **CISSP Domain:**

  - Secondary: Software Development Security

### VEN-ONBOARD-002 — ensureProfileShell: No Session Ownership Cross-Check

**VENOM SECURITY FINDING**
- **Finding ID:** VEN-ONBOARD-002
- **Location:** apps/VCSM/src/features/auth/controllers/profileOnboarding.controller.js (ensureProfileShell function); apps/VCSM/src/features/auth/dal/onboarding.dal.js (upsertProfileShellDAL)
- **Application Scope:** VCSM (apps/VCSM)
- **Platform Surface:** PWA · Supabase Table/View
- **Trust Boundary:** Authenticated Citizen (registering)
- **Boundary Violated:** Actor Ownership Contract — profile write must be scoped to authenticated user's own profile
- **Contract Violated:** Actor Ownership Contract
- **Current behavior:** `ensureProfileShell({ userId, email })` accepts `userId` entirely from the caller with no internal `supabase.auth.getUser()` call. `upsertProfileShellDAL` executes:
  ```js
  supabase.from('profiles').upsert({ id: userId, email, created_at, updated_at })
  ```
  No `.eq('id', auth.uid())` filter exists. The only ownership guarantee is that callers pass the correct authenticated `userId` — a convention, not an enforced constraint.
- **Risk:** If any calling code path passes a different `userId` than the authenticated session's `user.id`, a profile row will be created/overwritten for that target user. The attack surface is limited to callers of `ensureProfileShell` in the register/callback flow — but the latent risk is that a future caller (or refactor) could supply a different userId without breaking the type contract.
- **Severity:** MEDIUM
- **Exploitability:** MEDIUM — requires an attacker to control the `userId` value passed into `ensureProfileShell`. In the current call graph, userId comes from the register controller's session — but the absence of a session cross-check makes this fragile. Combined with absent RLS (see VEN-ONBOARD-003), this path is exploitable if the caller ever breaks the convention.
- **Attack Preconditions:** Authenticated user account; ability to influence the `userId` passed into `ensureProfileShell` through a refactor or new caller; profiles RLS absent.
- **Blast Radius:** Single actor — can only affect one profile per invocation; but if exploited, overwrites target user's profile shell (email, timestamps)
- **Identity Leak Type:** Ownership inference — caller-controlled userId implies ownership assumption
- **Cache Trust Type:** None
- **RLS Dependency:** REQUIRED — DB-layer ownership enforcement (`auth.uid() = id`) is the only backstop if caller convention breaks; currently UNVERIFIED
- **Why it matters:** The VCSM architecture contract requires all write paths to enforce session ownership independently — not by convention. `ensureProfileShell` is a write surface that any code path in the register/callback pipeline can call. A single refactor that passes the wrong `userId` would silently create or overwrite a foreign profile with no error and no DB rejection (if RLS is absent).
- **Recommended mitigation:** Add session cross-check inside `ensureProfileShell`: `const { data: { user } } = await supabase.auth.getUser(); if (!user || user.id !== userId) throw new Error('Session mismatch')`. Also add `.eq('id', userId)` filter to `upsertProfileShellDAL` as defense-in-depth. Request DB audit of profiles table INSERT WITH CHECK policy (see VEN-ONBOARD-003).
- **Rationale:** Ownership must be enforced at the controller layer, not assumed by caller convention. This is the same security principle that the `completeOnboardingController` correctly implements (session pin at line 72) — `ensureProfileShell` should match this pattern.
- **Follow-up command:** ELEKTRA (patch advisory); CARNAGE (profiles RLS audit)
- **CISSP Domain:**
  - Primary: Identity and Access Management
  - Secondary: Security Architecture and Engineering, Software Development Security

### VEN-ONBOARD-003 — profiles Table RLS UNVERIFIED for INSERT/UPSERT

**VENOM SECURITY FINDING**
- **Finding ID:** VEN-ONBOARD-003
- **Location:** apps/VCSM/src/features/auth/dal/onboarding.dal.js (upsertProfileShellDAL, upsertCompletedOnboardingProfileDAL)
- **Application Scope:** VCSM (apps/VCSM)
- **Platform Surface:** PWA · Supabase Table/View
- **Trust Boundary:** Authenticated Citizen
- **Boundary Violated:** Actor Ownership Contract (database layer)
- **Contract Violated:** Actor Ownership Contract
- **Current behavior:** All profile upsert operations in the onboarding module write to `public.profiles` via the Supabase authenticated client. Neither DAL function includes a `.eq('id', auth.uid())` client-side filter. The write is passed to Supabase with `id: profileId` (or `id: userId`) as the upsert conflict key. RLS on the `public.profiles` table is assumed to enforce `auth.uid() = id` for INSERT/UPDATE but has not been verified in any governance artifact.
- **Risk:** If `public.profiles` has no INSERT WITH CHECK policy enforcing `auth.uid() = id`, any authenticated user could upsert any profile row by controlling the `id` field. Combined with VEN-ONBOARD-002 (ensureProfileShell caller-controlled userId), this is the most direct path to a foreign profile write.
- **Severity:** MEDIUM (becomes HIGH if confirmed absent in DB audit)
- **Exploitability:** MEDIUM — requires confirmed RLS absence to exploit; if RLS is present, the DB rejects the unauthorized upsert cleanly.
- **Attack Preconditions:** Authenticated user; profiles table INSERT/UPDATE RLS absent or misconfigured; caller-controlled userId into ensureProfileShell.
- **Blast Radius:** Single actor per invocation — targeted profile overwrite; could affect display_name, username, visibility flags (publish, discoverable)
- **Identity Leak Type:** None directly; but a successful exploit writes false identity data for a target profile
- **Cache Trust Type:** Public-profile-sensitive — publish=true and discoverable=true are set here; a forged write makes a victim profile discoverable
- **RLS Dependency:** REQUIRED — this is the critical DB-layer backstop for all profile upsert paths
- **Why it matters:** The `public.profiles` table is the identity foundation of the platform. Username, display name, and discoverability are all written here. An absent or misconfigured RLS policy is the single point of failure that converts VEN-ONBOARD-002 from a latent risk to an active exploit. This table has been flagged in prior runs (VEN-REG-002, ELEK-REG-003) and has not yet been audited.
- **Recommended mitigation:** DB audit of `public.profiles` — confirm: (1) INSERT WITH CHECK (auth.uid() = id) policy exists; (2) UPDATE USING (auth.uid() = id) policy exists; (3) upsert is correctly blocked by RLS when id ≠ auth.uid(). Delegate to CARNAGE or DB command.
- **Rationale:** Every upsert write path in the onboarding module relies on this table. Without confirmed RLS, the defense-in-depth model has a missing layer. VEN-REG-002 and ELEK-REG-003 have both flagged this and it remains OPEN — this run re-escalates.
- **Follow-up command:** CARNAGE (schema review); DB (RLS policy inspection)
- **CISSP Domain:**
  - Primary: Security Architecture and Engineering
  - Secondary: Identity and Access Management, Security Assessment and Testing

### VEN-ONBOARD-004 — No Minimum Age Enforcement

**VENOM SECURITY FINDING**
- **Finding ID:** VEN-ONBOARD-004
- **Location:** apps/VCSM/src/features/auth/controllers/onboarding.controller.js (completeOnboardingController:97-105); apps/VCSM/src/features/auth/model/onboarding.model.js (computeAgeFromBirthdateModel)
- **Application Scope:** VCSM (apps/VCSM)
- **Platform Surface:** PWA · Supabase Table/View
- **Trust Boundary:** Authenticated Citizen (newly registered)
- **Boundary Violated:** None explicit in current contracts — this is a gap in the business rules
- **Contract Violated:** Security and Risk Management (COPPA / age-gate compliance)
- **Current behavior:** `computeAgeFromBirthdateModel` returns an integer age (minimum: 0 for someone born today). The controller uses this for `isAdult: age >= 18` computation. No minimum age check exists:
  ```js
  const age = computeAgeFromBirthdateModel(normalized.birthdate)
  if (age == null) {
    return { ok: false, error: { message: 'Invalid birthdate.' }, ... }
  }
  // No: if (age < 13) return { ok: false, error: ... }
  ```
  A user can complete onboarding with birthdate = yesterday (age = 0) and gain full platform access with `is_adult: false`.
- **Risk:** Platform allows users of any age to complete onboarding and gain access. If the platform has age-based content restrictions (e.g., COPPA 13+ requirement for US users, 18+ for Void Realm content), the absence of a minimum age gate means underage users can self-report any age. The `is_adult` flag correctly gating 18+ content is good — but there is no floor on age, and a 5-year-old's birthdate is accepted as valid.
- **Severity:** MEDIUM
- **Exploitability:** HIGH — any authenticated user can enter any past birthdate; no verification mechanism exists
- **Attack Preconditions:** Valid authenticated account; ability to enter a false birthdate (the field is a date picker with max=today only — no minimum date constraint)
- **Blast Radius:** Multi-actor — any user can misrepresent age; affects is_adult flag computation platform-wide
- **Identity Leak Type:** None
- **Cache Trust Type:** None
- **RLS Dependency:** NONE — this is a business logic enforcement gap, not a DB policy gap
- **Why it matters:** The `is_adult` field is used to gate the Void Realm (18+ content per platform memory). Without a minimum age enforcement, two problems exist: (1) users can set `is_adult: true` by entering a birth year making them exactly 18 when they are actually younger; (2) there is no platform-wide minimum age requirement enforced (COPPA compliance for US users requires 13+). The birthdate field has a UI max=today constraint but no minimum constraint — a user born in 1900 is accepted, and a user who enters today's date as birthdate gets age=0 with no rejection.
- **Recommended mitigation:** Add minimum age check in `completeOnboardingController` after `computeAgeFromBirthdateModel`: if `age < MIN_AGE` (e.g., 13 or 18, depending on platform policy) return `{ ok: false, error: { message: 'You must be at least [N] years old to join.' } }`. Also add `min` attribute to the birthdate input in `Onboarding.jsx`. Determine the platform's age policy (13+? 18+?) and document it as a business rule in BEHAVIOR.md.
- **Rationale:** Age gating belongs at the controller layer (server-side enforcement), not in the UI only. The current architecture correctly computes `is_adult` but never enforces a floor. This is a compliance risk if the platform operates in jurisdictions with child protection laws.
- **Follow-up command:** WOLVERINE (policy decision required); IRONMAN (ownership of age-gate business rule)
- **CISSP Domain:**
  - Primary: Security and Risk Management
  - Secondary: Identity and Access Management

### VEN-ONBOARD-005 — Sex Field Not Validated at Controller Layer After Normalization

**VENOM SECURITY FINDING**
- **Finding ID:** VEN-ONBOARD-005
- **Location:** apps/VCSM/src/features/auth/controllers/onboarding.controller.js:82-90; apps/VCSM/src/features/auth/model/onboarding.model.js (normalizeSexValueModel)
- **Application Scope:** VCSM (apps/VCSM)
- **Platform Surface:** PWA · Supabase Table/View
- **Trust Boundary:** Authenticated Citizen
- **Boundary Violated:** None — internal validation gap
- **Contract Violated:** None explicit
- **Current behavior:** `normalizeOnboardingFormModel` normalizes the sex field via `normalizeSexValueModel`:
  ```js
  export function normalizeSexValueModel(value) {
    const raw = String(value ?? '').trim().toLowerCase()
    if (!raw) return null
    return SEX_VALUES[raw] ?? null  // SEX_VALUES = { male: 'Male', female: 'Female' }
  }
  ```
  Any value not in `{ male, female }` normalizes to `null`. The controller validates `displayName`, `usernameBase`, and `birthdate` for presence but does NOT validate `sex`:
  ```js
  if (!normalized.displayName || !normalized.usernameBase || !normalized.birthdate) {
    return { ok: false, error: { message: 'Please complete all required fields.' }, ... }
  }
  // sex is not checked — proceeds with sex: null if unexpected value
  ```
  Result: `upsertCompletedOnboardingProfileDAL` is called with `sex: null` if an unexpected sex value is submitted.
- **Risk:** A user who bypasses the client-side select (e.g., via direct API call or browser manipulation) can submit an arbitrary sex value that normalizes to null, completing onboarding with a null sex despite the UI requiring it. This creates a profile that is "incomplete" by the model's definition (`isProfileShellIncompleteModel` checks sex ≠ ''), potentially causing UI inconsistencies or re-triggering onboarding gates.
- **Severity:** LOW
- **Exploitability:** LOW — requires bypassing the client-side select constraint; the hook also enforces `form.sex.trim() !== ''` before calling the controller. Direct API-level exploitation is possible but has limited impact.
- **Attack Preconditions:** Authenticated user; ability to bypass UI select constraint (browser DevTools or direct Supabase client call)
- **Blast Radius:** Single actor — self-targeting only; no cross-actor impact
- **Identity Leak Type:** None
- **Cache Trust Type:** None
- **RLS Dependency:** NONE — this is an application validation gap
- **Why it matters:** The controller should validate all required fields independently of the UI. Defense-in-depth requires the controller to reject null sex after normalization, matching the behavior declared in the module BEHAVIOR.md (which states sex is required).
- **Recommended mitigation:** Add sex validation in `completeOnboardingController` after normalization:
  ```js
  if (!normalized.displayName || !normalized.usernameBase || !normalized.birthdate || !normalized.sex) {
    return { ok: false, error: { message: 'Please complete all required fields.' }, ... }
  }
  ```
- **Rationale:** Every required field should be validated at the controller layer, not only in the hook or UI. This is a one-line fix.
- **Follow-up command:** ELEKTRA (patch advisory — trivial one-line fix)
- **CISSP Domain:**
  - Primary: Software Development Security
  - Secondary: Security and Risk Management

### VEN-ONBOARD-006 — generate_username RPC: No Rate Limiting or Enumeration Protection Visible

**VENOM SECURITY FINDING**
- **Finding ID:** VEN-ONBOARD-006
- **Location:** apps/VCSM/src/features/auth/dal/onboarding.dal.js (generateUsernameDAL)
- **Application Scope:** VCSM (apps/VCSM)
- **Platform Surface:** PWA · Supabase RPC
- **Trust Boundary:** Authenticated Citizen
- **Boundary Violated:** None confirmed — audit gap
- **Contract Violated:** None explicit
- **Current behavior:** `generateUsernameDAL` calls `supabase.rpc('generate_username', { _display_name, _username })` with arbitrary display name and username base. The RPC presumably reads existing usernames to generate a unique sanitized variant. No rate limiting or enumeration protection is visible at the app layer (controller calls this once per onboarding save). The RPC behavior, ownership constraints, and DB-level rate limiting are unverified from governance artifacts.
- **Risk:** (1) Username enumeration — if the RPC returns a generated username that reveals whether a base username is taken (e.g., `john` → `john123` because `john` is taken), an attacker could probe username availability by repeatedly attempting onboarding or directly calling the RPC. (2) No visible rate limiting means repeated calls are possible from a valid authenticated session. Impact is LOW — username availability is generally not sensitive — but the pattern is worth flagging for DB audit.
- **Severity:** LOW
- **Exploitability:** LOW — requires authenticated session; impact is information disclosure only (username existence)
- **Attack Preconditions:** Authenticated user account; repeated onboarding attempts or direct RPC calls
- **Blast Radius:** SEO/indexing — username enumeration could assist in actor discovery
- **Identity Leak Type:** Actor correlation (username existence confirmation)
- **Cache Trust Type:** None
- **RLS Dependency:** UNVERIFIED — RPC's internal ownership and rate-limiting logic unknown
- **Why it matters:** RPCs callable from the authenticated client with arbitrary input need DB-level constraints. The `generate_username` RPC behavior and security posture are unaudited. This is flagged as a DB audit item rather than an active vulnerability.
- **Recommended mitigation:** DB audit of `generate_username` RPC — confirm: (1) it does not leak user data beyond the generated username; (2) it has no side effects; (3) Supabase's built-in connection pooling and auth limits provide sufficient rate limiting for this use case. No app-layer change required unless DB audit reveals a risk.
- **Rationale:** RPCs with arbitrary input should be audited. This is a low-priority audit item.
- **Follow-up command:** DB (RPC audit)
