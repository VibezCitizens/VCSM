### DB-Layer: actor_can_manage_profile (post-migration 20260523230000)

```sql
CREATE OR REPLACE FUNCTION vport.actor_can_manage_profile(p_profile_id uuid)
RETURNS boolean LANGUAGE sql STABLE SECURITY DEFINER
AS $$
  SELECT EXISTS (
    SELECT 1
    FROM vport.profile_actor_access paa
    JOIN vc.actor_owners ao ON ao.actor_id = paa.actor_id
    WHERE paa.profile_id = p_profile_id
      AND ao.user_id     = auth.uid()
      AND paa.status     = 'active'
      AND COALESCE(ao.is_void, false) = false
  );
$$;
```

**The legacy `owner_user_id` branch was surgically removed in migration `20260523230000`.
The function now resolves exclusively through `vc.actor_owners`.**

No `owner_user_id` in the UPDATE policy. The canonical actor_owners chain is the sole

## 4. Identity Contract Validation

### Contract Requirements

```
actorId = vc.actors.id           ← canonical identity
Ownership authority = vc.actor_owners
Ownership must never be inferred from: profileId | vportId | userId | owner_user_id
```

### Compliance Assessment

| Check | Result |
|---|---|
| Ownership resolves through `actor_owners`? | YES — controller layer + DB RLS |
| `owner_user_id` used as ownership signal in new code? | NO — only as secondary DAL guard in settings write DAL (defense-in-depth, not primary gate) |
| `profileId` / `vportId` used as ownership signal? | NO |
| `user.id` as ownership signal in `assertActorOwnsVportActor`? | NO — derives `userProfileId` from `requesterActor.profile_id` then hits `actor_owners` |
| `updateVport()` violates Identity Contract? | NOT APPLICABLE — no production callers |

### Note on Secondary DAL Guards

`vports.write.dal.js` uses `.eq('owner_user_id', userId)` as a secondary WHERE clause guard
in `setVportBusinessCardSettingsDAL` and `setVportDirectoryVisibleDAL`. These are DEFENSE-IN-DEPTH
guards that fire AFTER `assertActorOwnsVportActorController` has already verified `actor_owners`
at the controller layer. They are not the primary ownership signal. They are a belt-and-suspenders
layer that could be deprecated if `actor_owners` were the sole DB-layer gate on those write paths.

This is a known architectural pattern documented in migration comments (VPD-V-FIX-001, VPD-V-FIX-002).

## 5. Scanner Accuracy Assessment

### Scanner Assumption

> UPDATE without `owner_user_id` = ownership vulnerability

### Why This Is Wrong

**Reason 1: Dead-code targeting.**
`updateVport()` has no active production callers. It exists only in a migration barrel
explicitly marked for removal. Flagging a dead-code path as HIGH severity misrepresents
the actual attack surface.

**Reason 2: Wrong ownership model assumed.**
The scanner recommends `.eq('owner_user_id', user.id)`. Under the VCSM Identity Contract,
`owner_user_id` is a legacy column — not the ownership authority. Ownership authority is
`vc.actor_owners`. Adding `owner_user_id` to `updateVport()` would:
  - Violate the Identity Contract
  - Create a false sense of security
  - Potentially allow a former owner to access a transferred VPORT (the exact stale-ownership
    risk documented in migration `20260523230000`)

**Reason 3: Architecture layer blindness.**
The scanner sees DAL WHERE clauses but cannot see:
  - Controller-layer ownership gates that fire BEFORE the DAL
  - DB RLS policies (`profiles_update_by_actor_owner`) that block unauthorized UPDATEs
    at the DB level regardless of what the DAL sends

Even if `updateVport()` were called by a malicious caller, the DB RLS policy would
block the update unless `actor_can_manage_profile()` returns true — which requires
a verified `vc.actor_owners` entry.

## 6. Scanner Rule Gap Analysis

### Rule ARCH-IDENTITY-001 — Required

Flag any application-layer ownership check that uses:
- `owner_user_id`
- `user.id` / `auth.uid()` in a WHERE clause directly
- `profileId` / `vportId` as ownership signals

Instead of:
- `vc.actor_owners` linkage (via controller or DAL)
- `assertActorOwnsVportActorController` in the call chain
- `assertSessionOwnsVportActorController` in the call chain

**Rationale:** The existing secondary DAL guards in `vports.write.dal.js` are acceptable
as defense-in-depth ONLY when a controller-layer `actor_owners` gate precedes them.
Any path that uses `owner_user_id` as the PRIMARY ownership check without a prior
`actor_owners` verification is a contract violation.

### Rule ARCH-DEAD-CODE-001 — Recommended

Before flagging a DAL function, verify it has at least one active production caller.
A DAL function exported only through a migration barrel (flagged for removal) or
only used in diagnostics/dev code should be classified separately from active paths.

Classification tiers:
- **ACTIVE**: called from controller → hook → screen in production path
- **DORMANT**: defined but no active production caller (lower severity, different remediation)
- **DEV-ONLY**: called only from diagnostics, tests, or dev utilities (informational)

### Rule ARCH-RLS-AWARE-001 — Recommended

When evaluating a `vport.profiles` UPDATE operation, the scanner should check for:
- `profiles_update_by_actor_owner` RLS policy presence
- `actor_can_manage_profile()` in the policy body
- `actor_owners` in the function resolution chain

If all three are present, DB-layer defense is active. A finding should be downgraded
from HIGH to DEFENSE-IN-DEPTH unless the controller layer is also missing a gate.

### Rule ARCH-RPC-SAFE-001 — Recommended

SECURITY DEFINER RPCs (`create_vport`, `soft_delete_vport`, `set_business_card_publish_state`)
handle ownership internally. Scanner should detect RPC call patterns and suppress
client-side ownership findings for those paths.

## Verdict

**Scanner Finding Classification: FALSE_POSITIVE**

**Severity Reassessment: INFO** (from HIGH)

Rationale: The flagged function has zero active production callers. The DB-layer RLS
policy independently enforces ownership through `vc.actor_owners` via
`actor_can_manage_profile()`. Even if `updateVport()` were called, the DB would reject
unauthorized writes. The scanner recommendation (add `owner_user_id`) would violate the
Identity Contract.

**Separate Code Hygiene Note (LOW):**
`updateVport()` exists in `vport.public.js` migration barrel without a controller wrapper
or actor_owners guard. If this function ever gains a production caller before the barrel
is removed, it would create an ownership gap. The remediation is Phase 2 barrel removal
as already documented in the file header — not adding `owner_user_id`.
